# 安居乐寓“AI管家”战略演进规划

**——从“智能找房”到“全生命周期无人化服务”的跨越**

## 一、 战略综述

本项目旨在利用最前沿的 **Agent（智能体）** 技术，重塑安居乐寓的服务流程。我们将从单一的“找房工具”进化为覆盖**租前、租中、租后**全链路的“AI数字管家”，目标是实现业务流程的深度自动化，预计承接 **70%** 的标准化管家工作量，显著实现降本增效。

## 二、 演进路线图：从 1.0 到 3.0 的技术跨越

### 第一阶段：1.0 智能化工作流探索期（奠基与验证）

> **阶段关键词：** 场景验证、确定性交付、结构化意图识别
> **核心理念：** “摸着石头过河” —— 在AI落地的早期，通过严谨的流程编排，率先跑通了“对话找房”的业务闭环。

* **技术架构特征：**
* **基于 Dify 工作流 的逻辑编排：** 我们利用 Dify 的工作流能力，搭建了一套高稳定性的业务流程。
* **结构化意图识别 (NLU)：** 引入了 DeepSeek/Qwen 等大模型，成功实现了从“用户口语”到“标准业务参数”（如区域、预算、房型）的精准提取。
* **所见即所得的 API 联动：** 率先打通了后端业务系统（API 接口），实现了用户在对话框输入需求，系统直接调用 `找房` 接口返回房源卡片的功能，系统直接调用 `找管家` 接口返回管家信息和管家名片的功能。


* **历史贡献与价值：**
* **验证了“AI+找房”的真伪命题：** 1.0 版本上线后，用户真实的对话数据证明了“自然语言找房”相比“传统筛选器”具有极高的接受度，为后续投入指明了方向。
* **建立了标准化的数据交互规范：** 我们在这一阶段定义好了 API 的输入输出标准（JSON Schema），沉淀了如“意图路由”、“参数继承”等核心逻辑，为 3.0 的智能体进化打下了坚实的数据地基。
* **所获荣誉：**1.0版本，在2025年集团AI创新大赛中荣获一等奖
* **保障了初期交付的稳定性：** 采用“有向无环图（DAG）”的模式，通过预设的 `If-Else` 分支逻辑，确保了在早期模型能力尚未完全成熟时，业务回答的可控性和准确性，避免了AI“胡说八道”。


* **面临的瓶颈（为何要升级）：**
* **响应太慢：** 与主流的通用AI相比、与竞品相比，我们的AI找房速度实在太慢（12s左右），无法满足用户对“快”的需求，用户体验差。
* **缺乏“柔性决策”能力：** 1.0 版本更像是一个“听话的高级执行者”，它能完美执行“找3000元的房”的指令，但缺乏在无明确指令下的主动分析与推荐能力（即思考能力）。


### 第二阶段：2.0 高速工作流时代（当前线上版本）

> **阶段关键词：** 极速响应、意图路由、精准推荐
> **核心变革：** 引入“意图分发（路由）”架构，从 1.0 的“线性机械判断”进化为“单脑决策、极速执行”。

#### 1. 核心技术升级（VS 1.0 版本）

* **架构重构：从“IF-ELSE 迷宫”到“中央大脑 (意图分发)”**
* **1.0 模式：** 旧版本通过堆砌了十几个 `If-Else` 和 `指派` 节点来逐一检查参数（如“有没有区域？”“有没有预算？”），逻辑臃肿，维护困难，每多走一个分支都会增加耗时。
* **2.0 模式：** 引入了一个名为 `中央大脑` 的核心大语言模型节点。它利用 COT（思维链）技术，一次性完成“意图识别、参数提取、指代消解（如理解‘第一套’）”的工作。这使得整个链路从复杂的网状结构变成了清晰的 **“大脑决策 -> 分发执行”** 结构。

* **模型与速度革命：从 12s 到 2s 的飞跃**
* **极速模型：** 2.0 版本全面接入了 `qwen-flash` 等超高速模型。相比 1.0 使用的较慢推理模型（Deepseek-R1），2.0 实现了**毫秒级**的意图理解。
* **链路缩短：** 得益于 意图分发 架构，用户的一句话不再需要经过层层关卡，而是直接命中目标接口。实测单次找房任务耗时从 **12秒 骤降至 2-3秒**，给用户带来了“快如闪电”的即时反馈体验。

#### 2. 业务价值提升

* **颗粒度精细化：从“推楼盘”到“推房源”**
* 1.0 版本更多是停留在“推荐某个门店和房型”的层面。
* 2.0 版本则是对接口进行了改造，返回的数据从房型升级到了具体的房间。另外，通过优化的 Python 数据处理节点 (`找房结果输出`)，能够解析后端返回的具体库存，直接向用户推荐**具体的房型、价格、具体的房间号**（如“西乡永丰店 · 单间 · 1500元/月”），并自动生成“通勤超方便”、“价格洼地”等智能推荐语。


* **知识库检索更精准：**
* 引入了重排序（Rerank）机制，使得 AI 在回答“押金怎么算”、“能不能养猫”等非标业务问题时，准确率显著提高，大大减轻了人工客服的压力。

#### 3. 局限性分析（“快、准、蠢”）

尽管 2.0 版本在效率上做到了极致，但本质上它仍是一个 **工作流（有向无环图）**，导致了其“智能的天花板”：

* **缺乏“深度思考”能力（蠢）：** 2.0 的路径是线性的（用户输入 -> 意图分发 -> AI执行 -> AI输出）。一旦 意图分发 决定了去“搜房”，哪怕搜出来的结果用户不满意，系统也无法像真人一样自动反思：“是不是我条件卡太死了？要不要帮用户扩大一点范围再搜一次？”它只能机械地报错或通过预设的简单逻辑进行安抚，无法进行复杂的多步推理。
* **无法处理复合任务：** 面对“帮我找一套宝安的房，帮我分析出性价比最高的那套，如果能的话帮我算算价格”这种跨多个业务域的指令，2.0 的单线程模式往往无能为力，只能处理单一意图。

**总结：** 2.0 版本是一个完美的“执行者”，它执行指令快准狠，但缺乏“参谋长”的变通与规划能力。这正是我们向 **3.0 Agent 自主智能体** 进化的根本动力。

### 第三阶段：3.0 Agent 智能决策时代（近期规划/开发中）

> **阶段关键词：** 自主决策、深度推理、全能交互
> **核心变革：** 彻底摒弃有向无环图（DAG）的束缚，采用 **Agent** 架构，从“指令执行者”进化为“业务思考者”。

#### 1. 核心架构质变：从“走迷宫”到“给工具箱”

* **2.0 工作流模式 (工作流)：**
* **逻辑：** 像走迷宫，路径是产品经理画死的。用户输入 -> 意图分发 -> 如果A走左边 -> 如果B走右边。
* **痛点：** **“快、准、蠢”**。它只能处理预设好的单线任务。一旦用户问“这三套哪套离地铁最近且最便宜？”，或者“帮我找房并顺便问问能不能养猫”，2.0 就会因为没有预设这条复合路径而卡壳。

* **3.0 智能体模式 (Agent)：**
* **逻辑：** **“大模型 + 工具箱”**。我们不再画流程图，而是给 AI 配备了三个核心工具：搜房、找管家、知识库。
* **自主权：** AI 会根据用户的自然语言，**自行判断**该用哪个工具，甚至**连续调用**。
* *场景举例：* 用户说“帮我找西乡的房，要能养猫的”。
* *Agent 思考：* 先调用 搜房工具 找房 -> 再调用 知识库工具 查养宠规则 -> 最后结合两份数据生成回复。这是 2.0 完全做不到的。

#### 2. 深度业务价值：从“罗列数据”到“优中选优”

* **深度推理与筛选：**
* 2.0 查到什么就吐什么，是个“传声筒”。
* 3.0 拥有了**分析能力**。当接口返回 20 套房源时，Agent 不会一股脑全扔给用户，而是会读取 业务系统返回的 字段（价格、面积、距离、地铁站、户型），结合用户需求（如“追求性价比”），在后台进行**二次计算和筛选**，最终只呈现 **Top 5** 最匹配的房源。

* **类人化的交互体验：**
* **对标通用大模型：** 体验上接近 DeepSeek、豆包等顶尖模型。它不再是冷冰冰的机器回复，而是能理解复杂的上下文，能进行多轮追问，让用户感觉是在和一个**“安居乐寓定制版超级AI”**聊天。
* **流式输出：** 虽然 Agent 的深度思考导致全量耗时增加（约 30s，相比 2.0 的 2s 慢不少），但通过流式输出，用户能实时看到 AI 的分析过程（如“正在为您比价...”、“正在查询管家信息...”），这种**“过程透明化”**反而增强了用户对结果专业度的信任。

#### 3. 风险与挑战（客观评估）

虽然 3.0 足够智能，但我们必须正视其带来的新挑战：

* **成本激增：**
* 由于 Agent 需要将所有工具的定义以及大量的中间思考过程都输入给大模型，**Token 消耗量预计是 2.0 版本的 3 倍以上**。这直接增加了运营成本。


* **服务边界的风控：**
* 2.0 的流程是写死的，绝对可控。
* 3.0 是 AI 生成的，虽然灵活，但也存在**“服务边界不可控”**的风险。例如 AI 可能会过度热情地答应用户“帮您预留”等它实际做不到的操作，或者在没有数据的情况下编造房源点评（幻觉）。
* **对策：** 我们在 系统提示词 中植入了严格的 **“负向约束”** 和 **“数据落地协议”**，明确禁止其承诺未授权的服务，并强制要求所有推荐必须基于业务系统返回的字段数据，严禁瞎编。

## 三、 未来愿景：构建“AI数字管家”虚拟劳动力体系

### 1. 战略定位：从“辅助工具”到“AI员工”

未来的“AI管家”不再是一个简单的问答机器人，我们将基于 **多智能体协作** 架构，打造一支**7x24小时在岗的数字化管家团队**。

它具备**独立思考、自主决策、跨系统执行**的能力，预计可直接替代 **70%** 的标准化管家工作量，让安居乐寓的运营模式从“人力密集型”向“技术密集型”转型。

### 2. 全生命周期无人化服务场景

我们将开发一系列专职的子Agent，负责客户的全生命流程，协同完成以下闭环：

#### **(1) 售前阶段：无人化金牌销售**

* **全能接待：** 超越简单的找房，Agent将接管从“咨询”、“自助看房”、“比价”到“签约”的全部流程。
* **联动自助看房：** 用户选中房源后，Agent直接调用自助看房相关能力，为用户预约心仪房源，可进行现场自助看房。用户到达现场，Agent通过语音进行“伴随式讲解”。
* **动态定价：** 这是一个核心突破点。Agent将调用业务系统内置复杂的定价逻辑，根据用户选择的租期（长租/短租）、身份权益（如深梦启航）、当前优惠政策，**自主计算最终成交价**，彻底告别“人工算价、人工报价”的低效环节。
* **自动签约闭环：** 确认意向后，Agent自动引导录入租客信息、生成电子合同、发起人脸识别签约、可直接完成首期款项的支付。

#### **(2) 租中阶段：全时段贴身管家**

* **复杂业务自动化：**
* **智能换房：** Agent可以根据用户意图，推荐换房房源，用户通过自助看房相关能力进行现场看房。用户确认后，Agent将自动调用业务系统换房相关能力，计算新旧房源差价、换房手续费及退补金额，自动生成换房账单，用户仅需对新房间电子合同进行确认签字。
* **续约管理：** Agent根据用户剩余租期，提前预判续约意愿，根据业务系统内置复杂的定价逻辑自动计算续约价格，一键办理续约。

* **工单与生活服务：**
* **维修/投诉：** 用户可通过自然语言，向Agent描述问题，Agent自动创建工单，工单根据业务系统配置的处理流程，自动派发给维修师傅或管家处理。
* **日常缴费：** 自动查询水电用量，查询账单并推送支付入口。
* **门禁管理：** 处理日常远程开门禁、等高频请求。

* **营销与增值服务：**
* **深度活动整合：** 针对“深梦扬帆”等集团重点活动，不再是简单的转发海报。它拥有**接口调用权限**，可直接核验用户资格（是否符合毕业生标准），实现“对话即报名”、“对话即核验”。
* **营销活动：** 营销活动可通过agent进行报名参加，活动流程中了解活动进程 和活动结果。


#### **(3) 租后阶段：全流程自助办理**

* **转租：** Agent可以根据用户意图，自动调用业务系统转租的相关能力，计算转租相关费用，自动生成转租账单，新用户仅需对转租电子合同进行确认签字，而原租户则无需任何操作，只需等待退款审批通过后退款到账即可。
* **退租：** Agent将自动调用业务系统退租相关能力，计算退租相关费用，自动生成退租账单，系统自动发起退款审批流程，用户仅需等待退款审批通过后退款到账即可。
* **退款服务：** 用户可以用自然语言，向Agent进行退款服务的查询，agent调用相关业务系统能力，查询退款状态；在原路退款失败时，若需使用代付进行退款，agent也可以替用户填报退款银行账号。

## 四、 资源需求与保障

为了实现从 2.0 到“AI员工”的跨越，我们需要公司在**经费预算**层面提供强有力的支持。这不仅仅是采购软件，而是对“数字劳动力”的投资。

### 1. 专项经费支持

* **大模型 API 调用费用（长期运营成本）：**
* **现状：** 3.0 Agent 模式需要消耗大量的 Token 用于思考和工具调用。
* **预估：** 单次服务成本约为 2.0 版本的 **3倍**。
* **价值主张：** 虽然 API 成本上升，但相比于节省下来的**管家薪资、社保及管理成本**，这笔投入是极具性价比的（AI 的边际成本远低于人力）。


* **AI Coding 辅助开发费用（研发提效成本）：**
* 为了加速庞大的后端接口对接和 Agent 逻辑开发，我们需要采购 **AI Coding 工具**（如字节跳动的 trae）的企业版授权。这将使我们的代码开发效率提升 50% 以上，缩短项目上线周期。


* **未来平台迁移费用（Coze/字节生态）：**
* 考虑到未来可能需要更强的agent能力，我们计划在下一阶段探索 **字节跳动 Coze** 平台。届时可能涉及相应的平台订阅费或 Token 包年费用。
