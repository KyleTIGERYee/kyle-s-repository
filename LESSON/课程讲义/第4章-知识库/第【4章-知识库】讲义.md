# 第一部分：RAG原理与工程复杂度（25分钟）

## 1.1 什么是RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将外部知识库与大语言模型结合的技术架构。它的核心思想是：当用户提问时，先从知识库中检索相关信息，然后将这些信息作为上下文提供给大模型，让大模型基于这些上下文生成回答。

### RAG的基本工作流程

### RAG的两大核心优势

#### 优势一：解决大模型的知识边界问题

大语言模型的知识来自训练数据，存在三个固有限制：

1. **知识截止时间**：模型只能知道训练数据中的信息，无法获取最新知识
2. **领域知识缺失**：专业领域的内部文档、私有数据不在训练集中
3. **幻觉问题**：模型会"编造"看似合理但实际错误的信息

RAG通过实时检索外部知识库，有效解决了这些问题。

#### 优势二：可解释性和可追溯性

与纯大模型生成相比，RAG的回答可以追溯：
- **来源明确**：知道答案来自哪篇文档
- **可验证性**：可以人工检查检索到的片段
- **可更新**：只需更新知识库，无需重新训练模型

## 1.2 RAG的核心技术栈（工程复杂度详解）

RAG看似简单，但要构建一个生产级的RAG系统，需要掌握一整套复杂的技术栈。让我们逐一拆解：

### 第一层：文档处理层（Document Processing）

#### 1.2.1 文档解析（Document Parsing）

不同格式的文档需要不同的解析策略：

| 文档类型 | 解析难点 | 常用工具 |
|---------|---------|---------|
| **PDF** | 版式复杂、表格识别、图文混排 | PyPDF2, pdfplumber, unstructured |
| **Word** | 样式保留、目录结构、批注处理 | python-docx, mammoth |
| **HTML** | 标签清理、脚本去除、样式分离 | BeautifulSoup, readability-lxml |
| **扫描件** | OCR识别、版面分析、文字定位 | Tesseract, PaddleOCR, Azure OCR |
| **图片** | 图表理解、文字提取、上下文关联 | GPT-4V, Claude Vision |

**工程挑战**：
- PDF中的表格可能被错误解析为纯文本，丢失结构信息
- 扫描件的OCR准确率受图像质量影响，手写体识别困难
- 多栏排版的文档需要智能判断阅读顺序

#### 1.2.2 文档切片（Chunking）—— RAG最复杂的环节

文档切片是RAG系统中**最关键也最难**的环节。切片策略直接影响检索质量。

**为什么需要切片？**

大语言模型有上下文长度限制（如GPT-4的128K tokens），而企业文档往往很长（一份合同可能几十页）。我们必须将长文档切分成小块，才能：
1. 适配模型的上下文窗口
2. 提高检索精度（小块更精准）
3. 降低计算成本

**常见的切片策略**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     文档切片策略对比                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  策略1：固定长度切片（Fixed-length Chunking）                    │
│  ┌────────┬────────┬────────┬────────┬────────┐                │
│  │ Chunk1 │ Chunk2 │ Chunk3 │ Chunk4 │ Chunk5 │                │
│  │ 512tk  │ 512tk  │ 512tk  │ 512tk  │ 512tk  │                │
│  └────────┴────────┴────────┴────────┴────────┘                │
│  优点：简单、快速                                               │
│  缺点：可能切断语义（如"如果...那么..."被分开）                    │
│                                                                 │
│  策略2：递归字符切片（Recursive Character Chunking）              │
│  ┌─────────────┬─────────────┬─────────────┐                   │
│  │  段落1      │   段落2     │   段落3     │                   │
│  │  (完整)     │   (完整)    │   (完整)    │                   │
│  └─────────────┴─────────────┴─────────────┘                   │
│  优点：保持段落完整性                                           │
│  缺点：段落长度不一，可能超出限制                                 │
│                                                                 │
│  策略3：语义切片（Semantic Chunking）                            │
│  ┌─────────────────┬─────────────────┬────────┐                │
│  │   主题A相关      │    主题B相关     │ 主题C  │                │
│  │   (语义完整)     │    (语义完整)    │ (完整) │                │
│  └─────────────────┴─────────────────┴────────┘                │
│  优点：语义连贯，检索精准                                        │
│  缺点：需要额外的语义分析模型，计算成本高                          │
│                                                                 │
│  策略4：结构感知切片（Structure-aware Chunking）                  │
│  ┌────────┬────────┬────────┬────────┐                        │
│  │ 标题1  │ 章节1  │ 章节2  │ 章节3  │                        │
│  │(元数据)│(内容)  │(内容)  │(内容)  │                        │
│  └────────┴────────┴────────┴────────┘                        │
│  优点：保留文档结构，支持层级检索                                 │
│  缺点：需要文档格式标准化                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**切片的核心难题**：

> **"Chunk Size Trade-off"（切片大小权衡）**

想象你在整理一本厚厚的教科书：

**切片太小**（如100 tokens，约70个汉字）：
- ✅ 就像把书切成一张张便签，找特定知识点时很精准
- ❌ 但你可能只看到"因为..."，却看不到"所以..."
- ❌ 回答一个问题需要拼很多张便签，容易遗漏

**切片太大**（如2000 tokens，约1400个汉字）：
- ✅ 就像把书按章节分开，每章内容完整
- ❌ 但找一个小知识点时，会带进来很多无关内容
- ❌ 几章内容就塞满模型的"脑子"，装不下更多

**Dify中的切片设置**：

在Dify知识库中，你可以在创建知识库时设置**分段规则**：


**实际案例**：

假设有一份退款政策文档：
```
第3条：退款条件
如果用户满足以下条件，可以申请退款：
1. 购买后7天内提出申请
2. 产品未使用且包装完整
3. 提供原始购买凭证

⚠️ 注意：如果商品已使用，即使包装完好也不予退款。

第4条：退款流程
用户需要按照以下步骤操作：
第一步：登录账户，进入"我的订单"
第二步：找到对应订单，点击"申请退款"
第三步：填写退款原因，上传凭证
...
```

**切片场景分析**：

| 切片位置 | 检索结果 | 问题 |
|---------|---------|------|
| 在第3条和第4条之间切 | 只检索到"退款条件" | 用户问"如何申请退款"时，系统不知道具体操作步骤 |
| 在第4条中间切 | "退款流程"被切成两半 | 可能只检索到"第一步"，丢失后续步骤 |
| 把⚠️警告单独切 | 警告信息被孤立 | 用户可能看不到关键限制条件 |

这就是**切片导致的信息丢失**——就像拼图被强行分开，即使找到了其中一块，也可能拼不出完整的画面。

**在Dify中如何避免？**

1. **使用父子分段**：保持文档的层级结构（父段落包含子段落）
2. **设置重叠区域**：让相邻切片有重叠内容，保持连贯性
3. **手动调整**：对于关键文档，手动检查切片点是否合理

**ANN算法（Approximate Nearest Neighbor）—— 近似最近邻算法**：

想象你在一个巨大的图书馆找一本书：

**暴力搜索**：一本本翻，直到找到最相似的（太慢！）

**ANN算法**：先建立一个"导航地图"
- **HNSW（分层导航小世界）**：像地铁线路图，先坐快线到大致区域，再换乘慢线精确到达
- **IVF（倒排文件索引）**：先把书按主题分类（聚类），先在相关类别里找
- **PQ（乘积量化）**：把书的特征压缩成"简写版"，节省存储空间

**工程挑战**：
- 索引构建时间长（就像给100万本书建立索引需要很长时间）
- ANN是近似算法，可能漏掉真正相似的向量（就像导航可能推荐次优路线）
- 参数调优（如HNSW的M、efConstruction）需要经验

### 第三层：检索层（Retrieval Layer）

#### 1.2.5 检索策略（Retrieval Strategies）

**基础检索：向量相似度检索**

```python
# 伪代码示例
def basic_retrieve(query, vector_db, top_k=5):
    # 1. 将查询转换为向量
    query_vector = embedding_model.encode(query)
    
    # 2. 在向量数据库中搜索最相似的top_k个向量
    results = vector_db.search(query_vector, top_k=top_k)
    
    # 3. 返回对应的文本片段
    return [result.text for result in results]
```

**高级检索技术**：

#### 1. 混合检索（Hybrid Retrieval）—— 双剑合璧

**什么是混合检索？**

想象你在图书馆找一本书：
- **向量检索**就像你告诉管理员"我想找一本关于人工智能的书"，管理员根据主题相关性给你推荐
- **关键词检索**就像你直接报出书名《深度学习》，管理员精确找到这本书

但有时候，你需要两者结合：
- 你想找"那本讲AI的书，书名里有'智能'两个字"
- 单独用向量检索，可能找不到书名精确匹配
- 单独用关键词检索，可能错过内容相关但书名不同的好书

**混合检索 = 向量检索 + 关键词检索**

#### RRF（Reciprocal Rank Fusion）算法详解

**RRF是什么？**

RRF（倒数排名融合）是一种**将多个检索结果列表合并成一个列表**的算法。它的核心思想是：**排名越靠前的结果，权重越高**。

**通俗比喻**：

想象有两个评委给歌手打分：
- 评委A的排名：歌手X第1名，歌手Y第3名，歌手Z第5名
- 评委B的排名：歌手Y第1名，歌手Z第2名，歌手X第4名

怎么确定最终排名？简单平均不行（因为第1名和第4名平均是2.5，但意义不同）。

RRF的做法：**用倒数来计算分数**

**RRF计算公式**：

```
RRF分数 = 1 / (k + 排名)

其中：
- k是常数（通常取60，防止排名靠后的结果分数过于接近）
- 排名是结果在原始列表中的位置（第1名=1，第2名=2...）
```

**具体例子**：

```
向量检索结果：          关键词检索结果：
1. 文档A               1. 文档C
2. 文档B               2. 文档A
3. 文档C               3. 文档D
4. 文档D               4. 文档B

计算RRF分数（k=60）：

文档A：
- 向量排名：1 → 分数 = 1/(60+1) = 1/61 = 0.0164
- 关键词排名：2 → 分数 = 1/(60+2) = 1/62 = 0.0161
- RRF总分 = 0.0164 + 0.0161 = 0.0325

文档B：
- 向量排名：2 → 分数 = 1/62 = 0.0161
- 关键词排名：4 → 分数 = 1/64 = 0.0156
- RRF总分 = 0.0317

文档C：
- 向量排名：3 → 分数 = 1/63 = 0.0159
- 关键词排名：1 → 分数 = 1/61 = 0.0164
- RRF总分 = 0.0323

文档D：
- 向量排名：4 → 分数 = 1/64 = 0.0156
- 关键词排名：3 → 分数 = 1/63 = 0.0159
- RRF总分 = 0.0315

最终排名：
1. 文档A (0.0325) ← 两个检索都认为靠前
2. 文档C (0.0323) ← 关键词检索第1名
3. 文档B (0.0317)
4. 文档D (0.0315)
```

**RRF的优势**：

1. **无需训练**：不需要机器学习，纯数学计算
2. **公平融合**：不同检索方式的结果可以公平比较
3. **对排名敏感**：第1名和第2名的差距，比第10名和第11名的差距更大
4. **鲁棒性强**：即使某个检索方式没有返回某个文档，也能正常计算

**在Dify中使用RRF**：

当你在Dify中选择**混合检索**时，系统会自动使用RRF算法：

```
Dify混合检索内部流程：
1. 向量检索 → 返回Top K结果（带相似度分数）
2. 全文检索 → 返回Top K结果（带匹配分数）
3. RRF融合 → 计算每个文档的综合分数
4. 重排序（可选）→ 使用Rerank模型进一步优化
5. 返回最终结果
```

#### 2. 重排序（Re-ranking）—— 精益求精

**什么是重排序？**

想象你在招聘：
1. **初筛（召回）**：HR从1000份简历中快速筛选出100份符合条件的（看学历、经验等硬性指标）
2. **复筛（重排序）**：部门主管仔细审阅这100份，从中选出最匹配的5份（看项目经验、技能匹配度）

**RAG中的重排序**：
- **第一步（召回）**：用轻量级模型（如Bi-Encoder）快速从10000个片段中找出100个候选
- **第二步（重排序）**：用精细模型（如Cross-Encoder）仔细比较这100个，选出最相关的Top 5

**为什么需要两步？**

| 阶段 | 模型 | 速度 | 精度 | 作用 |
|-----|------|------|------|------|
| 召回 | Bi-Encoder | 快（毫秒级） | 中等 | 快速缩小范围 |
| 重排序 | Cross-Encoder | 慢（秒级） | 高 | 精细排序 |

**比喻**：
- Bi-Encoder像**相亲角的快速浏览**——看照片、基本条件，快速筛选
- Cross-Encoder像**深度面试**——详细了解每个人的背景、能力，精确匹配

**Dify中的Rerank设置**：

在Dify知识库中，你可以在检索设置中启用**Rerank模型**：

```
知识库 → 检索设置 → 启用Rerank模型：
├─ Cohere Rerank
├─ Jina AI Rerank
└─ 其他自定义Rerank模型
```

启用后，Dify会：
1. 先用向量检索/全文检索召回Top K（如Top 20）
2. 用Rerank模型对这20个片段进行精细重排序
3. 返回重排序后的Top N（如Top 3）给LLM

**效果对比**：

| 场景 | 无Rerank | 有Rerank | 提升 |
|-----|---------|---------|------|
| 找"退款流程" | 可能返回"退款政策" | 更精准返回"退款操作步骤" | 准确率+20% |
| 找"iPhone 15价格" | 可能返回"iPhone 14" | 精确匹配"iPhone 15" | 准确率+35% |

#### 3. 查询扩展（Query Expansion）
   - 用LLM生成查询的多种表述
   - 或者用伪相关反馈（Pseudo-Relevance Feedback）扩展查询

4. **多向量检索（Multi-Vector Retrieval）**
   - 每个文档用多个向量表示（如摘要向量、标题向量、段落向量）
   - 提高召回率

#### 1.2.6 查询重写（Query Rewriting）

用户的原始查询往往不够优化，需要进行重写：

| 技术 | 目的 | 示例 |
|-----|------|------|
| **Query Expansion** | 扩展同义词 | "电脑" → "电脑/计算机/PC" |
| **HyDE** | 生成假设文档 | 查询→生成假答案→用假答案检索 |
| **Step-back Prompting** | 抽象化查询 | "2023年深圳房价" → "中国房价趋势" |
| **Sub-query Decomposition** | 分解复杂查询 | "A和B的区别" → 分别查A和B |

#### 1.2.7 上下文组装（Context Assembly）

检索到的多个片段需要组装成连贯的上下文：

```

│                    上下文组装策略                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│                                                                 │

```

# 第二部分：RAG的局限性与范式转移（25分钟）

## 2.1 RAG的七大失败点

根据Deakin大学的研究论文《Seven Failure Points When Engineering a Retrieval Augmented Generation System》，RAG系统存在以下核心失败点：

### FP1: 内容缺失（Missing Content）

**问题描述**：用户问了一个问题，但知识库中没有相关内容。RAG系统不会回答"我不知道"，而是会基于检索到的无关内容生成一个看似合理但实际错误的答案。

**案例**：
```
用户问："你们支持支付宝付款吗？"
知识库：只有微信支付和银行卡支付的文档

RAG系统可能回答："我们支持多种支付方式，包括微信支付、
银行卡支付等。"（暗示支持支付宝，实际不支持）
```

### FP2: 错过排名靠前的文档（Missed the Top Ranked Documents）

**问题描述**：正确答案在知识库中，但向量相似度检索没有把它排在前面。这可能是因为：
- 查询和答案的语义表达方式差异大
- 嵌入模型对特定领域的理解不够
- ANN算法的近似误差

### FP3: 不在上下文中（Not in Context）

**问题描述**：正确答案被检索到了，但因为以下原因没有进入LLM的上下文：
- 检索到太多片段，正确答案被截断
- 重排序环节把正确答案过滤掉了
- 上下文长度限制导致无法包含所有相关内容

### FP4: 无法提取（Not Extracted）

**问题描述**：正确答案在上下文中，但LLM没有正确提取。这可能是因为：
- 上下文被其他噪声信息干扰
- 答案需要跨多个片段推理，但LLM没有做到
- 答案表达过于隐晦，LLM没有理解

### FP5: 错误格式（Wrong Format）

**问题描述**：用户要求特定格式的回答（如表格、JSON），但LLM没有遵循。

### FP6: 具体性不准确（Incorrect Specificity）

**问题描述**：答案在正确的方向上，但不够具体或过于具体。例如：
```
用户问："退款多久到账？"
错误回答："退款会在一定时间内到账。"（太笼统）
正确回答："退款会在3-7个工作日内原路返回。"
```

### FP7: 不完整（Incomplete）

**问题描述**：答案不完整，遗漏了部分信息。这通常是因为：
- 检索到的片段不完整（切片问题）
- LLM没有综合所有相关信息

## 2.2 RAG的根本性局限

除了具体的失败点，RAG还存在一些**结构性局限**：

### 局限一：切片导致的信息丢失

这是RAG最核心的局限。让我们深入分析：

```
┌─────────────────────────────────────────────────────────────────┐
│                    切片导致的信息丢失案例                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  原始文档：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 第5节：退款政策                                            │   │
│  │                                                          │   │
│  │ 5.1 退款条件                                              │   │
│  │ 用户申请退款必须满足以下所有条件：                          │   │
│  │ 1. 订单完成支付后7天内提出申请                              │   │
│  │ 2. 商品未使用且包装完好                                    │   │
│  │ 3. 提供原始购买凭证（订单号或发票）                          │   │
│  │                                                          │   │
│  │ ⚠️ 注意：如果商品已使用，即使包装完好也不予退款。            │   │
│  │                                                          │   │
│  │ 5.2 退款流程                                              │   │
│  │ 第一步：登录账户，进入"我的订单"                           │   │
│  │ 第二步：找到对应订单，点击"申请退款"按钮                    │   │
│  │ 第三步：填写退款原因，上传商品照片                          │   │
│  │ 第四步：等待审核（1-2个工作日）                            │   │
│  │ 第五步：审核通过后，款项将在3-7个工作日内原路返回            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  切片后（假设每片200字符）：                                     │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐            │
│  │ 片段1：       │ │ 片段2：       │ │ 片段3：       │            │
│  │ 退款条件...   │ │ 注意：如果... │ │ 退款流程...   │            │
│  │ (5.1节内容)   │ │ (警告信息)    │ │ (5.2节内容)   │            │
│  └──────────────┘ └──────────────┘ └──────────────┘            │
│                                                                 │
│  用户提问："我买了商品但没使用，包装也完好，为什么退款被拒？"    │
│                                                                 │
│  问题分析：                                                      │
│  - 用户的问题暗示"商品未使用+包装完好=应该退款"                  │
│  - 但关键信息"如果商品已使用，即使包装完好也不予退款"            │
│    在片段2中，且是警告形式                                       │
│  - 如果检索只返回片段1和3，LLM看不到警告信息                      │
│  - LLM可能回答："根据退款条件，您应该可以退款，请联系客服..."     │
│  - 实际原因可能是用户误解了"未使用"的定义                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键洞察**：
- 文档中的信息是**网状关联**的，但切片把它们变成了**孤立片段**
- 警告、例外、条件等关键信息往往分散在不同位置
- 人类的阅读理解依赖于**全局上下文**，但RAG只提供**局部片段**

### 局限二：向量相似度 ≠ 语义相关性

向量检索基于**语义相似度**，但相似度不等于相关性：

```
用户查询："如何申请退款？"

检索结果可能包含：
✓ "退款申请流程如下..."（相关）
✗ "退款政策不可更改"（相似度高，但不回答"如何"）
✗ "退款金额将原路返回"（相似度高，但不回答"如何申请"）
```

### 局限三：无法处理复杂推理

很多问题需要**跨文档推理**或**多步推理**：

```
用户问："我是企业客户，想退租，需要准备什么材料？"

这需要：
1. 找到"企业客户"的定义和范围

- 没有显式的推理步骤，答案可靠性低
```

# 第三部分：全量文档知识库架构详解（30分钟）

## 3.1 PageIndex：无向量推理式RAG

PageIndex是由VectifyAI提出的新一代RAG框架，其核心理念是：**用推理代替相似度，用树状结构代替扁平向量**。

### PageIndex的核心特点

```
┌─────────────────────────────────────────────────────────────────┐
│                    PageIndex 核心特点                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ❌ No Vector DB（无向量数据库）                                  │
│     不使用向量相似度检索，而是用文档结构和LLM推理                 │
│                                                                 │
│  ❌ No Chunking（无文档切片）                                     │
│     文档按自然章节组织，不是人工切片                              │
│                                                                 │
│  ✅ Human-like Retrieval（类人检索）                              │
│     模拟人类专家阅读长文档的方式：先看目录，再深入章节             │
│                                                                 │
│  ✅ Better Explainability（更好的可解释性）                       │
│     检索过程基于推理，可追溯、可解释，有明确的页码和章节引用        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### PageIndex的工作流程

PageIndex的工作分为两个阶段：

#### 阶段一：构建树状索引（Indexing）

```
原始文档（PDF/Markdown）
    ↓
PageIndex生成器
    ↓
树状索引结构（JSON）

示例：
{
  "title": "2023年财务报告",
  "node_id": "0001",
  "start_index": 1,
  "end_index": 50,
  "summary": "本年度财务报告涵盖收入、支出、利润等关键指标...",
  "nodes": [
    {
      "title": "1. 执行摘要",
      "node_id": "0002",
      "start_index": 1,
      "end_index": 3,
      "summary": "概述公司2023年的整体表现...",
      "nodes": [...]
    },
    {
      "title": "2. 收入分析",
      "node_id": "0003",
      "start_index": 4,
      "end_index": 15,
      "summary": "详细分析各业务线的收入情况...",
      "nodes": [
        {
          "title": "2.1 产品A收入",
          "node_id": "0004",
          "start_index": 4,
          "end_index": 8,
          "summary": "产品A在2023年的收入为..."
        },
        {
          "title": "2.2 产品B收入",
          "node_id": "0005",
          "start_index": 9,
          "end_index": 15,
          "summary": "产品B在2023年的收入为..."
        }
      ]
    }
  ]
}
```

**关键设计**：
- 每个节点包含：标题、ID、页码范围、摘要
- 层级结构反映文档的自然组织结构
- 摘要让LLM快速了解节点内容，无需阅读全文

#### 阶段二：基于推理的检索（Retrieval）

```
用户查询："产品A在第三季度的收入是多少？"
    ↓
LLM推理：这个问题涉及"产品A"和"收入"
    ↓
树搜索：
  1. 查看根节点摘要 → 确定这是财务报告
  2. 查看子节点 → 找到"2. 收入分析"
  3. 深入查看 → 找到"2.1 产品A收入"
  4. 读取"2.1 产品A收入"的完整内容
    ↓
找到答案 → 生成回答
```

**树搜索 vs 向量检索**：

| 对比项 | 向量检索 | 树搜索 |
|-------|---------|--------|
| 检索逻辑 | 计算相似度，选top-k | 基于推理，逐层导航 |
| 可解释性 | 低（为什么选这个向量？） | 高（为什么选这个章节？） |
| 信息粒度 | 固定大小片段 | 自然章节（可变长度） |
| 跨章节关联 | 困难 | 自然（通过树结构） |
| 长文档处理 | 效果下降 | 效果稳定 |

### PageIndex的性能表现

PageIndex在FinanceBench（金融问答基准测试）上取得了**98.7%的准确率**，显著优于传统向量RAG方案。

**原因分析**：
1. 金融报告结构清晰，树状索引效果好
2. 问答通常针对特定章节，树导航精准
3. 完整章节上下文避免了切片导致的信息丢失

### 各阶段详解

#### M - Map（全景加权导航）

**核心问题**：用户问了一个问题，如何快速定位可能相关的文档？

**解决方案**：多维加权检索引擎

```python
# 伪代码示意
def map_navigation(user_query):
    # 1. 提取意图词
    intent_words = extract_keywords(user_query)
    # 例如："退房流程" → ["退房", "流程"]
    
    # 2. 同义词扩展
    expanded_words = []
    for word in intent_words:
        expanded_words.extend(synonyms_dict.get(word, [word]))
    # 例如："退房" → ["退房", "退租", "办理退房", "退租手续", "退房结算"]
    
    # 3. 在system_map.json中搜索
    candidates = search_system_map(expanded_words)
    
    # 4. 多维度打分
    scored_results = []
    for doc in candidates:
        score = (
            keyword_match_score(doc, expanded_words) * 0.4 +
            semantic_similarity_score(doc, user_query) * 0.3 +
            business_relevance_score(doc, user_query) * 0.2 +
            recency_score(doc) * 0.1
        )
        scored_results.append((doc, score))
    
    # 5. 返回Top 10
    return sorted(scored_results, key=lambda x: x[1], reverse=True)[:10]
```

## 3.3 PageIndex vs M-R-E-R-G：核心重合点分析

| 维度 | PageIndex | M-R-E-R-G | 核心共识 |
|-----|-----------|-----------|---------|
| **哲学基础** | Vectorless, Reasoning-based | 全量文档检索，拒绝切片 | **No Chunking** |
| **索引方式** | 树状结构索引 | 全景地图 + 同义词扩展 | **结构化导航 > 扁平检索** |
| **检索逻辑** | 基于推理的树搜索 | 级联递归探测 | **推理 > 相似度** |
| **阅读策略** | 完整节点内容 | 绝对的无损精读 | **全量上下文** |
| **生成方式** | 基于完整上下文 | 基于完整依赖树 | **全局理解** |

**共同的核心主张**：

1. **No Chunking（拒绝切片）**
   - 文档切片是导致信息丢失的根本原因
   - 保持文档的完整性和自然结构

2. **Reasoning-based（基于推理）**
   - 用LLM的推理能力代替机械的相似度计算
   - 检索过程应该是可解释、可追溯的

3. **Hierarchical Navigation（层级导航）**
   - 文档有自然的层级结构（章节、段落）
   - 利用这种结构进行智能导航

4. **Full Context（全量上下文）**
   - 把完整文档喂给LLM，而不是片段
   - 让LLM自己决定关注哪些部分

# 第五部分：决策框架——何时用RAG，何时用全量文档（5分钟）

## 5.1 决策矩阵

| 场景特征 | 推荐方案 | 原因 |
|---------|---------|------|
| **文档数量** > 1000篇 | RAG | 全量超出上下文限制 |
| **文档数量** < 100篇 | 全量文档 | 可以全部放入上下文 |
| **单篇长度** > 100页 | 全量文档 + 树索引 | 避免切片，用树导航 |
| **查询类型**：简单事实 | RAG | 快速检索即可 |
| **查询类型**：复杂流程 | 全量文档 | 需要完整上下文 |
| **查询类型**：跨文档关联 | 全量文档 | 需要多文档推理 |
| **更新频率**：高频 | 全量文档 | 无需重建索引 |
| **精度要求**：极高 | 全量文档 | 避免切片损失 |
| **响应速度**：要求极高 | RAG | 向量检索更快 |

## 5.2 混合策略

在实际应用中，可以采用混合策略：

```
┌─────────────────────────────────────────────────────────────────┐
│                    混合知识库架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  第一层：快速过滤（RAG）                                         │
│  ├─ 使用轻量级向量检索快速定位相关文档                           │

│  └─ 让大模型深度理解并生成答案                                   │
│                                                                 │
│  优势：兼顾效率和精度                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### search.py 模板

```python
#!/usr/bin/env python3
import os
import sys
import json
from pathlib import Path

def search_documents(query, docs_dir="references/documents"):
    """
    简单的关键词检索
    返回匹配的文档路径列表
    """
    results = []
    query_lower = query.lower()
    
    # 遍历所有.md文件
    for md_file in Path(docs_dir).rglob("*.md"):
        content = md_file.read_text(encoding='utf-8')
        
        # 简单匹配：检查关键词是否在内容中
        if query_lower in content.lower():
            # 计算匹配度（这里简化处理）
            score = content.lower().count(query_lower)
            results.append({
                "path": str(md_file),
                "score": score
            })
    
    # 按匹配度排序
    results.sort(key=lambda x: x["score"], reverse=True)
    return results[:10]  # 返回Top 10

if __name__ == "__main__":
    query = sys.argv[1] if len(sys.argv) > 1 else ""
    results = search_documents(query)
    print(json.dumps(results, ensure_ascii=False, indent=2))
```
