精讲：Token —— AI 世界的“颗粒度”

为什么 1 个汉字不等于 1 个 Token？

大模型并不直接“读取”字符，而是将文本切割为最频繁出现的字符组合。 在中文环境下，1 个汉字平均约占 1.5 到 2 个 Tokens。

典型比例：
🇺🇸 1 个英文单词 ≈ 1 - 1.3 Tokens
🇨🇳 100 个汉字 ≈ 150 - 180 Tokens

精讲：System Prompt —— 塑造 AI 的源头

黄金四要素架构

1

身份 (Identity)

你是谁？(e.g., 资深租房专家)

2

任务 (Task)

你要做什么？(e.g., 分析财务报告)

3

准则 (Rules)

怎么做？(e.g., 输出必须是 JSON)

4

边界 (Boundary)

不要做什么？(e.g., 严禁回答医疗建议)

专业级示例片段

# 身份
你是安居乐寓的合规审查官。
# 核心任务
根据用户上传的电子合同，对比《公司租赁法务规范》，找出潜在的风险条款。
# 约束
1. 仅针对法律条款进行分析。
2. 如果合同内容模糊，必须在报告中标明。

提示：给 AI “具体的负面约束”比“泛泛的正面引导”更有效。

精讲：Few-shot —— 用少量示例“对齐”期望

BAD CASE

低质量示例

用户：我想退房。
AI：好的，请联系管家。

用户：押金什么时候退？
AI：一般 3 天。

风险：描述过于口语化，AI 无法捕捉专业语境。
GOOD CASE

高质量示例 (结构化)

Input:

"我要申请提前退宿，深梦扬帆活动期的"

Output:

{"status": "REFUND_REQUEST", "policy": "FESTIVAL_SPECIAL_TERM", "next_step": "CHECK_INVENTORY"}

优势：提供了明确的 JSON 结构和业务字段，AI 会强力模仿。

精讲：CoT (思维链) —— 拒绝“跳跃式”错误

Step 1: 观察原始问题

“如果我今天申请退房，且我是年付客户，我能退多少钱？”

Step 2: 显性逻辑链 (CoT)

“首先，查询当月剩余天数；其次，计算年付折扣是否需要收回；最后，根据公式计算总额。”

Step 3: 最终输出

基于上述逻辑，您的退款总额为 4520.5 元。

为什么 CoT 有效？

模型的神经网络擅长模式识别，但不擅长复杂的逻辑预判。 **“一步一步想”**（Let's think step by step）是通过把中间状态显性化，强制模型将计算资源分配给逻辑环节，从而极大降低计算幻觉。

精讲：上下文窗口 —— 消失的记忆

System: 你是客服... (最早的信息，面临溢出)
History 1: 用户询问房价...
History 2: 回复价格 5000...
Input: 那如果我只住半个月呢？ (当前最重要)

Context Window Limit

常见的坑

“我输入明明没超，为什么模型胡言乱语？”

真相：模型总算力窗口 = System + 历史轮次 + 你的输入 + 输出。

解决方案

摘要压缩、关键信息持久化、向量检索注入。

核心思想：只喂给 AI 最相关的那几口“粮食”。

精讲：节点 (Node) —— 工作流的原子

INPUT 层

输入变量

String / Number / File

LOGIC 层

处理节点

LLM / 代码执行 / API 转换

OUTPUT 层

输出变量

用于下游节点引用

设计准则：高内聚，低耦合

每一个节点只解决一个颗粒度的问题。如果一个 LLM 节点既要分析又要输出 JSON 还要做纠错，请拆解成三个节点。

精讲：变量作用域 —— 数据的可见性

G

全局变量 (Global)

在工作流的**任何一个节点**都能看到并引用。通常用于：

  • 用户初始输入 (UserInput)
  • API 配置 Token
  • 应用级环境变量
L

局部变量 (Local)

仅在**特定作用域内**有效。例如：

  • 循环内部的索引 (Index)
  • 某个 API 节点的中间返回
  • 条件判断的布尔值

⚠️ 同名变量冲突警告：尽量给局部变量加上节点命名前缀（如 `api_house_list`）以防止在后续引用时混淆。

分流逻辑：If / Else

接收咨询问题
投诉建议 ➔ 转人工
房源咨询 ➔ AI 回复

循环逻辑：Loop

核心：退出条件 (Exit Policy)

严禁设计没有上限的循环。必须配置“最大重试次数”或“目标达成判断”。

精讲：RAG —— 让 AI 读懂企业专有文档

1. 文档拆解 (Slicing)

2. 向量嵌入 (Embedding)

3. 向量检索 (Search)

什么是检索增强生成？

它是“翻书考试”。在生成答案前，先去外部大海中捞出那根针，再把针喂给 AI。

精讲：切片策略 —— 这里的坑最深

糟糕的切片

...退房规则如下：一、年付客户退押金应当...

--- 切片标记 (强制 200 字切断) ---

...提前 30 天申请。二、月付客户则不需要...

后果：当用户问“年付退押金怎么办”时，AI 只拿到了上半段，**完全丢失了“提前 30 天”这一关键条件**。

优秀的切片 (按语义归位)

安居乐寓业务切片 001:

描述：年付客户退房规则。
包含：退款公式、提前申请时限、违约条款。

原则：**切片大小必须包含完整的逻辑闭环**。如果知识库质量差，模型的推理能力再强也无法弥补。

精讲：ReAct 范式 —— Agent 的“灵魂循环”

THOUGHT (思考)
ACTION (行动)
OBSERVATION (观察)

为什么叫 ReAct？

Reasoning (推理) + Acting (行动)。
不仅像 CoT 一样思考，还要与外部工具发生真实的互动，并根据结果调整思考方向。

精讲：工具描述 —— 决定 AI “懂不懂”你

BAD

模糊描述

"description": "查询房源信息"

后果：当用户问“福田有没有房子”时，模型可能不调用此工具，或者不知道该传什么参数。

PRO

精准描述 (业务驱动)

"description": "当你需要查询安居乐寓在特定行政区、价格区间内的房源明细时，请调用此工具。它能返回包含房号、朝向、租金的完整列表。"

优势：模型明确了“何时用”以及“能拿回什么”，大幅降低调用幻觉。

精讲：JSON Schema —— Agent 的通信协议

TOOL DEFINITION (STRUCTURE)
{ "name": "search_apartments", "parameters": { "type": "object", "properties": { "district": { "type": "string", "enum": ["福田", "南山", "龙华"] }, "max_price": { "type": "number" }, "is_balcony": { "type": "boolean" } }, "required": ["district"] } }

Schema 的三大死穴

  • 类型声明 (Type)：String 绝不能传成 Number
  • 强制项 (Required)：模型必须生成这些参数
  • 枚举值 (Enum)：限定模型只能选这些分类

“如果 Schema 没写对，AI 会像一个拿错地图的司机，拼命想开车却去不了目的地。”

精讲：MCP —— AI 工具的“通用接口”

MODEL CONTEXT PROTOCOL

为什么我们要学协议？

以往，每个 AI 想要调用 Google 日历或 GitHub 都要单独写适配代码。 **MCP 是一套开放协议**，它让只要符合协议的任何工具 (Server)，都能被只要符合协议的任何模型 (Client，如 Claude Desktop) 一秒调用。

架构精髓

Server (工具提供方) ➔ Protocol (桥梁) ➔ Client (模型对话框)

翻转墙：常见误区大扫偏

MYTH 1

提示词越长就越准确？

真相：核心指令要前置。过长的干扰信息反而会稀释关注度。

MYTH 2

RAG 检索数量越多越好？

真相：Top 3-5 为佳。太多会引入噪音，导致模型得出自相矛盾的结论。

MYTH 3

给 Agent 所有工具最强大？

真相：过多选择会导致决策失灵。工具应当“按需精简”。

MYTH 4

循环是为了处理无限可能？

真相：没有退出条件的循环是毁灭性的。必须严设最大步数。

这就是工程师思维：承认边界，尊重逻辑
