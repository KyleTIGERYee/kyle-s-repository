# 第4章 知识库 - 参考资料

## 核心概念速查

### RAG核心概念

| 概念 | 定义 | 关键点 |
|-----|------|--------|
| **RAG** | Retrieval-Augmented Generation，检索增强生成 | 先检索，后生成 |
| **Embedding** | 嵌入，将文本转换为向量表示 | 语义压缩，便于相似度计算 |
| **Chunking** | 文档切片，将长文档切分成小块 | 切片策略直接影响检索质量 |
| **Vector DB** | 向量数据库，存储向量嵌入 | 支持相似度检索 |
| **ANN** | Approximate Nearest Neighbor，近似最近邻 | 加速大规模向量检索 |
| **KNN** | K-Nearest Neighbors，K近邻算法 | 找出最相似的K个向量 |

### 全量文档知识库核心概念

| 概念 | 定义 | 关键点 |
|-----|------|--------|
| **PageIndex** | 无向量推理式RAG框架 | No Vector DB, No Chunking |
| **M-R-E-R-G** | Map-Reflect-Explore-Read-Generate协议 | ajly-knowledge-base的核心工作流程 |
| **Tree Index** | 树状索引结构 | 模拟文档的层级结构 |
| **No Chunking** | 拒绝文档切片 | 保持文档完整性 |
| **Reasoning-based** | 基于推理的检索 | 用LLM推理代替相似度计算 |
| **Full Context** | 全量上下文 | 把完整文档喂给LLM |

## PageIndex vs M-R-E-R-G 对比详解

### 架构哲学对比

```
┌─────────────────────────────────────────────────────────────────┐
│                    传统RAG架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  文档 → 切片 → 嵌入 → 向量数据库 → 相似度检索 → Top K片段 → LLM   │
│                                                                 │
│  核心逻辑：相似度匹配                                            │
│  问题：切片导致信息丢失，相似度≠相关性                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    PageIndex架构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  文档 → 树状索引 → 推理导航 → 完整节点 → LLM                     │
│                                                                 │
│  核心逻辑：树搜索 + 推理                                         │
│  优势：无切片，可解释，类人检索                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    M-R-E-R-G架构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  查询 → Map检索 → Reflect反思 → Explore级联 → Read精读 → Generate │
│                                                                 │
│  核心逻辑：全景导航 + 依赖推理 + 级联探测 + 无损精读              │
│  优势：企业级复杂场景，跨模块关联                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 核心重合点

| 重合点 | PageIndex实现 | M-R-E-R-G实现 | 共同价值 |
|-------|--------------|--------------|---------|
| **No Chunking** | 按自然章节组织 | 强制完整阅读 | 信息无损失 |
| **Reasoning-based** | 树搜索推理 | Map+Reflect+Explore | 可解释、可追溯 |
| **Hierarchical** | 树状索引 | system_map.json | 结构化导航 |
| **Full Context** | 完整节点内容 | 完整文档内容 | 全局理解 |

## RAG失败点速查

### 七大失败点

| 失败点 | 英文 | 描述 | 典型案例 |
|-------|------|------|---------|
| **FP1** | Missing Content | 知识库中没有相关内容 | 问了一个不存在的问题 |
| **FP2** | Missed Top Ranked | 正确答案没排前面 | 相似度计算偏差 |
| **FP3** | Not in Context | 检索到了但没进上下文 | 被截断或过滤 |
| **FP4** | Not Extracted | 在上下文中但没提取 | LLM理解失败 |
| **FP5** | Wrong Format | 格式错误 | 要求JSON但返回文本 |
| **FP6** | Incorrect Specificity | 具体性不准 | 太笼统或太细节 |
| **FP7** | Incomplete | 不完整 | 遗漏部分信息 |

### 失败点根因分析

```
FP1 → 知识库覆盖度问题
   → 解决：完善知识库，增加边界检测

FP2 → 嵌入模型或检索算法问题
   → 解决：模型微调，混合检索

FP3 → 上下文长度或重排序问题
   → 解决：增加上下文长度，优化重排序

FP4 → LLM理解能力问题
   → 解决：提示工程，更强的LLM

FP5 → 提示工程问题
   → 解决：优化提示模板

FP6 → 检索或生成策略问题
   → 解决：调整检索参数，后处理

FP7 → 切片或综合推理问题
   → 解决：优化切片，多文档推理
```

## 决策框架速查

### 何时使用RAG

✅ **适合RAG的场景**：
- 文档数量 > 1000篇
- 查询类型为简单事实查找
- 对响应速度要求极高
- 已有成熟的向量数据库基础设施
- 团队有RAG工程经验

### 何时使用全量文档

✅ **适合全量文档的场景**：
- 文档数量 < 100篇
- 单篇文档 > 100页
- 查询涉及复杂流程和推理
- 需要跨文档关联分析
- 对精度要求极高，不能容忍切片损失
- 文档更新频繁

### 混合策略

```
大规模文档库（>1000篇）
    ↓
第一层：RAG快速筛选 → Top 10-20篇
    ↓
第二层：全量文档精读 → 完整理解
    ↓
生成答案
```

## 常见问题FAQ

### Q1: RAG和全量文档知识库可以共存吗？

**A**: 可以，推荐混合策略：
- 用RAG做第一层快速筛选（从大量文档中找到相关文档）
- 用全量文档做第二层深度理解（完整阅读筛选出的文档）

### Q2: 如果文档数量超过上下文限制怎么办？

**A**: 有以下几种方案：
1. 使用更大的上下文模型（如Gemini 2M）
2. 采用分层架构：先筛选文档，再精读
3. 使用PageIndex的树状导航，按需加载节点

### Q3: 全量文档知识库的响应速度会不会很慢？

**A**: 取决于几个因素：
- 文档读取速度：本地文件很快，远程存储有延迟
- LLM处理速度：长上下文确实更慢
- 优化方案：预加载、缓存、流式输出

### Q4: 如何评估知识库的效果？

**A**: 可以从以下几个维度评估：
- **准确率**：回答是否正确
- **召回率**：是否遗漏了相关信息
- **完整性**：回答是否全面
- **可追溯性**：能否追溯到信息来源

### Q5: 企业级知识库还需要向量数据库吗？

**A**: 视情况而定：
- 如果文档数量 < 1000，可以尝试纯全量文档方案
- 如果文档数量 > 1000，建议保留向量数据库做第一层筛选
- 未来随着上下文窗口增大，向量数据库的角色可能会弱化
