## 第二部分：核心概念速查表（25分钟）

### 2.1 第一章核心概念

| 概念 | 一句话定义 | 关键要点 | 常见误区 |
|-----|-----------|---------|---------|
| **Token** | 大模型处理文本的最小单位 | 1个汉字≈1.5 tokens，英文单词≈1 token | 以为1个字=1个token |
| **Temperature** | 控制输出随机性的参数 | 0=确定性，1=创造性，代码用0，创意用0.7-1 | 所有场景都用默认值0.7 |
| **System Prompt** | 给AI的"角色说明书" | 定义身份、能力边界、行为准则 | 写得太长，核心指令被淹没 |
| **Few-shot** | 给AI看几个示例再让它做 | 示例要典型，输入输出格式一致 | 示例质量差，反而误导模型 |
| **CoT** | Chain of Thought，思维链 | 让AI"一步一步想"，提高推理准确性 | 简单问题也用CoT，浪费token |
| **上下文窗口** | 模型能"记住"的最大文本量 | 包括System+History+Input+Output | 只算输入，忘记算历史记录 |

### 2.2 第二章核心概念

| 概念 | 一句话定义 | 关键要点 | 常见误区 |
|-----|-----------|---------|---------|
| **节点（Node）** | 工作流中的单个处理单元 | 输入→处理→输出，有明确的输入输出定义 | 节点之间数据类型不匹配 |
| **变量（Variable）** | 工作流中传递数据的容器 | 注意作用域：全局变量 vs 局部变量 | 同名变量覆盖，导致数据混乱 |
| **条件分支（If/Else）** | 根据条件走不同路径 | 条件要写完整，考虑边界情况 | 只写if不写else，遗漏分支 |
| **循环（Loop）** | 重复执行直到满足条件 | 一定要有退出条件，避免死循环 | 忘记更新循环变量，导致死循环 |
| **知识检索（Retrieval）** | 从知识库找相关内容 | 需要配置切片大小、检索数量、相似度阈值 | 检索结果太多，超出上下文限制 |
| **RAG** | 检索增强生成 | 检索→注入上下文→生成回答 | 以为RAG万能，不看知识库质量 |

### 2.3 第三章核心概念

| 概念 | 一句话定义 | 关键要点 | 常见误区 |
|-----|-----------|---------|---------|
| **Agent** | 能自主决策并使用工具的AI | 观察→思考→行动→再观察的循环 | 把Agent当成简单的API调用 |
| **ReAct** | Reasoning + Acting 范式 | 让AI"先想后做"，边做边想 | 不给AI思考空间，直接要答案 |
| **Tool/Function** | Agent可调用的外部能力 | 要有清晰的描述，让AI知道什么时候用 | 描述模糊，AI不知道何时调用 |
| **Function Calling** | 大模型输出结构化调用指令 | 定义schema，模型输出JSON格式调用 | schema定义不清，解析失败 |
| **Skill** | 封装好的可复用Agent能力 | 包含：描述 + 参数定义 + 实现逻辑 | 和Tool概念混淆，其实Skill是更上层的封装 |
| **MCP** | Model Context Protocol | 标准化的工具调用协议（如Claude的MCP） | 以为MCP是特定技术，其实是通用概念 |

### 2.4 关键概念对比

#### Prompt Engineering vs 工作流 vs Agent

| 维度 | Prompt Engineering | 工作流（Workflow） | Agent |
|-----|-------------------|-------------------|-------|
| **控制方式** | 单次对话，人工控制 | 预定义流程，自动执行 | 自主决策，动态调整 |
| **适用场景** | 简单问答、内容生成 | 固定流程的任务 | 复杂、需要灵活决策的任务 |
| **复杂度** | 低 | 中 | 高 |
| **可预测性** | 高 | 高 | 中（有随机性） |
| **典型应用** | 文案生成、代码辅助 | 审批流程、数据处理 | 智能客服、研究助手 |

#### Tool vs Skill vs Workflow

| 维度 | Tool（工具） | Skill（技能） | Workflow（工作流） |
|-----|------------|--------------|-------------------|
| **层级** | 最底层原子能力 | 中层封装 | 上层编排 |
| **复用性** | 可被Skill和Workflow调用 | 可被Agent和Workflow调用 | 独立运行或被调用 |
| **包含内容** | 函数实现 + Schema定义 | 描述 + 参数 + 内部逻辑 | 节点 + 连线 + 逻辑 |
| **典型示例** | 天气查询API、计算器 | "查询安居乐寓知识库" | "AI找房助手完整流程" |

## 第四部分：常见误区澄清（10分钟）

### 4.1 提示词工程误区

| 误区 | 正确理解 |
|-----|---------|
| "提示词越长越好" | 精准 > 冗长，核心指令要前置 |
| "一个提示词解决所有问题" | 复杂任务要拆解，多轮对话比单轮长提示更有效 |
| "Temperature越高越有创意" | 创意任务0.7-0.9足够，太高会导致输出不稳定 |
| "Few-shot示例越多越好" | 3-5个高质量示例 > 10个低质量示例 |

### 4.2 工作流误区

| 误区 | 正确理解 |
|-----|---------|
| "工作流节点越多越好" | 节点多≠能力强，简洁的流程更易维护 |
| "变量可以随便命名" | 统一命名规范（如camelCase），避免冲突 |
| "循环不需要退出条件" | 必须设置明确的退出条件，防止死循环 |
| "知识检索结果越多越好" | Top 3-5足够，太多会超出上下文限制 |

### 4.3 Agent误区

| 误区 | 正确理解 |
|-----|---------|
| "Agent就是调用API" | Agent的核心是**自主决策**，不是简单封装 |
| "Tool越多越好" | 给Agent太多工具会降低决策准确性 |
| "ReAct就是多轮对话" | ReAct是"思考→行动→观察"的循环，不是简单的多轮 |
| "Skill和Tool是一回事" | Skill是更高层的封装，可以包含多个Tool和逻辑 |

### 4.4 概念辨析快问快答

**Q1：CoT和ReAct有什么区别？**
- CoT：让模型"一步一步想"（纯思考）
- ReAct：让模型"边想边做"（思考+行动+观察）

**Q2：工作流和Agent怎么选？**
- 流程固定、步骤明确 → 工作流
- 需要灵活决策、多步推理 → Agent

**Q3：RAG和知识库是什么关系？**
- 知识库 = 存储文档的地方
- RAG = 使用知识库的方法（检索+生成）
- 第四章会学另一种方法：全量文档

**Q4：Tool和Function Calling是什么关系？**
