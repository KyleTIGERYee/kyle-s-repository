# 文档生成质量检查清单

本文档定义生成功能说明文档后的质量检查标准。

---

## 产出物完整性首检

生成新文档时，**严禁缺少以下物理交付环节**：

- [ ] 符合安居乐寓行文规范的完整 Markdown 文档实体
- [ ] 根据分类，文档必须存放在正确的目录下：`references/documents/features/`（业务级迭代） 或 `references/documents/modules/`（系统底层模块手册）

**⚠️ 强调红线：文档未落盘，且未触发建图，均视为工作未完成**

---

## 结构检查

- [ ] 包含"概述"章节
- [ ] 包含"操作指引"或"功能模块"章节
- [ ] 章节顺序符合规范（概述→功能→权限→配置→对接）
- [ ] 标题层级正确（一级#、二级##、三级###）
- [ ] 文档开头重复标题（纯文本）

---

## 术语检查

- [ ] 使用"PC端"而非"电脑端/web端"
- [ ] 使用"小程序"而非"微信端/移动端"
- [ ] 使用"租客"指代租房客户
- [ ] 使用"管家"指代一线业务人员
- [ ] 使用"店长"指代门店负责人
- [ ] 术语使用一致（不混用"租客"和"客户"）
- [ ] 首次出现的专业术语适当解释

---

## 格式检查

- [ ] 按钮名称使用粗体（**提交**、**保存**）
- [ ] 菜单路径使用【】（【字典设置】-【租客字典】）
- [ ] 界面提示文字使用""（"操作成功"）
- [ ] 字段名使用代码块（`userPhone`）
- [ ] 配置项名称使用粗体（**参数名称**）
- [ ] 列表符号统一（无序用-，有序用数字）
- [ ] 表格格式正确（|分隔，表头分隔线）

---

## 内容检查

- [ ] 语言风格正式、专业（不用口语化表达）
- [ ] 数字使用阿拉伯数字（5天、30天）
- [ ] 金额格式统一（300元、100元）
- [ ] 日期格式统一（YYYY-MM-DD）
- [ ] 流程描述清晰（步骤明确、逻辑连贯）
- [ ] 包含必要的注意事项
- [ ] 权限说明完整（如适用）
- [ ] 配置说明清晰（如适用）

---

## 完整性与防御性检查

- [ ] 概述章节说明功能目的和范围
- [ ] 操作步骤包含前置条件
- [ ] 异常情况有处理说明
- [ ] 凡与周边财管 / 审管联动的，相关关联系统有明显文字抛出，以此诱导未来阅读大模型触发级联追溯机制。

---

## 常见错误

| 错误类型 | 错误示例 | 正确示例 |
|----------|----------|----------|
| 术语不规范 | 电脑端 | PC端 |
| 格式错误 | 点击提交按钮 | 点击**提交**按钮 |
| 菜单路径错误 | 字典设置-租客字典 | 【字典设置】-【租客字典】 |
| 缺少章节 | 直接写功能点 | 先写#概述 |
| 口语化 | 用户可以点这里 | 点击**按钮** |

---

## 检查通过标准

以上所有 Checklist 检查项目未被触发驳回，方可宣布本次发版撰写通过。如有未通过的单项，必须返工对 Markdown 源文件打磨重写修补。

---

## 终态交付动作检查

完成所有撰写审查后，你必须确认收尾动作执行到位：

1. **核心产物落盘归档** - 按规范写入 `.md` 文件（存入 features/ 或 modules/）。
2. **大树感官刷新 (非常关键)** - 确保终端调度系统已经执行入库脚本（注意：使用 SKILL.md 中定义的 `SKILL_ROOT` 绝对路径）：
   调用 `python3 <SKILL_ROOT>/scripts/update_node.py` 将文档的摘要和关键词挂载进大地图。
   只有脚本跑完，系统大脑地图才会接纳这篇新鲜出炉的文章。
3. **输出完工报告** - 向用户播报你写了什么、存在哪里，以及已经刷新了地图。
