## 1.1 工作流概念

### 1.1.1 什么是工作流

**工作流（Workflow）**是指业务过程的部分或整体在计算机应用环境下的自动化，是对工作流程及其各操作步骤之间业务规则的抽象和描述。

**通俗理解**：

想象你在工厂的生产线上，每个工人负责一道工序，产品按顺序经过各道工序最终完成。工作流就是把这个过程数字化，让AI按预设的步骤自动完成任务。

> "Dify 工作流（即 Dify 的 Workflow）是 Dify 平台最核心的功能之一，它在 AI 应用中负责 流程编排（orchestration） —— 让多个模型调用、工具调用、逻辑节点、变量处理协同工作。"
> 
> —— 来源：《dify工作流的原理》

### 1.1.2 为什么需要工作流

**Prompt Engineering的局限性**：

> "prompt 工程，看上去像去找一个人，让他从头到尾一次性写完一篇文章，其实挺难的。而agentic workflow 却友好的多:1，投喂一个文本，先写一个大纲"
> 
> —— 来源：《用AI Agent工作流思维替代一下prompt 工程思维?》

| 场景 | 仅用Prompt | 使用工作流 |
|-----|-----------|-----------|
| 简单问答 | ✅ 适合 | ⚠️ 过度设计 |
| 多步骤任务 | ❌ 难以控制 | ✅ 清晰可控 |
| 条件分支 | ❌ 不稳定 | ✅ 精确控制 |
| 多模型协作 | ❌ 难以实现 | ✅ 天然支持 |
| 数据处理 | ❌ 容易出错 | ✅ 稳定可靠 |

**工作流的核心优势**：

1. **可控性**：每个步骤都精确可控
2. **稳定性**：避免AI"自由发挥"导致的不确定性
3. **可复用**：一次搭建，多次使用
4. **可维护**：逻辑清晰，易于调试和修改
5. **可协作**：多节点、多模型协同工作

### 1.1.3 AI应用三大支柱

> "他们分别对应不同的业务场景，比如:Workflow:HR提效(简历筛选、身份证录入);RAG:AI客服;Agent:复杂任务(根据问题自动搜索网址后生成材料如PPT、博客);"
> 
> —— 来源：《梳理AI应用三大支柱:Workflow、RAG、Agent》

| 技术 | 核心能力 | 典型场景 |
|-----|---------|---------|
| **Workflow** | 流程编排、自动化 | 简历筛选、数据录入、审批流程 |
| **RAG** | 知识检索、问答 | 智能客服、知识库问答 |
| **Agent** | 自主决策、工具调用 | 复杂任务执行、多步骤规划 |

**三者关系**：

```
┌─────────────────────────────────────────┐
│              AI应用系统                  │
├─────────────────────────────────────────┤
│  Agent层  │  自主决策、任务规划、工具选择  │
├─────────────────────────────────────────┤
│ Workflow层│  流程编排、节点执行、数据流转  │
├─────────────────────────────────────────┤
│   RAG层   │  知识检索、上下文增强         │
├─────────────────────────────────────────┤
│   LLM层   │  语义理解、文本生成           │
└─────────────────────────────────────────┘
```

### 1.1.4 主流工作流平台

> "在这样的背景下，工作流式(Workflow-based)的Agent构建平台应运而生，并迅速成为主流。无论是开源的 n8n、Dify，还是字节跳动推出的 Coze它们不约而同地采用了相似的模型。"

> —— 来源：《AI Agent(Workflow)工作流平台技术原理分析》

| **n8n** | 开源 | 传统工作流+AI，集成能力强 | 业务流程自动化 |

## 1.2 工作流核心原理

### 1.2.1 DAG有向无环图

> "画布上的每一个工作流，本质上是一个有向无环图(DAG, Directed Acyclic Graph)。"
> 
> —— 来源：《AI Agent(Workflow)工作流平台技术原理分析》

**什么是DAG？**

DAG（Directed Acyclic Graph）是有向无环图的缩写：
- **有向（Directed）**：节点之间的连接有方向，A→B表示从A到B
- **无环（Acyclic）**：图中不存在循环路径，不会A→B→C→A
- **图（Graph）**：由节点和边组成的结构

**为什么工作流必须是DAG？**

1. **避免死循环**：确保工作流能正常结束
2. **确定执行顺序**：明确每个节点的执行先后
3. **便于调试**：问题定位更清晰

**DAG示例**：

```
┌─────────┐    ┌─────────┐    ┌─────────┐
│  开始   │───→│  处理A  │───→│  处理B  │
└─────────┘    └────┬────┘    └────┬────┘
                    │              │
                    ↓              ↓
               ┌─────────┐    ┌─────────┐
               │  处理C  │←───│  判断   │
               └────┬────┘    └─────────┘
                    │
                    ↓
               ┌─────────┐
               │  结束   │
               └─────────┘
```

### 1.2.2 工作流 vs 提示词工程

| 对比维度 | 提示词工程 | 工作流 |
|---------|-----------|--------|
| **复杂度** | 适合简单任务 | 适合复杂任务 |
| **可控性** | 依赖模型能力 | 精确控制每个步骤 |
| **稳定性** | 输出可能不稳定 | 输出稳定可预期 |
| **调试难度** | 难以定位问题 | 可逐节点调试 |
| **开发成本** | 低 | 中等 |
| **维护成本** | 低 | 中等 |

**选择建议**：

- **简单任务**（单轮问答、文本生成）：用提示词工程
- **复杂任务**（多步骤、条件判断、数据处理）：用工作流
- **极复杂任务**（需要自主决策）：用Agent

## 1.3 参数提取与结构化输出


### 1.3.1 什么是参数提取

**参数提取**是将用户自然语言转化为结构化数据的过程，让AI从文本中识别出关键信息：

```
用户输入: "我想在西乡租个单间，预算3000左右"
                ↓
           LLM 提取
                ↓
结构化输出: {"area": "西乡", "layoutName": "单间", "minPrice": "0", "maxPrice": "3000"}
```

**在租房场景中**：

| 用户表达 | 提取的参数 |
|---------|-----------|
| "我想租一个单间，预算1000" | {layoutName:单间, minPrice:0, maxPrice:1000} |
| "我想在西乡租房" | {area:西乡} |
| "预算3000以内" | {minPrice:0, maxPrice:3000} |
| "宝安区一房一厅" | {area:宝安区, layoutName:一房一厅} |

### 1.3.2 结构化输出

**结构化输出**让AI返回JSON格式的数据，便于后续节点处理：

```json
{
  "area": "西乡",
  "layoutName": "单间",
  "minPrice": "0",
  "maxPrice": "3000"
}
```

> **说明**：结构化输出是LLM节点的核心技能，通过配置输出格式，LLM可以稳定地输出我们需要的JSON结构。

## 1.4 节点功能详解

> "包括start/end node、llm node、if else node、knowledge retrieval node、question classifier node、tool node、http node、loop node、variable assigner node等等,基本覆盖了ai应用开发业务编排所需要的全部语义表达。"
> 
> —— 来源：《以Dify架构设计为例，一篇文看懂AI原生应用开发平台》

### 1.4.1 输入节点（Start/Input）

**功能**：接收用户输入或外部数据，作为工作流的起点。

**配置项**：
- 输入参数定义（名称、类型、默认值）
- 参数校验规则
- 输入提示信息

**Dify vs Coze对比**：

| 特性 | Dify | Coze |
|-----|------|------|
| 输入参数定义 | ✅ 支持多参数 | ✅ 支持多参数 |
| 默认值 | ✅ 支持 | ✅ 支持 |
| 系统变量 | {{userinput.query}} | {{USER_INPUT}} |

### 1.4.2 大模型节点（LLM）

**功能**：调用大语言模型进行推理和生成。

**配置项**：
- 选择模型（通义千问等）
- 设置提示词（Prompt）
- 配置Temperature等参数
- 定义输入变量
- 定义输出格式（结构化输出）

**Dify vs Coze对比**：

| 特性 | Dify | Coze |
|-----|------|------|
| 模型选择 | 多模型支持 | 多模型支持 |
| 结构化输出 | ✅ JSON模式 | ✅ 结构化输出 |

### 1.4.3 变量节点（Variable/Assigner）

**功能**：存储和传递中间结果，在工作流中传递数据。

**使用场景**：
- 保存LLM的输出结果
- 在不同节点间传递数据
- 临时存储计算结果

**变量类型**：

| 类型 | 说明 | 示例 |
|-----|------|------|
| String | 字符串 | "西乡" |
| Number | 数字 | 3000 |
| Boolean | 布尔值 | true/false |
| Object | 对象 | {"area": "西乡", "layoutName": "单间"} |
| Array | 数组 | ["西乡", "南山", "福田"] |

### 1.4.4 条件判断节点（Condition/If-Else）

**功能**：实现分支逻辑，根据条件决定执行路径。

**配置项**：
- 条件表达式
- True分支
- False分支

**示例场景**：

```
用户预算判断：
  IF layoutName == "开间":
    → 推荐单间房源
  ELSE IF layoutName == "一居室":
    → 推荐一居室房源
```

### 1.4.5 输出节点（End/Output）

**功能**：返回工作流执行结果。

> "coze的workflow提供了两种输出模式:
一,返回一些变量值,然后让聊天模型基于这些变量值回复用户;
二,在workflow里拼接好输出..."
> 
> —— 来源：《Coze制作信息检索Bot中Workflow的输入输出设置》

**输出模式**：

**模式一：返回变量**
- 工作流返回结构化数据
- 由AI根据数据生成回复
- 适合需要进一步处理的场景

**模式二：直接输出**
- 工作流直接生成完整回复
- 适合简单场景
- 响应速度更快

**示例配置**：

```yaml
# 模式一：返回变量
输出节点:
  类型: variables
  输出:
    - name: recommendations
      value: {{llm_result}}
    - name: count
      value: {{item_count}}

# 模式二：直接输出
输出节点:
  类型: text
  内容: |
    为您找到{{count}}套房源：
    {{recommendations}}
```

## 1.5 实战：搭建参数提取工作流

### 1.5.1 场景说明

**业务需求**：
搭建一个能从用户自然语言中**提取找房参数**的工作流，实现结构化输出。

> **本课核心**：第1课聚焦于LLM的**参数提取**和**结构化输出**能力，这是工作流中最基础、最常用的技能。

**本课实战目标**：
- 接收用户自然语言输入
- 使用LLM从自然语言中提取找房参数
- 输出结构化JSON结果
- 掌握结构化输出的配置方法

**为什么先学参数提取？**
- 参数提取是工作流的基础能力，几乎所有业务场景都会用到
- 结构化输出让LLM的结果可以被后续节点稳定处理
- 意图识别、知识库检索等进阶功能将在后续课程学习

### 1.5.2 工作流设计

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  输入   │───→│  LLM    │───→│ 变量赋值│───→│  输出   │
│ (用户  │    │ (参数   │    │ (保存  │    │ (返回  │
│  查询)  │    │  提取)  │    │  结果) │    │  JSON) │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
```

### 1.5.3 节点配置详解

**节点1：输入节点**

```yaml
# Dify配置
节点类型: Start
节点ID: start_node

配置:
  输入参数:
    - 名称: user_query
      类型: string
      描述: 用户查询内容
      必填: true
      示例: "我想在西乡租个单间，预算3000左右"
```

**节点2：LLM节点（参数提取）**

```yaml
# Dify配置
节点类型: LLM
节点ID: param_extractor

配置:
  模型: qwen-flash (通义千问)
  
  提示词: |
    你是一个租房参数提取助手。请从用户的找房需求中提取以下参数：
    
    用户需求：{{start_node.user_query}}
    
    需要提取的参数：
    - area: 区域（如：宝安区、西乡、龙华区）
    - layoutName: 户型（如：单间、一房一厅、两房一厅）
    - minPrice: 最低预算（数字，单位：元）
    - maxPrice: 最高预算（数字，单位：元）
    
    请以JSON格式返回：
    {
      "area": "提取的区域，未提及则为空字符串",
      "layoutName": "提取的户型，未提及则为空字符串",
      "minPrice": "最低预算，未提及则为空字符串",
      "maxPrice": "最高预算，未提及则为空字符串"
    }
    
    注意：
    1. 只返回JSON，不要有其他内容
    2. 如果用户说"预算3000左右"，minPrice设为"0"，maxPrice设为"3000"
    3. 如果用户说"3000以内"，minPrice设为"0"，maxPrice设为"3000"
  
  Temperature: 0.1
  
  结构化输出:
    schema:
      type: object
      properties:
        area:
          type: string
          description: 区域
        layoutName:
          type: string
          description: 户型
        minPrice:
          type: string
          description: 最低预算
        maxPrice:
          type: string
          description: 最高预算
      required:
        - area
        - layoutName
        - minPrice
        - maxPrice
```

**节点3：变量节点**

```yaml
# Dify配置
节点类型: Assigner
节点ID: save_result

配置:
  变量赋值:
    - 名称: extracted_area
      值: {{param_extractor.area}}
    - 名称: extracted_layoutName
      值: {{param_extractor.layoutName}}
    - 名称: extracted_minPrice
      值: {{param_extractor.minPrice}}
    - 名称: extracted_maxPrice
      值: {{param_extractor.maxPrice}}
```

**节点4：输出节点**

```yaml
# Dify配置
节点类型: End
节点ID: output_result

配置:
  输出模式: 直接输出
  
  输出内容:
    - 名称: area
      值: {{save_result.extracted_area}}
    - 名称: layoutName
      值: {{save_result.extracted_layoutName}}
    - 名称: minPrice
      值: {{save_result.extracted_minPrice}}
    - 名称: maxPrice
      值: {{save_result.extracted_maxPrice}}
    - 名称: status
      值: "success"
```

## 1.6 实战练习

### 练习1：参数提取工作流搭建

**任务**：搭建一个"租房参数提取"工作流

**要求**：
1. 接收用户自然语言输入
2. 使用LLM节点提取：区域（area）、户型（layoutName）、预算（minPrice/maxPrice）
3. 使用变量节点保存结果
4. 输出结构化JSON

**测试用例参考**：

| 用户输入 | 预期提取 |
|---------|---------|
| 我想租一个1000元左右的单间 | {layoutName: 单间, minPrice: 0, maxPrice: 1000} |
| 我想在宝安区租房 | {area: 宝安区} |
| 有没有一房一厅推荐 | {layoutName: 一房一厅} |
| 预算3000以内 | {minPrice: 0, maxPrice: 3000} |
| 西乡两房一厅，预算5000左右 | {area: 西乡, layoutName: 两房一厅, minPrice: 0, maxPrice: 5000} |

**评分标准**：
- 节点连接正确（30%）
- 提示词设计合理（30%）
- 结构化输出配置正确（25%）
- 能处理边界情况（15%）

3. "一千多" - 预算应该如何处理？

**要求**：
1. 针对上述问题优化提示词
2. 测试优化前后的效果对比
3. 记录优化过程

**输出**：
- 优化前的提示词
- 优化后的提示词
- 测试对比结果

### 练习3：扩展参数提取

**任务**：在基础工作流上扩展更多参数

**新增参数**：
- distance: 距离（如：离地铁500米）
- storeNames: 门店名称（如：安居乐寓西乡店）

**要求**：
1. 更新LLM节点的提示词
2. 更新变量节点和输出节点

## 1.7 本课小结

### 核心概念回顾

1. **工作流（Workflow）**：业务流程的自动化编排
2. **DAG**：有向无环图，工作流的底层结构
3. **参数提取**：将自然语言转化为结构化数据
4. **结构化输出**：让LLM返回JSON格式，便于后续节点处理
5. **节点（Node）**：工作流的基本单元
6. **边（Edge）**：节点间的连接和数据流向

### 基础节点类型

| 节点 | 功能 | 使用场景 |
|-----|------|---------|
| 输入节点 | 接收数据 | 工作流起点 |
| LLM节点 | AI推理 | 文本处理、信息提取、参数提取 |
| 变量节点 | 数据传递 | 节点间数据流转 |
| 条件节点 | 分支判断 | 逻辑控制 |
| 输出节点 | 返回结果 | 工作流终点 |

### 关键技能

- 理解工作流与提示词工程的区别和适用场景
- 掌握DAG的基本概念
- 能够搭建包含3-5个节点的基础工作流
- **熟练使用LLM进行参数提取和结构化输出**
- 能够设计合理的提示词提高提取准确率

### 常见问题

**Q1：结构化输出不稳定怎么办？**
- 检查JSON Schema是否正确
- 在提示词中强调"只返回JSON，不要有其他内容"
- 尝试调低Temperature（建议0.1-0.3）

**Q2：参数提取不准确怎么办？**
- 在提示词中增加更多示例
- 明确说明每个参数的提取规则
- 处理好"未提及"的情况

**Q3：Coze和Dify配置有什么区别？**
- 变量引用语法不同：Dify用`{{node.variable}}`，Coze用`{{VARIABLE}}`
- 结构化输出配置界面略有差异
- 核心逻辑相同，学会一个可以迁移
