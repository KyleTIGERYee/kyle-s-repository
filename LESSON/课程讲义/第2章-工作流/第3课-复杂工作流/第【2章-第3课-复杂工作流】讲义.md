## 一、课程导入（10分钟）

### 1.1 从简单到复杂

在前两节课中，我们已经学习了：
- **基础节点**：输入、LLM、变量、条件、输出
- **进阶技能**：意图识别、知识库检索、代码节点

这些技能已经能够帮助我们构建一个简单的智能客服。但实际业务场景往往更加复杂，需要更多的功能来提升用户体验。

**场景一：多意图路由**
> 用户可能会说"帮我找宝安区的房子，我想约第一套看房"，这包含了两个意图：
> - search_house：找宝安区房源
> - find_butler：预约看房

**场景二：状态管理**
> 用户说"太远了，换个福田的吧"，需要理解这是在修改之前搜索的区域参数，需要更新搜索状态。

**场景三：空结果处理**
> 用户搜索条件过于苛刻没有房源时，不能简单说"没有"，而是应该智能推荐其他方案。

## 三、对话状态管理（15分钟）

### 3.1 为什么需要状态管理

在多轮对话中，需要记住用户之前提供的信息：

| 对话轮次 | 用户输入 | 提取参数 | 状态更新 |
|---------|---------|---------|---------|
| 第1轮 | "我想在宝安区租房" | {area:宝安区} | {area:宝安区} |
| 第2轮 | "预算2000左右" | {budget:2000} | {area:宝安区,budget:2000} |
| 第3轮 | "有没有一房一厅" | {room_type:一房一厅} | {area:宝安区,budget:2000,room_type:一房一厅} |
| 第4轮 | "太远了，换个福田" | {area:福田区} | {area:福田区,budget:2000,room_type:一房一厅} |

### 3.2 Dify对话变量

Dify提供对话变量（Conversation Variables）来保存状态：

```yaml
对话变量配置:
  - 名称: search_state
    类型: string
    描述: 存储用户的搜索条件
    示例: '{"area":"宝安区","budget":2000}'
```

### 3.3 代码节点实现状态管理

```python
# State Manager 代码节点
import json

def main(router_output, current_state_str):
    """
    状态管理器：解析LLM输出，管理搜索状态
    """
    
    # 1. 解析LLM输出
    router_data = {}
    try:
        if isinstance(router_output, dict):
            router_data = router_output
        else:
            raw_text = str(router_output)
            start_idx = raw_text.find('{')
            end_idx = raw_text.rfind('}')
            if start_idx != -1 and end_idx != -1:
                json_str = raw_text[start_idx:end_idx+1]
                router_data = json.loads(json_str)
    except Exception as e:
        router_data = {"intent": "chat", "params_update": {}}
    
    # 2. 提取数据
    intent = router_data.get("intent", "chat")
    new_params = router_data.get("params_update", {})
    
    # 3. 读取旧状态
    state = {}
    if current_state_str:
        try:
            state = json.loads(str(current_state_str))
        except:
            state = {}
    
    # 4. 智能合并逻辑
    # 如果切换了区域(area)，则清空门店(storeNames)
    if new_params.get("area") and new_params["area"] != state.get("area"):
        state["storeNames"] = ""
        state["address"] = ""
    
    # 更新参数
    for key, value in new_params.items():
        if value is not None and str(value) != "":
            state[key] = value
    
    # 5. 准备API Payload
    api_payload = state.copy()
    if intent == "search_house":
        if "minPrice" not in api_payload:
            api_payload["minPrice"] = "0"
        if "maxPrice" not in api_payload:
            api_payload["maxPrice"] = "10000"
    
    # 6. 返回结果
    return {
        "intent": intent,
        "new_state_str": json.dumps(state, ensure_ascii=False),
        "area": state.get("area", ""),
        "budget": state.get("budget", ""),
        "room_type": state.get("room_type", "")
    }
```

### 4.3 API结果解析

```python
# 解析搜房API返回的JSON
import json

def main(response_data):
    try:
        data_dict = json.loads(response_data)
        
        # 获取房源列表
        store_list = data_dict.get('data', {}).get('list', [])
        
        if not store_list:
            return {"result": "抱歉，当前条件下暂无匹配房源"}
        
        # 提取前3套房源
        actual_display_list = store_list[:3]
        
        # 生成推荐文案
        output_parts = ["为您精选了以下几套高性价比房源！\n"]
        
        for i, store in enumerate(actual_display_list):
            store_name = store.get("storeName", "未知门店")
            address = store.get("storeAddress", "暂无地址")
            price = store.get("layoutList", [{}])[0].get("price", 0)
            
            output_parts.append(
                f"{i+1}. {store_name}\n"
                f"   价格：{int(price)}元/月起\n"
                f"   地址：{address}\n"
            )
        
        output_parts.append("\n看中哪套可以直接对我说呢！")
        
        return {"result": "\n".join(output_parts)}
        
    except Exception as e:
        return {"result": f"数据处理异常: {str(e)}"}
```

## 五、空结果处理（Recovery机制）（15分钟）

### 5.1 为什么需要Recovery

当用户搜索条件过于苛刻时，API可能返回空结果。此时不能简单告诉用户"没有"，而应该：
1. 分析原因（区域饱和？预算过低？户型不匹配？）
2. 提供建议（换个区域？增加预算？换户型？）
3. 引导用户尝试

### 5.2 空结果分析工作流

```
搜索结果为空
      │
      ▼
┌─────────────────┐
│  Search Analyst │
│ (LLM: 分析原因) │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  原因诊断       │
├─────────────────┤
│ • location_    │
│   saturated    │ → 建议换区域
│ • budget_      │
│   too_low      │ → 建议加预算
│ • layout_      │
│   mismatch     │ → 建议换户型
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Recovery Agent │
│ (LLM: 生成回复) │
└────────┬────────┘
         │
         ▼
用户： "南山区没房，建议看看宝安区，性价比更高！"
```

### 5.3 空结果分析提示词

```python
# Search Analyzer 提示词
"""
# Role
Search Diagnostic Expert (搜索诊断专家)

# Context
用户进行了搜房，但系统返回了空结果。

# Knowledge Base
- 房型范围：仅有开间、一居室、二居室、宿舍
- 租赁模式：仅整租
- 区域迁徙逻辑：
  - 南山/福田没房 -> 推荐宝安/龙华
  - 罗湖/盐田没房 -> 推荐布吉/龙岗

# Thinking Protocol
1. 条件解析：提取area、budget、layoutName
2. 病因诊断：
   - 库存/区域限制 -> 建议换区域
   - 业务不符(合租) -> 建议换户型
   - 预算过低 -> 建议提预算
3. 策略制定（严格优先级）：
   - Priority 1: 换区域（默认首选）
   - Priority 2: 换户型
   - Priority 3: 提预算（最后手段）

# Output Schema
{
  "diagnosis": "location_saturated | budget_too_low | business_mismatch",
  "suggestion_strategy": {
    "action": "change_area | change_layout | increase_budget",
    "value": "建议的具体值"
  }
}
"""
```

### 学习建议

1. **先完成基础作业**：确保掌握完整工作流后再挑战创意实践
2. **选题建议**：进阶作业建议选择自己熟悉的业务场景，便于设计意图和测试用例
3. **分步调试**：每完成一个节点就进行测试，避免最后排查困难
4. **关注边界情况**：用户输入往往不按套路出牌，多测试异常输入
5. **善用日志**：开启Dify的调试模式，查看每个节点的输入输出

### 常见问题

**Q1：意图识别不准确怎么办？**
- 检查提示词中的意图定义是否清晰
- 增加few-shot示例帮助模型理解
- 调低Temperature（建议0.1-0.3）
- 在提示词中强调"无法归类时使用chat"

**Q2：状态管理丢失数据怎么办？**
- 确认对话变量已正确创建
- 检查代码节点的JSON解析逻辑
- 注意空值处理，避免覆盖已有状态

**Q3：API调用失败如何排查？**
- 检查URL、Headers、Body格式是否正确
- 确认认证信息是否有效
- 查看Dify运行日志中的错误信息
- 使用Postman先测试API是否可调用

**Q4：Recovery机制如何设计优先级？**
- 换区域 > 换户型 > 提预算（用户接受度从高到低）
- 基于业务数据调整推荐策略
- 保持建议的友好性和引导性

**Q5：进阶作业如何选择选题？**
- 选择自己熟悉的业务场景
- 考虑数据获取的可行性
- 从简单意图开始，逐步扩展功能

## 附录：常见问题解答

### Q1：意图识别和参数提取哪个更重要？

**A**：两者同样重要。意图识别决定处理流程，参数提取决定搜索结果质量。都需要精心设计。

### Q2：对话状态管理怎么做？

**A**：使用平台提供的对话变量功能，结合代码节点进行状态读写。

### Q3：如何处理API调用失败？

**A**：
1. 设置重试机制
2. 添加fail-branch处理
3. 返回友好提示

### Q4：Recovery机制如何设计？

**A**：
1. 分析空结果原因
2. 根据优先级给出建议
3. 用友好的话术表达
