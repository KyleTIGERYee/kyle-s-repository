## 二、复杂Agent架构设计（25分钟）

### 2.1 架构演进路径

#### 阶段一：单体Agent（Monolithic Agent）

```
┌─────────────────────────────────────┐
│           单体Agent架构              │
├─────────────────────────────────────┤
│  用户输入 → LLM → 工具调用 → 输出   │
│                                     │
│  特点：                              │
│  • 简单直接，易于理解                 │
│  • 适合单一任务场景                   │
│  • 难以处理复杂业务流程               │
└─────────────────────────────────────┘
```

**适用场景**：
- 单一意图识别（如只处理房源查询）
- 工具数量有限（3-5个）
- 业务流程线性（无复杂分支）

#### 阶段二：分层Agent（Layered Agent）

```
┌─────────────────────────────────────┐
│           分层Agent架构              │
├─────────────────────────────────────┤
│           路由层（Router）           │
│    意图识别 → Agent选择 → 任务分发   │
├─────────────────────────────────────┤
│           业务层（Agents）           │
│  ┌─────────┐ ┌─────────┐ ┌────────┐ │
│  │咨询Agent│ │预约Agent│ │合同Agent│ │
│  └─────────┘ └─────────┘ └────────┘ │
├─────────────────────────────────────┤
│           工具层（Tools）            │
│  ┌─────────┐ ┌─────────┐ ┌────────┐ │
│  │房源查询 │ │日历API  │ │文档生成│ │
│  └─────────┘ └─────────┘ └────────┘ │
└─────────────────────────────────────┘
```

**核心改进**：
- **路由层**：统一入口，智能分发
- **业务层**：专业Agent各司其职
- **工具层**：共享工具，避免重复开发

#### 阶段三：Multi-Agent协作系统

```
┌─────────────────────────────────────────────────────┐
│              Multi-Agent协作架构                     │
├─────────────────────────────────────────────────────┤
│                  协调者（Orchestrator）              │
│         任务分解 → 进度跟踪 → 结果整合              │
├─────────────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ 需求分析 │  │ 方案设计 │  │ 执行实施 │          │
│  │  Agent   │←→│  Agent   │←→│  Agent   │          │
│  └──────────┘  └──────────┘  └──────────┘          │
│       ↑              ↑              ↑              │
│       └──────────────┼──────────────┘              │
│                      ↓                              │
│              ┌──────────────┐                      │
│              │  共享记忆池  │                      │
│              │  (Memory)    │                      │
│              └──────────────┘                      │
├─────────────────────────────────────────────────────┤
│              消息总线（Message Bus）                 │
│         A2A协议通信 → 状态同步 → 异常广播            │
└─────────────────────────────────────────────────────┘
```

**核心特性**：
1. **协调者模式**：中央调度，避免Agent间直接耦合
2. **共享记忆**：跨Agent的上下文保持
3. **消息总线**：标准化通信协议（如A2A）

### 2.2 Multi-Agent架构设计模式

#### 模式一：流水线模式（Pipeline）

```
输入 → [Agent A] → [Agent B] → [Agent C] → 输出
         ↓            ↓            ↓
       提取信息    分析处理     生成结果
```

**应用场景**：安居乐寓租赁合同生成
```
用户需求 → [信息提取Agent] → [条款分析Agent] → [合同生成Agent] → 合同文档
              ↓                    ↓                    ↓
           识别关键信息         匹配法律条款          生成标准合同
```

**优点**：
- 流程清晰，易于理解
- 每个Agent职责单一
- 便于并行优化

**缺点**：
- 延迟累积（每个Agent都有处理时间）
- 单点故障（中间Agent失败导致全流程失败）

#### 模式二：委员会模式（Committee）

```
                    ┌→ [Agent A] ─┐
                    │   (评估师)   │
输入 → [分发器] ─┼→ [Agent B] ─┼→ [决策器] → 输出
                    │   (分析师)   │
                    └→ [Agent C] ─┘
                       (审核员)
```

**应用场景**：房源定价建议
```
房源信息 → [分发器] → [市场分析Agent] ─┐
                      [成本核算Agent] ─┼→ [定价决策Agent] → 建议价格
                      [竞品分析Agent] ─┘
```

**优点**：
- 多维度分析，结果更全面
- 可配置投票机制
- 适合需要多方评估的场景

**缺点**：
- 成本较高（调用多个Agent）
- 需要解决冲突和分歧

#### 模式三：主从模式（Master-Slave）

```
              ┌→ [Worker A]
[Master] ─┼→ [Worker B]
              ├→ [Worker C]
              └→ [Worker D]
```

**应用场景**：批量房源信息处理
```
[调度Agent] ─┬→ [处理Worker 1] → 房源A信息
             ├→ [处理Worker 2] → 房源B信息
             ├→ [处理Worker 3] → 房源C信息
             └→ [处理Worker 4] → 房源D信息
```

**优点**：
- 并行处理，效率高
- 易于扩展（增加Worker即可）
- 适合批处理任务

**缺点**：
- 需要任务分解能力
- 结果合并逻辑复杂

### 2.3 架构选型决策树

```
开始
│
├─ 任务是否单一？
│  ├─ 是 → 单体Agent
│  └─ 否 → 继续
│
├─ 是否需要多步骤处理？
│  ├─ 是 → 流水线模式
│  └─ 否 → 继续
│
├─ 是否需要多维度评估？
│  ├─ 是 → 委员会模式
│  └─ 否 → 继续
│
├─ 是否需要并行处理？
│  ├─ 是 → 主从模式
│  └─ 否 → 分层Agent
│
└─ 结束
```

## 三、性能优化策略（20分钟）

### 3.1 延迟优化

#### 问题分析

Agent系统的延迟来源：
```
总延迟 = LLM调用延迟 + 工具调用延迟 + 网络传输延迟 + 处理逻辑延迟
        ↓              ↓              ↓              ↓
      500-3000ms    100-1000ms     50-200ms      10-100ms
```

#### 优化策略一：模型选型优化

| 场景 | 推荐模型 | 延迟 | 成本 | 质量 |
|-----|---------|-----|-----|-----|
| 简单意图识别 | GPT-3.5/Claude 3 Haiku | 低 | 低 | 中 |
| 标准对话 | GPT-4o mini/Claude 3 Sonnet | 中 | 中 | 高 |
| 复杂推理 | GPT-4o/Claude 3 Opus | 高 | 高 | 极高 |
| 代码生成 | GPT-4o/Claude 3.5 Sonnet | 高 | 高 | 极高 |

**安居乐寓实践**：
```
用户输入 → [意图识别(轻量模型)] → 判断复杂度
                ↓
        ├─ 简单查询 → GPT-3.5快速响应
        └─ 复杂咨询 → GPT-4深度处理
```

#### 优化策略二：缓存机制

```
┌─────────────────────────────────────┐
│           多级缓存架构               │
├─────────────────────────────────────┤
│ L1: 内存缓存（Redis/Memcached）     │
│     • 高频查询结果                   │
│     • TTL: 5-10分钟                  │
│     • 命中率: 60-70%                 │
├─────────────────────────────────────┤
│ L2: 向量缓存（Vector DB）            │
│     • 相似问题匹配                   │
│     • 语义缓存                       │
│     • 命中率: 20-30%                 │
├─────────────────────────────────────┤
│ L3: 持久化存储（Database）           │
│     • 历史对话记录                   │
│     • 用户偏好配置                   │
└─────────────────────────────────────┘
```

**缓存键设计**：
```python
# 示例：房源咨询缓存键
cache_key = f"house_query:{hash(query)}:{user_preferences}"

# 缓存内容
{
    "query": "南山区两室一厅",
    "result": [...],
    "timestamp": "2025-01-30T10:00:00Z",
    "ttl": 600  # 10分钟
}
```

#### 优化策略三：异步处理

```
同步处理（用户体验差）：
用户请求 → [处理中...3秒] → 响应

异步处理（用户体验好）：
用户请求 → [立即返回任务ID] → 后台处理 → [推送/轮询结果]
              ↓
         "正在为您查询，请稍候..."
```

**安居乐寓应用**：
```
复杂报告生成流程：
1. 用户请求："生成南山区Q1租赁市场分析报告"
2. 立即响应："报告生成中，预计需要2分钟，任务ID: RPT-2025-001"
3. 后台异步：
   - 数据收集Agent并行查询多个数据源
   - 分析Agent处理数据
   - 报告Agent生成文档
4. 完成通知：推送/邮件发送报告链接
```

#### 优化策略四：流式输出

```
非流式（等待全部生成）：
[等待3秒] → 一次性显示全部内容

流式（逐字显示）：
"正" → "正在" → "正在为" → "正在为您" → ...
（用户感知延迟降低70%）
```

### 3.2 成本控制

#### 成本构成分析

```
月度成本构成（示例）：
┌─────────────────────────────────────┐
│ LLM API调用          60%  ¥12,000   │
│ 向量数据库           15%  ¥3,000    │

│ 其他工具API          10%  ¥2,000    │
└─────────────────────────────────────┘
总计: ¥20,000/月
```

#### 降本策略

**策略1：Prompt压缩**
```
优化前：
"你是一个专业的房屋租赁顾问，你需要帮助用户找到合适的房源。
用户的问题是：{query}
请参考以下信息：{long_context}
请给出详细的回答..."
（Token数: 500）

优化后：
"租赁顾问。问题：{query}。参考：{compressed_context}"
（Token数: 80，节省84%）
```

**策略2：响应截断**
```python
# 设置最大Token限制
max_tokens = 500  # 而非无限制的2000+

# 对于简单问题，限制输出长度
if intent == "simple_query":
    max_tokens = 150
```

**策略3：批量处理**
```
逐个处理（成本高）：
10个请求 × 10次API调用 = 100次调用

批量处理（成本低）：
10个请求 → 合并为1个批次请求 → 1次API调用
（节省90%开销）
```

**策略4：模型降级**
```
自动降级策略：
- 首次请求：GPT-4（高质量）
- 如果质量满足阈值 → 后续同类请求使用GPT-3.5
- 监控质量指标，必要时回退
```

## 五、容错与故障恢复（15分钟）

### 5.1 错误分类与处理策略

| 错误类型 | 示例 | 处理策略 |
|---------|-----|---------|
| **网络错误** | API超时、连接失败 | 重试 + 降级 |
| **模型错误** | LLM输出异常、格式错误 | 重试 + 兜底回复 |
| **工具错误** | 数据库查询失败 | 缓存读取 + 告警 |
| **业务错误** | 参数无效、权限不足 | 友好提示 + 引导 |
| **系统错误** | 内存不足、服务崩溃 | 熔断 + 快速失败 |

### 5.2 重试机制设计

#### 指数退避重试

```python
import time
import random

def retry_with_backoff(func, max_retries=3, base_delay=1):
    """
    指数退避重试策略
    第1次重试: 1秒 + 随机抖动
    第2次重试: 2秒 + 随机抖动
    第3次重试: 4秒 + 随机抖动
    """
    for attempt in range(max_retries):
        try:
            return func()
        except Exception as e:
            if attempt == max_retries - 1:
                raise e
            
            # 计算退避时间
            delay = base_delay * (2 ** attempt)
            jitter = random.uniform(0, 1)
            sleep_time = delay + jitter
            
            print(f"Attempt {attempt + 1} failed, retrying in {sleep_time:.2f}s...")
            time.sleep(sleep_time)
```

### 5.4 熔断机制

```
熔断器状态流转：

┌─────────┐    失败率>阈值    ┌─────────┐
│  CLOSED │ ───────────────→ │  OPEN   │
│ (正常)  │                  │ (熔断)  │
└─────────┘                  └────┬────┘
     ↑                            │
     │     超时后尝试半开          │ 拒绝请求
     └────────────────────────────┘
              ┌─────────┐
              │ HALF-OPEN│
              │(半开)   │
              └─────────┘
```

**安居乐寓熔断配置**：
```python
class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=60):
        self.failure_threshold = failure_threshold  # 连续失败5次触发熔断
        self.recovery_timeout = recovery_timeout    # 60秒后尝试恢复
        self.failure_count = 0
        self.last_failure_time = None
        self.state = "CLOSED"  # CLOSED, OPEN, HALF_OPEN
    
    def call(self, func, *args, **kwargs):
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "HALF_OPEN"
            else:
                raise CircuitBreakerOpen("Service temporarily unavailable")
        
        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise e
```
