## 3.1 概念解析

### 3.1.1 上下文窗口（Context Window）

**什么是上下文窗口？**

上下文窗口（Context Window）是指模型能同时处理的输入文本长度，通常以tokens为单位。

> "这就是上下文窗口(Context Window):它指模型能同时处理的输入文本长度(通常以 tokens 为单位，1token 约等于0.75 个英文单词或 0.5 个中文汉字)，是衡量模型'短期记忆容量'..."
> 


**通俗理解**：

想象你在读一本书，上下文窗口就是你一次能看到的页面数量。如果窗口太小，你可能会忘记前面发生了什么；如果窗口太大，处理速度会变慢，成本也会增加。

**常见模型的上下文窗口对比**：

| 厂商 | 最强模型 | 上下文窗口 |
|-----|---------|-----------|
| DeepSeek | DeepSeek V3.2 | 128K tokens |
| 阿里巴巴 | Qwen3-Max-Thinking | 256K tokens |
| 月之暗面 | Kimi K2.5 | 256K tokens|
| OpenAI | ChatGPT-5.2 | 400K tokens |
| Anthropic | Claude Opus 4.5 | 200K tokens |
| Google | Gemini 3.0 Pro | 1M tokens |

**上下文窗口包含什么？**

```
上下文窗口 = 
  系统指令（System Prompt）
  + 历史对话（Conversation History）
  + 当前输入（User Input）
  + 知识库检索结果（RAG Results）
```

### 3.1.2 Token计算与窗口管理

**Token换算参考**：

| 语言 | 1 Token约等于 |
|-----|--------------|
| 英文 | 0.75个单词 |
| 中文 | 0.5-0.7个汉字 |
| 代码 | 0.5-1个字符 |

**实际计算示例**：

文本："你好，我想了解西乡店的房源情况，预算在3000元左右。"

Token估算：
- 你好（2字）→ ~1 tokens
- 我想了解西乡店的房源情况（11字）→ ~8 tokens
- 预算在3000元左右（9字）→ ~7 tokens
- 标点符号 → ~3 tokens

总计：约19 tokens
可使用https://dashscope.console.aliyun.com/tokenizer进行计算

**窗口超限会发生什么？**

当对话内容超过上下文窗口限制时：
1. **截断（Truncation）**：最早的对话被删除
2. **摘要（Summarization）**：将早期对话压缩成摘要
3. **报错**：部分系统会直接拒绝处理
4. **幻觉（Hallucination）**：模型可能会编造信息来填补空白

### 3.1.3 注意力保持与"遗忘"

> "context=attention注意力:这两个单词是同一概念,注意力是有局限的,不是注意所有目标,而是注意力集中在特定上下文目标中。"
> 

**为什么长对话会"遗忘"？**

即使上下文窗口足够大，模型也可能"忘记"早期的内容，这是因为：

1. **注意力稀释**：信息太多，每个词获得的注意力减少
2. **位置偏见**：模型对后面的信息更敏感
3. **信息压缩**：模型必须压缩大量信息，细节可能丢失

**实际案例**：

```
对话第1轮：
用户："我叫张三，想找西乡的房"
AI："好的张三，西乡有很多优质房源..."

...（经过20轮对话）...

对话第22轮：
用户："我上次说的预算你还记得吗？"
AI：（可能已经忘记用户叫张三，也可能忘记预算）
```

## 3.2 Coze中的记忆机制

### 3.2.1 短期记忆：携带上下文轮数

**什么是短期记忆？**

在Coze中，短期记忆通过"携带上下文轮数"设置来控制。

**设置位置**：
Bot设置 → 对话配置 → 携带上下文轮数

**如何选择轮数？**

| 轮数设置 | 适用场景 | 优缺点 |
|---------|---------|--------|
| 1-3轮 | 简单问答、独立任务 | 成本低、速度快，但缺乏连续性 |
| 5-10轮 | 一般对话、需求挖掘 | 平衡成本和体验 |
| 10-20轮 | 深度咨询、复杂任务 | 体验好，但成本高 |
| 不限制 | 长文档分析、深度对话 | 体验最佳，成本最高 |

**成本与体验的平衡**：

```
轮数越多 = 上下文越长 = Token消耗越多 = 成本越高

但轮数太少 = 缺乏连续性 = 用户体验差

最佳实践：根据业务场景选择合适的轮数
- 客服场景：5-10轮
- 销售场景：10-15轮
- 技术支持：不限制或20轮以上
```

### 3.2.2 长期记忆：数据库功能

**为什么需要长期记忆？**

短期记忆会随着对话结束而消失，但业务需要记住：
- 用户的基本信息（姓名、电话）
- 用户的偏好（预算、区域）
- 历史交互记录（投诉、建议）

> "Agent的记忆机制通常划分为短期记忆（short-term memory）和长期记忆(long-term memory)，短期记忆决定了Agent在微观任务上的即时表现，而长期记忆则作为持久知识库，决定Agent的综合能力。"
> 
> —— 来源：《原来让AI拥有记忆这么简单?手把手教你实现Agent长短期记忆》

**Coze 数据库的核心作用**：

> "以 Coze 数据库为代表的持久化知识与状态管理组件，通过为 AI Agent 指挥官提供统一的上下文与历史决策参考，使多 Agent 系统能够在共享认知基础上协同运行。"
> 
> —— 来源：《AI Agent 指挥官如何避免"重复思考"?Coze 数据库的关键作用》

**数据库适用场景**：

| 场景 | 存储内容 | 示例 |
|-----|---------|------|
| 用户画像 | 姓名、电话、职业 | 张三，138xxxx，IT工程师 |
| 租房偏好 | 预算、区域、户型 | 3000元，西乡，单间 |
| 服务记录 | 投诉、维修、换房 | 2024-01空调维修 |
| 营销标签 | 活动参与、优惠券 | 深梦启航参与者 |

### 3.2.3 变量系统：全局变量与会话变量

> "变量（Variables） 是工作流中存储和传递数据的核心工具。理解变量及其类型，能让你更灵活地控制数据流，实现复杂逻辑。"
> 
> —— 来源：《Coze工作流-变量以及变量的类型讲解》

**变量类型对比**：

| 变量类型 | 生命周期 | 作用范围 | 使用场景 |
|---------|---------|---------|---------|
| **用户输入变量** | 单次对话 | 当前消息 | 获取用户输入 |
| **系统变量** | 整个会话 | 当前用户 | 跨轮对话保持 |
| **全局变量** | 永久 | 所有用户 | 配置信息、常量 |


**变量使用示例**：

```
场景：记住用户的租房偏好

1. 用户输入："我想在西乡租房，预算3000"

2. 提取变量：
   - {{user_name}} = "张三"
   - {{preferred_area}} = "西乡"
   - {{budget}} = "3000"

3. 存储到数据库：
   - 用户ID: 12345
   - 偏好区域: 西乡
   - 预算: 3000

4. 后续对话引用：
   "{{user_name}}，根据您之前提到的{{budget}}元预算，
   我为您在{{preferred_area}}找到了几套新房源..."
```

## 3.3 RAG基础：知识库如何增强记忆

### 3.3.1 什么是RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种通过集成外部知识源来增强大语言模型能力的技术。

> "RAG(Retrieval Augmented Generation)是一种通过集成外部知识源来增强大型语言模型(LLM)能力的范式。"
> 
> —— 来源：《2万字解读:RAG(检索增强生成)起源、演进与思考研究报告》

**为什么需要RAG？**

1. **知识截止**：大模型的知识有截止日期
2. **私有知识**：模型不知道企业的内部信息
3. **幻觉问题**：模型可能编造不存在的信息
4. **实时信息**：需要获取最新数据

### 3.3.2 RAG工作流程

> "你不会凭空编造答案，而是会：检索 (Retrieval)：先去查阅相关的书籍、论文、报告或数据库，找到最相关的信息片段。理解与整合 (Comprehension & Integration)：阅读这些信息，理解其含义，并将其与原有的知识整合。"
> 
> —— 来源：《RAG (Retrieval-Augmented Generation) 原理详解与实例》

**RAG三步走**：

```
用户提问 → 检索相关知识 → 整合到上下文 → 生成回答

示例：
用户问："安居乐寓西乡店的押金怎么算？"

Step 1: 检索
→ 从知识库找到"西乡店押金政策"文档

Step 2: 整合
→ 将政策内容加入上下文

Step 3: 生成
→ 基于检索到的信息回答用户
```

### 3.3.3 向量数据库简介

> "今天我们正式开启第三个重磅专题——RAG 架构与数据工程之向量数据库。对于大数据老兵来说，数据库是我们的'后花园'。但 AI 时代的数据库(Vector DB)彻底颠覆了我们熟..."
> 
> —— 来源：《大数据工程师的 AI 觉醒:手把手带你拆解向量数据库》

**什么是向量数据库？**

传统数据库存储文本、数字等原始数据，而向量数据库存储的是"语义向量"——将文本转换为数字向量，捕捉语义信息。

**文本如何变成向量？**

```
文本："西乡地铁站附近的公寓"

向量化过程：
1. 分词：["西乡", "地铁站", "附近", "的", "公寓"]
2. 编码：将每个词转换为数字向量
3. 聚合：合并为整个句子的向量

结果：[0.23, -0.56, 0.89, ..., 0.12] （通常是几百到几千维）
```

**向量检索原理**：

```
用户提问："西乡地铁房"
         ↓
转换为向量：[0.25, -0.51, 0.92, ..., 0.15]
         ↓
计算相似度：
  文档A（西乡店介绍）：相似度 0.95 ← 最相关
  文档B（南山店介绍）：相似度 0.32
  文档C（福田店介绍）：相似度 0.28
         ↓
返回Top K个最相关文档
```

## 3.4 实战：创建能记住用户的Bot

### 3.4.1 需求分析

**业务场景**：
创建一个租房咨询Bot，能够：
1. 记住用户的姓名和联系方式
2. 记录用户的租房偏好（预算、区域、户型）
3. 在后续对话中主动使用这些信息

### 3.4.2 数据库设计

**数据表结构**：

```
表名：user_preferences

字段设计：
- user_id (主键): 用户唯一标识
- user_name: 用户姓名
- phone: 联系电话
- budget: 预算（元/月）
- preferred_area: 偏好区域
- room_type: 户型要求
- move_in_date: 期望入住时间
- special_requirements: 特殊需求
- created_at: 记录创建时间
- updated_at: 记录更新时间
```

### 3.4.3 Bot配置步骤

**Step 1: 创建数据库**

1. 进入Coze平台，选择你的Bot
2. 点击"数据库"标签
3. 创建新数据表，添加上述字段
4. 设置user_id为主键

**Step 2: 配置提示词**

```markdown
# 角色
你是安居乐寓的智能租房顾问"小安"。

## 任务
帮助客户找到合适的房源，并记住客户的偏好以便后续推荐。

## 记忆功能
你需要在对话中收集并记住以下信息：
1. 客户姓名（{{user_name}}）
2. 联系电话（{{phone}}）
3. 预算范围（{{budget}}）
4. 目标区域（{{preferred_area}}）
5. 户型要求（{{room_type}}）

## 工作流程
1. **初次接触**：主动询问客户的基本信息和需求
2. **信息收集**：将收集到的信息存入数据库
3. **后续对话**：引用已记住的信息，提供个性化服务

## 示例对话

客户："你好，我想租房"

小安："您好！很高兴为您服务😊 我是安居乐寓的租房顾问小安。

为了给您推荐最合适的房源，能否先了解一下：
1. 怎么称呼您？
2. 您希望在哪个区域租房？
3. 您的预算范围是多少？
4. 需要什么户型？"

---

（客户提供了信息）

小安："好的{{user_name}}，我已经记下了：
- 预算：{{budget}}元/月
- 区域：{{preferred_area}}
- 户型：{{room_type}}

接下来我会根据这些条件为您找房。您还有其他特殊要求吗？
比如是否需要近地铁、能否养宠物等？"
```

**Step 3: 设置工作流**

```
用户输入
  ↓
判断是否有用户信息
  ├─ 否 → 询问用户信息 → 存入数据库
  └─ 是 → 读取数据库 → 引用变量
  ↓
处理用户请求
  ↓
生成回复（引用用户偏好）
```

### 3.4.4 测试与验证

## 3.6 本课小结

### 核心概念回顾

1. **上下文窗口（Context Window）**：模型能同时处理的token数量，是"短期记忆容量"
2. **短期记忆**：对话中的历史消息，通过"携带上下文轮数"控制
3. **长期记忆**：持久化存储的用户信息，通过数据库实现
4. **变量系统**：用户输入变量、会话变量、全局变量
5. **RAG**：检索增强生成，通过外部知识库增强模型能力
6. **向量数据库**：存储语义向量，支持相似度检索

### 关键技能

- 理解上下文窗口的限制和成本影响
- 配置合适的上下文轮数
- 设计数据库表结构
- 在提示词中引用变量
- 实现Bot的记忆功能
