## 2.1 提示词方法论深化

### 2.1.1 BROKE框架详解

BROKE是一个融合OKR（目标与关键结果）方法论的提示词框架，特别适合工作任务类场景。

> "BROKE是Background, Role, Objectives, Key Result, Evolve的缩写，它包括五个部分：Background：阐述背景，为AI提供充足信息。Role:设定角色，让AI进入角色。Objectives:明确任务目标。Key Result:定义关键结果。Evolve:根据反馈进行优化。"


**BROKE五要素详解**：

| 要素 | 英文 | 说明 | 示例 |
|-----|------|-----|------|
| **B**ackground | 背景 | 阐述任务背景，提供充足信息 | "我们是一家深圳的长租公寓品牌，主要服务年轻白领群体" |
| **R**ole | 角色 | 设定AI扮演的角色 | "你是一位资深的房产销售顾问" |
| **O**bjectives | 目标 | 明确任务目标 | "为新客户推荐合适的房源" |
| **K**ey Result | 关键结果 | 定义成功的标准和输出格式 | "推荐3套房源，每套包含价格、面积、交通信息" |
| **E**volve | 演变 | 根据反馈进行优化 | "如果客户对价格敏感，优先推荐性价比高的房源" |

**BROKE框架示例**：

```markdown
【Background/背景】
安居乐寓是深圳领先的长租公寓品牌，主要面向22-35岁的年轻用户。
目前西乡店有10套空置房源，需要尽快出租。

【Role/角色】
你是一位经验丰富的公寓管家，擅长根据客户需求精准推荐房源。

【Objectives/目标】
为一位在南山科技园工作的客户推荐合适的房源。

【Key Result/关键结果】
请推荐3套最匹配的房源，每套包含：
1. 房源名称和户型
2. 月租金价格
3. 通勤时间（到南山科技园）
4. 3个核心卖点

【Evolve/演变】
- 如果客户预算有限，优先推荐性价比高的房源
- 如果客户注重生活品质，优先推荐装修较好的房源
```

### 2.1.2 CO-STAR框架详解

CO-STAR框架更侧重于内容创作和人际沟通场景，强调对风格、语气和受众的精细考量。

> "co-star框架(context,objective,style,tone,audience,response):这是一个更侧重于内容创作和人际沟通场景的框架,它额外强调了对风格、语气和受众的精细考量。"
> 


**CO-STAR六要素详解**：

| 要素 | 英文 | 说明 | 适用场景 |
|-----|------|-----|---------|
| **C**ontext | 背景 | 提供任务的宏观背景 | 所有场景 |
| **O**bjective | 目标 | 明确具体任务 | 所有场景 |
| **S**tyle | 风格 | 指定写作风格 | 内容创作 |
| **T**one | 语气 | 设定情感基调 | 客户沟通 |
| **A**udience | 受众 | 明确目标读者 | 营销文案 |
| **R**esponse | 响应格式 | 定义输出格式 | 所有场景 |

**CO-STAR框架示例**：

```markdown
【Context/背景】
安居乐寓西乡店新推出一批精装修公寓，需要制作宣传文案。

【Objective/目标】
写一段吸引目标客户的房源介绍文案。

【Style/风格】
现代、简洁、有温度，避免过于商业化。

【Tone/语气】
亲切友好，像朋友推荐一样自然。

【Audience/受众】
25-30岁、在南山或宝安工作的年轻白领，注重生活品质但预算有限。

【Response/响应格式】
- 标题（不超过15字）
- 正文（150-200字）
- 3个核心卖点（每点不超过10字）
- 行动号召语
```

### 2.1.3 框架选择指南

**什么时候用BROKE？**
- 工作任务类场景
- 需要明确交付物
- 有明确的KPI或成功标准
- 需要迭代优化

**什么时候用CO-STAR？**
- 内容创作类场景
- 需要控制风格和语气
- 面向特定受众
- 营销文案、客户沟通

## 2.2 进阶技巧

### 2.2.1 Few-Shot Prompting（少样本提示）

Few-Shot Prompting是通过提供少量示例，让模型学习任务模式的技术。

> "few-shot prompting:少样本提示(few-shot prompting)是一种提示工程技术,通过在给模型的问题中提供少量完成任务的示例,来引导模型生成符合期望的输出。这种方法利用了大型语言模型的上下文学习能力..."
> 

**为什么Few-Shot有效？**

> "few-shot examples会为大模型提供一些额外的知识，叫做in-context learning（一种学习范式，允许模型在给定的上下文中进行学习和推理，而无需真正更新模型参数）"
> 


**Few-Shot示例**：

```markdown
请将客户的口语化需求转换为标准查询参数。

示例1：
输入："我想找个西乡附近的两房，预算三千左右"
输出：
{
  "区域": "西乡",
  "户型": "两居室",
  "预算": 3000,
  "预算范围": "2800-3200"
}

示例2：
输入："南山科技园附近，单间，越便宜越好"
输出：
{
  "区域": "南山科技园",
  "户型": "单间",
  "预算": "最低",
  "优先级": "价格"
}

现在请处理：
输入："宝安中心，一房一厅，要近地铁，预算四千以内"
输出：
```

**Few-Shot最佳实践**：

1. **示例数量**：通常1-5个示例效果最佳
2. **示例质量**：选择有代表性的、边界清晰的示例
3. **示例多样性**：覆盖不同的输入类型
4. **格式一致性**：输入输出格式保持一致

### 2.2.2 分隔符的使用

分隔符用于隔离指令与数据，防止指令注入，提高提示词的清晰度和安全性。

**常用分隔符**：

| 分隔符 | 使用场景 |
|-------|---------|
| `"""` | 包裹长文本 |
| `###` | 分隔不同部分 |
| `---` | 分隔示例 |
| `<tag></tag>` | XML风格的结构化标记 |
| ` ``` ` | 代码块 |

**分隔符使用示例**：

```markdown
请分析以下客户反馈，并提取关键信息。

### 客户反馈 ###
"""
我上周入住西乡店的A栋302房间，整体体验还不错。
房间采光很好，但是空调有点吵，晚上影响睡眠。
另外希望能在公共区域增加一些健身器材。
"""

### 提取要求 ###
请提取以下信息：
1. 入住门店
2. 房间号
3. 正面反馈
4. 负面反馈
5. 改进建议

### 输出格式 ###
以JSON格式输出。
```

**为什么使用分隔符？**

1. **防止指令注入**：避免用户输入被误解为指令
2. **提高可读性**：清晰的结构便于理解和维护
3. **便于解析**：结构化内容更容易被程序处理

### 2.2.3 Markdown格式控制

> "将prompt markdown化,本质上是一次'软件工程化'的思维升级。"
> 

**Markdown在提示词中的作用**：

1. **强制模型输出结构化内容**
2. **提高提示词的可读性和可维护性**
3. **便于前端展示**

**常用Markdown语法**：

```markdown
# 一级标题 - 用于主要章节
## 二级标题 - 用于子章节
### 三级标题 - 用于细分内容

**加粗** - 强调重点
*斜体* - 次要信息

- 列表项1
- 列表项2
- 列表项3

1. 有序列表1
2. 有序列表2

| 表头1 | 表头2 |
|-------|-------|
| 内容1 | 内容2 |

> 引用块 - 用于示例或说明

```代码块``` - 用于代码或结构化数据
```

**Markdown格式控制示例**：

```markdown
请为客户生成一份房源对比表。

要求：
1. 使用Markdown表格格式
2. 包含以下列：房源名称、价格、面积、通勤时间、评分
3. 在最后一行添加"推荐结论"
4. 使用**加粗**标注最优项

示例输出格式：

| 对比项 | 房源A | 房源B | 房源C |
|--------|-------|-------|-------|
| 价格 | 2800元 | **2500元** | 3000元 |
| 面积 | 25㎡ | 22㎡ | **28㎡** |
| ... | ... | ... | ... |

**推荐结论**：推荐房源B，性价比最高。
```

## 2.4 实战：编写"安居乐寓金牌销售"System Prompt

### 2.4.1 需求分析

**业务场景**：
- 客户通过AI助手咨询租房
- 需要了解客户需求并推荐房源
- 要体现专业性和亲和力

**核心能力**：
1. 需求挖掘：主动询问客户预算、区域、户型
2. 房源推荐：根据需求推荐合适房源
3. 异议处理：解答客户疑问和顾虑
4. 促成转化：引导客户预约看房

### 2.4.2 使用BROKE框架编写

```markdown
【Background/背景】
安居乐寓是深圳领先的长租公寓品牌，在宝安、南山、福田等区域有20+门店。
主要客群是25-35岁的年轻白领，提供精装修、拎包入住的公寓产品。
目前主推"深梦启航"计划，为应届毕业生提供专属优惠。

【Role/角色】
你是安居乐寓的金牌销售顾问"小安"，拥有5年租房服务经验。
你热情、专业、有耐心，擅长倾听客户需求并提供精准推荐。

【Objectives/目标】
1. 通过对话了解客户的租房需求
2. 推荐3-5套最匹配的房源
3. 解答客户疑问，消除顾虑
4. 引导客户预约看房或留下联系方式

【Key Result/关键结果】
每次回复必须包含：
1. 对客户需求的理解和确认
2. 具体房源推荐（含名称、价格、核心卖点）
3. 明确的下一步行动建议

【Evolve/演变】
- 如果客户说"太贵了"：推荐性价比更高的房源或介绍优惠政策
- 如果客户说"再看看"：询问顾虑点，提供对比信息
- 如果客户表现出兴趣：立即引导预约看房
```

### 2.4.3 使用CO-STAR框架优化

```markdown
【Context/背景】
客户正在通过AI助手咨询安居乐寓的租房服务。

【Objective/目标】
以专业销售顾问的身份，帮助客户找到满意的房源。

【Style/风格】
专业但不生硬，热情但不夸张，像朋友推荐一样自然。

【Tone/语气】
友好、耐心、积极，使用emoji增加亲和力。

【Audience/受众】
25-35岁的年轻白领，可能是第一次租房，对深圳不熟悉。

【Response/响应格式】
1. 问候语（简短友好）
2. 需求确认（复述客户要求）
3. 房源推荐（3-5套，含关键信息）
4. 行动号召（引导下一步）
```

## 课后作业

1. **框架应用题**：为一个新场景（如"处理客户投诉"）分别用BROKE和CO-STAR框架编写提示词，比较两者的差异。

2. **Few-Shot实践题**：收集3-5个安居乐寓的真实客服对话，整理成Few-Shot示例，用于训练AI客服。

3. **优化迭代题**：将上节课编写的提示词，使用本课学到的技巧进行优化，并记录优化前后的对比。
