## 一、课程导入（10分钟）

### 1.1 从工作流到Agent

在前两章中，我们深入学习了Dify工作流的开发：
- **基础工作流**：输入、LLM、变量、条件、输出节点
- **进阶工作流**：知识库检索、HTTP请求、代码、意图识别
- **复杂工作流**：循环、并行、错误处理、性能优化

工作流的优势在于**确定性**和**可靠性**——按照预定义的路径执行，适合标准化、重复性的任务。

但现实中有许多任务：**复杂、多变、需要自主决策**。比如：

> **场景一：多意图对话**
> 用户说："我想在深圳租一套房子，预算5000左右，然后还要把管家微信推给我。"
> 
> 传统工作流：搜房和找管家是两个不同的意图分支，工作流难以应对有多个意图的对话。
> 
> AI Agent：能够理解用户意图，主动根据用户需求分别调用工具，给出推荐。

> **场景二：多变的问题**
> 用户问："这些房源里面哪个性价比最高？"
> 
> 传统工作流：需要预先配置所有可能的问题和答案，维护成本高。
> 
> AI Agent：自主判断用户意图，根据工具调用结果，进行分析自行生成回答。

### 1.2 什么是AI Agent？

> **观点**：AI Agent是具备自主感知、决策、执行与迭代能力的AI系统，核心是"能自己思考、自己行动、自己优化"，而非传统AI仅做被动响应。
> 
> **来源**：今日头条 - AI Agent(智能体):核心定义、能力与企业落地

**AI Agent（人工智能代理/智能体）**是一种能够**自主感知环境、做出决策并执行行动**的智能系统。

```

├─────────────────────────────────────────────────────────────────┤

│                                                                 │

│   • 推理分析        • 动态调整策略      • 与外部环境交互           │
│   • 学习优化        • 多步骤规划        • 反馈迭代改进             │
│                                                                 │
│   核心区别：从"被动响应"到"主动完成目标"                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 二、AI Agent核心概念（15分钟）

### 2.1 Agent的定义

> **观点**：智能代理（Agent）是一种能够在复杂环境中自主运作的计算实体，其智能行为依赖于四大核心能力：感知（Perception）、规划（Planning）、决策（Decision-making）和执行（Execution）。
> 
> **来源**：51CTO博客 - 3.2 Agent核心能力:感知、规划、决策与执行

**Agent的本质**：
- 不是简单的"问答机器人"
- 不是固定的"流程执行器"
- 而是**能够自主完成复杂任务的智能实体**

#### 能力一：感知（Perception）

**定义**：Agent理解和获取环境信息的能力。

**在Dify中的体现**：
- 理解用户输入的自然语言
- 识别用户意图和情绪
- 提取关键信息（实体识别）
- 感知上下文和历史对话

**示例**：
```
用户输入："我想在西乡租一套两室一厅的房子，预算5000左右"

感知结果：
├─ 意图：房源查询
├─ 区域：西乡
├─ 户型：两室一厅
├─ 预算：5000元/月
└─ 其他需求：未明确
```

#### 能力二：规划（Planning）

**定义**：Agent制定行动计划、拆解复杂任务的能力。

**在Dify中的体现**：
- 将复杂任务拆解为子任务
- 确定执行步骤和顺序
- 识别任务依赖关系
- 动态调整计划

**示例**：
```
任务：为用户找到合适的房源

规划结果：
Step 1: 提取用户需求参数
   └─ 区域=西乡, 户型=两室一厅, 最高预算=5000
   
Step 2: 调用房源搜索工具
   └─ 调用 search_house 工作流
   
Step 3: 分析房源优劣势
   └─ 对比价格、位置、配套
   
Step 4: 生成推荐回复
   └─ 整合信息，生成房源卡片
```

#### 能力三：决策（Decision-making）

**定义**：Agent评估选项、做出最优选择的能力。

**在Dify中的体现**：
- 选择最合适的工具/技能
- 决定是否需要追问用户
- 判断任务完成度
- 处理异常情况

**示例**：
```
场景：用户询问"押金怎么退？"

决策过程：
├─ 选项A：直接基于常识回答
│   └─ 评分：40分（可能不准确）
│
├─ 选项B：询问用户具体门店
│   └─ 评分：60分（安全但不够智能）
│
└─ 选项C：调用知识库检索工具获取最新政策
          根据返回结果生成准确回答
    └─ 评分：95分（准确、专业）

决策结果：选择选项C，调用 search_knowledge_base 工具
```

#### 能力四：执行（Execution）

**定义**：Agent调用工具、与外部系统交互、完成任务的能力。

**在Dify中的体现**：
- 调用工作流作为工具
- 与外部API交互
- 返回结果给用户

**示例**：
```
执行任务：查询房源信息

执行过程：
├─ 调用 search_house 工作流
│   └─ 参数：{area: "宝安区", layoutName: "二居室", maxPrice: "5000"}
│
├─ 接收返回结果
│   └─ 5套符合条件的房源
│
└─ 整合结果，生成房源卡片回复
```

## 三、Agent vs 工作流（10分钟）

### 3.1 本质区别：确定性 vs 自主性

> **观点**：本质差异：确定性 VS 自主性。工作流如同工业流水线，通过预定义代码路径严格编排LLM和工具的调用顺序。性能天平：可靠性 VS 灵活性。
> 
> **来源**：CSDN博客 - 工作流与AI Agents:揭示80%场景下的效能对比真相

```
┌─────────────────────────────────────────────────────────────────┐
│                工作流 vs Agent 对比                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   工作流 (Workflow)                    Agent (智能体)            │
│   ─────────────────                   ────────────────           │
│                                                                 │
│   📋 预定义路径                        🧠 自主决策               │
│      │                                   │                      │
│      ▼                                   ▼                      │
│   输入 ──▶ 节点A ──▶ 节点B ──▶ 输出    输入 ──▶ 理解 ──▶ 规划    │
│   (固定流程)                              │      │      │       │
│                                           ▼      ▼      ▼       │
│                                        决策 ◀── 执行 ──▶ 反馈    │
│                                        (动态循环)                │
│                                                                 │
│   特点：                               特点：                    │
│   • 确定性高                           • 灵活性强                │
│   • 可预测、可控                       • 自适应、智能            │
│   • 适合标准化流程                     • 适合复杂多变任务        │
│   • 易于调试和维护                     • 需要更多算力            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 详细对比

| 对比维度 | 传统工作流 | AI Agent | 说明 |
|---------|-----------|---------|------|
| **执行方式** | 按预定义路径执行 | 自主规划执行路径 | Agent更灵活 |
| **决策能力** | 基于条件的固定判断 | 基于理解的动态决策 | Agent更智能 |
| **工具使用** | 固定节点调用 | 自主选择和组合工具 | Agent更自主 |
| **容错能力** | 需要预先定义所有分支 | 动态处理异常情况 | Agent更鲁棒 |
| **开发成本** | 需要详细设计所有路径 | 需要设计能力和边界 | 各有优劣 |
| **运行成本** | 较低（确定性执行） | 较高（需要推理） | 工作流更经济 |
| **适用场景** | 标准化、重复性任务 | 复杂、开放式任务 | 互补关系 |

### 3.3 如何选择：工作流 vs Agent？

**选择工作流的情况**：
- ✅ 任务流程固定、标准化
- ✅ 需要高可靠性和确定性
- ✅ 对成本敏感
- ✅ 需要严格审计和合规

**选择Agent的情况**：
- ✅ 任务复杂、需要多步骤推理
- ✅ 用户输入开放、多变
- ✅ 需要自主决策和适应
- ✅ 追求智能化用户体验

**最佳实践：两者结合**

## 四、实操演示：搭建AI找房Agent（45分钟）

### 4.1 Dify平台Agent功能简介

Dify的Agent节点是构建智能体的核心组件，它具备以下特点：

1. **Function Calling**：自动识别需要调用的工具并生成调用参数
2. **多轮迭代**：支持最多5轮工具调用迭代
3. **工作流即工具**：可以将其他工作流作为工具集成到Agent中
4. **记忆能力**：支持对话历史上下文

### 4.3 步骤1：导入YAML工作流文件

首先，我们需要导入三个工具工作流，这些工作流将作为Agent的工具使用。

#### 4.3.1 导入房源接口工作流

1. 登录Dify平台，进入"工作室"
2. 点击右上角"导入YAML文件"
3. 选择 `房源接口.yml` 文件
4. 点击"确认导入"

**房源接口工作流说明**：

```yaml
# 工作流结构
开始节点 ──▶ HTTP请求节点 ──▶ 代码执行节点 ──▶ 结束节点

# 输入参数
- wxMiniAppId: 小程序ID（必填）
- token: 用户认证Token（必填）
- lat/lng: 用户经纬度（可选）
- storeNames: 门店名称（可选）
- area: 行政区（可选）
- address: 地址关键词（可选）
- distance: 距离范围km（可选）
- layoutName: 户型（开间/一居室/二居室/宿舍）
- maxPrice/minPrice: 价格范围（可选）

# 输出格式
返回Markdown表格，包含：预览图、门店名称、房间号、价格、面积、
户型、距离地铁、站点、站距、roomId
```

#### 4.3.2 导入管家接口工作流

1. 同样在"工作室"中导入 `管家接口.yml`
2. 工作流将自动创建

**管家接口工作流说明**：

```yaml
# 输入参数
- wxMiniAppId: 小程序ID（必填）
- token: 用户认证Token（必填）
- storeNames: 门店名称（可选）

# 输出格式
返回Markdown表格，包含：管家姓名、联系电话（可点击拨打）、
所属门店、企微二维码
```

#### 4.3.3 导入知识库检索工作流

1. 导入 `知识库workflow【勿动】.yml`
2. 该工作流已配置好知识库ID，用于检索租赁政策

**知识库工作流说明**：

```yaml
# 输入参数
- query: 用户查询问题（必填）

# 处理流程
开始节点 ──▶ 知识检索节点 ──▶ 代码执行节点 ──▶ 结束节点

# 输出格式
返回检索到的知识库内容文本
```

    必填: 否

    必填: 否
    
  address:

### 4.7 步骤5：配置工具调用策略并测试

#### 4.7.1 配置Agent策略

```yaml
Agent策略: Function Calling
策略提供商: langgenius/agent/agent
最大迭代次数: 5
```

**Function Calling说明**：
- Agent通过分析用户输入，自动判断需要调用哪些工具
- 根据工具返回结果，决定是继续调用工具还是生成最终回复
- 最多支持5轮迭代，防止无限循环

#### 4.7.2 配置记忆（上下文）

```yaml
记忆配置:
  窗口记忆:
    启用: 是
    窗口大小: 50
  查询提示词模板: |
    {{#sys.query#}}
    {{#sys.files#}}
```

### 5.5 常见问题

**Q1：结构化输出不稳定怎么办？**
- 检查JSON Schema是否正确，确保字段类型和必填项设置合理
- 在提示词中强调"只返回JSON，不要有其他内容"，减少模型输出无关内容
- 尝试调低Temperature（建议0.1-0.3），让输出更加确定

**Q2：参数提取不准确怎么办？**
- 在提示词中增加更多示例，展示不同输入下的参数提取结果
- 明确说明每个参数的提取规则和格式要求
- 处理好"未提及"的情况，为可选参数设置默认值

**Q3：Coze和Dify配置有什么区别？**
- 变量引用语法不同：Dify用`{{node.variable}}`，Coze用`{{VARIABLE}}`
- 结构化输出配置界面略有差异，Dify在LLM节点中配置，Coze有专门的结构化输出组件
- 核心逻辑相同，学会一个可以迁移到另一个平台

**Q4：Agent调用工具失败怎么办？**
- 检查工具参数是否正确传递，特别是form类型和llm类型的区别
- 查看工具工作流的运行日志，定位错误原因
- 确保工具工作流已发布，Agent只能调用已发布的工具

**Q5：如何调试Agent的决策过程？**
- 使用Dify的预览功能，查看Agent的思考过程和工具调用日志
- 在系统提示词中添加调试信息，观察Agent的理解是否正确
- 逐步简化提示词，定位导致问题的具体指令
