## 一、课程导入（5分钟）

### 1.1 为什么Agent需要工具？

在上一节课中，我们了解到Agent的四大核心能力：**感知、规划、决策、执行**。其中，**执行能力**是Agent与外部世界交互的关键。

但是，大语言模型（LLM）本身有一些局限：

| 局限 | 说明 | 示例 |
|-----|------|------|
| **知识截止** | 模型训练数据有时间限制 | 无法知道今天的天气 |
| **无法计算** | 复杂计算容易出错 | 精确数学运算 |
| **无法存储** | 不能持久化数据 | 保存用户信息 |
| **无法操作** | 不能执行实际操作 | 发送邮件、创建日程 |

**解决方案：工具调用（Tool Calling）**

> **观点**：工具调用(tool calling)是AI系统识别并使用外部功能的能力，极大扩展了模型的能力边界。例如，语言模型本身不能访问实时数据，但通过调用房源API，它可以提供当前房源信息；通过调用数据库查询工具，它可以获取最新产品信息。

## 二、工具调用基础概念（10分钟）

### 2.1 什么是工具调用？

**工具调用（Tool Calling）** 是指大语言模型根据用户的输入，自主判断是否需要调用外部工具（如API、函数、数据库等），并生成符合工具要求的结构化调用参数，最终由系统执行工具并将结果返回给模型。

**核心特点**：
1. **自主性**：模型自主决策是否使用工具
2. **结构化**：生成固定格式的调用参数（通常是JSON）
3. **扩展性**：通过工具无限扩展模型能力

### 2.2 Tool Calling vs Function Calling

在实际应用中，这两个术语经常被混用，但有一些细微差别：

| 术语 | 侧重点 | 使用场景 |
|-----|--------|---------|
| **Tool Calling** | 强调"工具"的概念 | 调用外部API、插件、服务等 |
| **Function Calling** | 强调"函数"的概念 | 调用代码函数、内置方法等 |

在Dify平台中，我们通常使用**工具（Tool）**或**工作流（Workflow）**来实现工具调用功能。

### 2.3 工具调用的价值

工具调用让Agent能够：

| 能力 | 说明 | 示例工具 |
|-----|------|---------|
| **获取实时信息** | 访问最新数据 | 房源API、天气API、新闻API |
| **执行计算** | 精确数学运算 | 计算器、代码解释器 |
| **数据存储** | 持久化信息 | 数据库、文件系统 |
| **外部操作** | 与外部系统交互 | 发送邮件、创建日程、下单 |
| **专业知识** | 访问专业数据库 | 房产知识库、法律条文库 |

## 三、Function Calling工作原理（15分钟）

### 3.1 核心概念：填槽与调用

> **观点**：Function Calling：自主"解析"内容并结构化【填槽】，自主"决策"使用工具并结构化【调用】。填槽：指大模型根据与用户的历史对话内容，将信息结构化的解析出来。槽位：可以理解为"参数"，比如用户询问深圳房源时，填槽的参数就是：城市、预算、户型等。
> 
> **来源**：今日头条 - AI大模型的双手-Function Calling调用外部工具的基本原理

**填槽（Slot Filling）**：
- 将用户的自然语言输入，提取并填充到预定义的参数结构中
- 类似于表单的自动填写

**调用（Function Call）**：
- 根据填槽结果，生成结构化的函数调用请求
- 系统执行对应的工具/函数

### 4.2 模式一：工作流模式（5个步骤）

#### 步骤4：添加代码执行节点（数据处理）

这是工作流模式的核心优势——可以编写Python代码处理返回数据：

```python
import json

def main(arg1):
    # 1. 基础解析
    try:
        data_obj = json.loads(arg1) if isinstance(arg1, str) else arg1
    except Exception as e:
        return {"result": f"❌ **JSON 解析异常**：{str(e)}"}

    # 2. 报错兜底逻辑
    code = data_obj.get("code")
    msg = data_obj.get("msg") or data_obj.get("message") or "未知错误"

    if code != 200:
        return {"result": f"⚠️ **查询失败**\n\n> **错误代码**：{code}\n\n> **提示信息**：{msg}"}

    # 3. 数据存在性校验
    data = data_obj.get("data", {})
    stores = data.get("list", []) if data else []
    
    if not stores:
        return {"result": f"ℹ️ **暂无房源信息**\n\n> 没能找到匹配的房源，换个条件试试吧！"}

    # 4. 构建 Markdown 表格
    header = ["预览图", "门店名称", "房间号", "价格(元)", "面积(㎡)", "户型", "距离地铁", "站点", "站距(m)", "roomId"]
    md_table = "| " + " | ".join(header) + " |\n"
    md_table += "| " + " | ".join(["---"] * len(header)) + " |\n"

    for store in stores:
        s_name = store.get("storeName", "-")
        m_name = store.get("nearMetroName", "-")
        st_name = store.get("nearStationName", "-")
        st_dist = store.get("nearStationDistance", "-")
        
        for room in store.get("layoutList", []):
            r_no = room.get("roomNumber", "-")
            price = room.get("price", "-")
            space = room.get("insideSpace", "-")
            layout = room.get("layoutName", "-")
            pic = room.get("homePic", "")
            r_id = room.get("roomId", "")
            
            img_md = f"![预览]({pic})" if pic else "无图"
            
            row = [
                img_md, s_name, r_no, str(price), str(space), 
                layout, m_name, st_name, str(st_dist), str(r_id)
            ]
            md_table += "| " + " | ".join(row) + " |\n"

    return {"result": md_table}
```

**代码解析**：

1. **JSON解析**：处理HTTP返回的JSON字符串
2. **错误兜底**：检查API返回的code，非200时返回友好错误信息
3. **空数据检查**：如果没有房源，返回提示信息
4. **Markdown表格构建**：将房源数据格式化为易读的Markdown表格

#### 步骤5：配置结束节点

添加结束节点，输出代码执行节点的result变量：

```yaml
  - result: {{#code.result#}}
```

### 4.3 模式二：自定义工具模式（3个步骤）

自定义工具模式适合快速接入外部API，无需复杂数据处理。

#### 步骤1：创建自定义工具

1. 进入Dify顶部菜单"工具"
2. 点击"创建自定义工具"
3. 填写工具信息：

```yaml
工具名称: 房源查询工具
工具描述: 查询安居乐寓平台的房源信息
API端点: https://uat-weapp.szajly.com/api/wechat/ai/searchHouse
请求方法: POST
```

#### 步骤2：配置Schema

在Schema编辑器中定义输入参数：

```yaml
openapi: 3.1.0
info:
  title: 房源查询工具
  description: 查询安居乐寓平台的房源信息
  version: "v1.0.0"
servers:
  - url: https://uat-weapp.szajly.com
paths:
  /api/wechat/ai/searchHouse:
    post:
      description: 查询房源信息
      operationId: searchHouses
      parameters:
        - name: applet-token
          in: header
          description: 用户Token
          required: true
          schema:
            type: string
        - name: wxMiniAppId
          in: header
          description: 小程序ID
          required: true
          schema:
            type: string
      requestBody:
        content:
          application/json:
            schema:
              type: object
              properties:
                city:
                  type: string
                  description: 城市名称
                minPrice:
                  type: integer
                  description: 最低价格
                maxPrice:
                  type: integer
                  description: 最高价格
                layoutName:
                  type: string
                  description: 户型名称
              required: ["city"]
      responses:
        "200":
          description: 查询成功
          content:
            application/json:
              schema:
                type: object
                properties:
                  code:
                    type: integer
                  data:
                    type: object
                  msg:
                    type: string
```

#### 步骤3：测试并发布

1. 填写测试参数
2. 点击"测试"验证API调用
3. 测试通过后点击"保存"

### 4.4 两种模式对比

| 对比维度 | 工作流模式 | 自定义工具模式 |
|---------|-----------|---------------|
| **开发步骤** | 5步（较复杂） | 3步（简单） |
| **数据处理** | ✅ 支持Python代码处理 | ❌ 直接返回API结果 |
| **错误处理** | ✅ 可自定义错误逻辑 | ⚠️ 依赖API返回 |
| **格式转换** | ✅ 可转换为Markdown等格式 | ❌ 原始JSON |
| **适用场景** | 复杂逻辑、数据转换 | 简单API调用 |
| **维护成本** | 中等 | 低 |
| **灵活性** | 高 | 中等 |

**选择建议**：
- 需要数据转换、错误兜底 → 使用**工作流模式**
- 快速接入、简单调用 → 使用**自定义工具模式**

```python
        return {"result": f"ℹ️ **暂无管家信息**\n\n> 该门店当前暂无负责管家。"}

    # 4. 构建 Markdown 表格
    header = ["姓名", "联系电话", "所属门店", "企微二维码"]
    md_table = "| " + " | ".join(header) + " |\n"
    md_table += "| " + " | ".join(["---"] * len(header)) + " |\n"

    for person in staff_list:
        name = person.get("realname", "-")
        # 优化电话显示，支持点击拨打
        phone = person.get("phone", "-")
        phone_md = f"[{phone}](tel:{phone})" if phone != "-" else "-"
        
        store = person.get("storeName", "-")
        pic_url = person.get("pictureUrl", "")
        
        # 处理企微二维码预览
        img_md = f"![{name}]({pic_url})" if pic_url else "暂无企微二维码"
        
        row = f"| {name} | {phone_md} | {store} | {img_md} |\n"
        md_table += row

    return {"result": md_table}
```

**关键优化点**：
1. **电话可点击拨打**：使用Markdown链接语法 `[电话](tel:电话号码)`
2. **企微二维码预览**：直接显示图片，方便用户扫码添加
3. **友好的空数据提示**：当门店没有管家时给出明确提示

#### 代码执行节点（防弹版）

```python
import json

def main(input_data):
    """
    防弹版提取函数：无论输入什么，保证不报错
    """
    # 1. 预设默认值
    result_text = ""
    
    try:
        # 数据预处理
        if input_data is None:
            data = []
        else:
            data = input_data

        # 如果传入的是字符串，尝试转成对象
        if isinstance(data, str):
            try:
                data = json.loads(data)
            except:
                pass
        
        # 兼容性处理
        if isinstance(data, dict):
            if 'result' in data: 
                data = data['result']
            elif 'data' in data: 
                data = data['data']
            else:
                data = [data]
        
        # 确保 data 是列表
        if not isinstance(data, list):
            data = []

        # 核心提取逻辑
        contents = []
        for item in data:
            if isinstance(item, dict):
                c = item.get('content', '')
                if c:
                    contents.append(str(c).strip())
        
        # 拼接结果
        result_text = "\n\n".join(contents)

    except Exception as e:
        result_text = f"数据处理异常: {str(e)}"

    # 必须返回字典，且 key 必须和输出变量名完全一致
    return {
        "combined_text": result_text
    }
```

**防弹版代码特点**：
1. **多层异常捕获**：从输入到输出的每个环节都有try-except
2. **类型兼容**：处理字符串、字典、列表各种类型
3. **字段兼容**：支持result/data等多种字段名
4. **永不崩溃**：即使出错也返回错误信息，不中断流程

自定义工具模式的错误处理相对简单：

## 附录：常见问题解答

### Q1：工作流模式和自定义工具模式可以互相转换吗？

**A**：
- 自定义工具可以嵌入到工作流中（通过HTTP请求节点调用）
- 工作流可以发布为工具供其他应用使用
- 建议：复杂工具先用工作流开发，稳定后可以封装为自定义工具

### Q2：代码执行节点支持哪些Python库？

**A**：
- Dify代码执行节点支持标准库（json、re、datetime等）
- 不支持第三方库（如requests、pandas等）
- 如需复杂处理，建议在HTTP请求前/后使用外部服务

### Q3：如何提高工具调用的准确率？

**A**：
1. 优化工具描述，让模型清楚知道什么时候用
2. 明确参数定义，提供示例值
3. 在Agent系统提示词中说明工具使用场景
4. 添加 few-shot 示例
5. 对复杂参数添加验证逻辑

### Q4：工具调用超时怎么处理？

**A**：
- 在HTTP请求节点设置合理的超时时间（建议10-30秒）
- 配置重试机制（建议3次）
- 在代码节点捕获超时异常并返回友好提示

### Q5：一个工作流应该包含多少个节点？

**A**：
- 没有固定限制，但建议保持简洁
- 简单工具：3-5个节点
- 复杂工具：5-10个节点
- 节点过多时考虑拆分为多个工作流

### Q6：如何调试工作流？

**A**：
