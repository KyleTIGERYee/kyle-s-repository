# 課後作業設計規範

本文檔規範課後作業的設計標準，確保作業結構清晰、要求明確。

---

## 作業分層設計

課後作業採用三層次設計，滿足不同學習需求：

| 層次 | 類型 | 目標 | 標籤 |
|-----|------|-----|------|
| **基礎作業** | 必做 | 鞏固核心技能 | `必做` |
| **進階作業** | 選做 | 提升實踐能力 | `選做` |
| **挑戰作業** | 加分 | 拓展應用場景 | `加分` |

### 每層作業必備要素

1. **任務描述**：明確要完成什麼
2. **具體要求**：詳細的執行步驟或條件
3. **測試用例**：便於驗證結果（如適用）
4. **提交內容**：需要提交的材料清單

---

## 提交內容規範

### 基礎作業提交內容

基礎作業的提交內容必須包含：

```
提交內容結構：
├── 一、基礎作業（必做）
│   ├── 1.1 Dify項目鏈接和名稱
│   ├── 1.2 工作流截圖
│   ├── 1.3 提示詞配置
│   ├── 1.4 測試結果表格
│   └── 1.5 問題與解決
├── 二、進階作業（選做）
└── 三、挑戰作業（選做）
```

### Dify項目信息要求

**必須放在 1.1 位置**，包含：
- 項目鏈接：Dify平台的項目URL
- 項目名稱：項目的具體名稱

### 提交內容列表格式

```markdown
**提交內容**：

1. Dify項目鏈接和項目名稱
2. 工作流截圖（包含完整節點連接）
3. LLM節點提示詞截圖
4. 結構化輸出配置截圖
5. 5個測試用例的運行結果截圖
6. 簡要說明遇到的問題和解決方案（200字以內）
```

## 學習建議格式

```markdown
### 學習建議

1. **先完成基礎作業**：確保掌握核心技能後再挑戰進階
2. **多測試邊界情況**：真實用戶輸入往往不按套路出牌
3. **記錄優化過程**：提示詞優化是迭代過程，記錄每次修改的原因和效果
4. **對比不同平台**：如果時間允許，在Coze和Dify上都嘗試一下，對比差異
```

## 常見問題格式

```markdown
### 常見問題

**Q1：結構化輸出不穩定怎麼辦？**
- 檢查JSON Schema是否正確
- 在提示詞中強調"只返回JSON，不要有其他內容"
- 嘗試調低Temperature（建議0.1-0.3）

**Q2：參數提取不準確怎麼辦？**
- 在提示詞中增加更多示例
- 明確說明每個參數的提取規則
- 處理好"未提及"的情況

**Q3：Coze和Dify配置有什麼區別？**
- 變量引用語法不同：Dify用`{{node.variable}}`，Coze用`{{VARIABLE}}`
- 結構化輸出配置界面略有差異
- 核心邏輯相同，學會一個可以遷移
```
