# 门店调价审核 Agent - 核心代码测试报告

## 一、测试概览

本次测试基于最新的 MCP 数据底座（从 `de_定价agent数据集` 中提取“创业花园”的真实数据结构），通过构建全链路自动化测试脚本 `run_all_business_scenarios.py`，模拟并验证了 PRD 中的核心业务场景。

- **测试目标**：验证 `core_logic.py` 中决策树和杠杆规则的准确性，确保与 PRD 业务逻辑一致。
- **测试用例数**：共计 25 个核心场景用例（覆盖 P1/P2 前置规则、杠杆规则、以及 R1~R60 四大空置期分支的边界情况）。
- **通过率**：3 / 25 （仅前置规则和极端杠杆规则通过，决策树存在多处重大异常）。

## 二、测试结论与发现的严重 Bug

在对 `core_logic.py` 进行全面测试后，发现代码与 PRD 之间存在严重的逻辑偏差和潜在的代码缩进错误，主要集中在以下几个方面：

### 1. 代码缩进错误导致大面积规则不可达（返回 UNKNOWN）
在空置期为 `60-90天` 和 `≥90天` 的逻辑中，`core_logic.py` 出现了致命的缩进错误：
- **Bug 表现**：当 `楼栋入住率 < 85%` 时，代码直接跌出 `if-else` 分支，返回 `[UNKNOWN] 未匹配到任何规则`。
- **根本原因**：代码第 513 行附近，本应属于 `楼栋入住率 < 85%` 的 `else` 分支（即 R42~R49），被错误地向右缩进了一级，变成了 `楼栋入住率 >= 85% 且 村入住率 <= 85%` 的处理逻辑。这导致 `R42-R49` 和 `R57-R60` 在楼栋入住率不达标时完全失效。

### 2. PRD 规则覆盖存在盲区（逻辑遗漏）
即使修复了上述缩进 Bug，目前的 PRD 逻辑设计本身也存在漏洞，导致部分场景无法命中规则：
- 例如：在 `60-90天` 分支中，PRD 仅定义了“楼栋≥85% 且 村＞85%” 以及 “楼栋＜85% 且 村＜85%” 的组合。
- **缺失场景**：如果现实中出现“楼栋≥85% 且 村≤85%” 或 “楼栋＜85% 且 村≥85%”，系统将无规则可循，引发异常。

### 3. 具体判断条件与 PRD 不一致
测试脚本中，即使给定了符合 PRD 描述的数据，部分用例仍然命中了错误的规则：
- **R3~R12 的判定偏离**：在 `空置<30天` 测试中，例如 TC-06（双高入住率但近期成交高）预期应该命中 R3 拒绝降价，但在测试中实际命中了 R1。这说明代码中的组合判断条件（尤其是对入住率的交叉判断）层级有问题。
- **降幅边界未按 PRD 落实**：代码中强行增加了 `R60-Gap` 等不在 PRD 表格中的自定义规则，且其触发逻辑与 PRD 对 R60 的定义（降幅>15% 时拒绝）存在语义上的冲突。

### 4. 杠杆规则（楼层限价）过于严苛
测试表明，杠杆规则的触发非常灵敏。在模拟测试中，当我们只调降某个单间的价格，而同户型上下楼层均保持原价时，极易触发 `杠杆规则：价格低于上下楼层，异常偏低` 被拦截。虽然这符合代码设定，但在业务实操中，如果店长只针对一套难租的尾盘申请降价，很可能会被杠杆规则“误杀”。

## 四、下一步建议

为了使门店调价审核 Agent 顺利上线并准确拦截，建议立即进行以下调整：

1. **重构 `core_logic.py` 的嵌套逻辑**：修复 `elif 空置天数 < 90` 和 `else` 分支中的缩进错误，严格对齐 PRD 中的树状图结构。
2. **完善 PRD 逻辑漏洞**：补充定义“楼栋与村级入住率表现相反”时的审核标准，防止代码进入死胡同。
3. **引入微小降幅豁免机制**：建议在杠杆规则中加入一定的容忍度（如 5% 以内），避免单套房源微调引发连环拦截。
