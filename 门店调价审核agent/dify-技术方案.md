## 1. 总体架构

```
┌─────────────────────────────────────────────────────────┐
│                  Chatflow「调价审核助手」                  │
│  ┌─────────────┐                                        │
│  │  Agent 节点   │── 接收文件 + 对话交互                   │
│  │  (LLM+工具)   │                                       │
│  └──────┬───────┘                                        │
│         │ 调用工具                                        │
│  ┌──────┴────────────────────────────────┐               │
│  │  工具1: 「调价审核引擎」(Workflow)       │               │
│  │  工具2: MCP查询 (list/query)           │               │
│  │  工具3: 邮件发送工具                    │               │
│  └───────────────────────────────────────┘               │
└─────────────────────────────────────────────────────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
   ┌────────────┐ ┌───────────┐ ┌───────────┐
   │ Workflow    │ │ MCP Server│ │ 邮件服务   │
   │ 审核引擎    │ │ (PgSQL)   │ │ (SMTP)    │
   └────────────┘ └───────────┘ └───────────┘
```

**为什么选混合方案？**
- **Chatflow** 提供对话交互，店长可以问问题、上传文件
- **Workflow** 用 Code 节点做确定性计算，不依赖 LLM 推理，保证审核准确性

## 2. 简化 Excel 模板设计

### 2.1 模板格式

店长只需填一个 sheet，结构极简：

```
┌──────────────────────────────────────┐
│  A列              │  B列             │
├───────────────────┼─────────────────┤
│  调价申请表        │                  │  ← 第1行：标题
│  门店             │  测试二店         │  ← 第2行
│  村               │  白石龙          │  ← 第3行
│  楼栋             │  民治白石龙老村44栋│  ← 第4行
│                   │                  │  ← 第5行：空行
│  房间号           │  调整后价格(元/月) │  ← 第6行：表头
│  201              │  1400            │  ← 第8行起：数据
│  202              │  1350            │
│  203              │  1500            │
│  ...              │  ...             │
└──────────────────────────────────────┘
```

### 2.2 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 门店 | ✅ | 店长所属门店名称 |
| 村 | ✅ | 楼栋所在村名（与MCP中的"村"字段一致） |
| 楼栋 | ✅ | 楼栋名称（与MCP中的"楼栋"字段一致） |
| 房间号 | ✅ | 需调价的房间号（与MCP中的"房间号"字段一致） |
| 调整后价格 | ✅ | 调整后的月租金（元） |

> **注意**：只填需要调价的房间，不需要调价的房间不用填。成本数据不再需要在Excel中填写，Agent 会自动从MCP「应收明细表」中获取。

### 2.3 模板文件

需要制作一个空白 Excel 模板文件供店长下载使用，可命名为 `调价申请模板.xlsx`。

## 3. Workflow 工作原理详解（通俗版）

> 这一节用最通俗的语言解释 Workflow 到底在干什么，帮助你理解每一步的作用。

### 3.1 打个比方

把整个 Workflow 想象成一条**工厂流水线**。店长上传的 Excel 就是**原材料**，最终输出的审核结果就是**产品**。中间每个节点就是流水线上的一个**工位**，每个工位只做一件事，做完把半成品传给下一个工位。

```
🏭 调价审核流水线

 📄Excel文件                                                          📋审核结果
    │                                                                    ▲
    ▼                                                                    │
┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
│ 工位1   │──▶│ 工位2   │──▶│ 工位3   │──▶│ 工位4   │──▶│ 工位5   │──▶│ 工位6   │
│ 收件    │   │ 拆包    │   │ 翻译    │   │ 查资料  │   │ 审核    │   │ 出报告  │
│(Start)  │   │(提取器) │   │(代码)   │   │(MCP)   │   │(代码)   │   │(End)   │
└────────┘   └────────┘   └────────┘   └────────┘   └────────┘   └────────┘
```

### 3.2 每个工位在做什么

---

#### 🔹 工位1：Start（收件台）

**做什么**：接收店长上传的 Excel 文件。

**类比**：就像快递收件台，把包裹接收进来。这个包裹就是 Excel 文件，但此时我们还不知道里面装了什么。

**输入**：Excel 文件
**输出**：一个文件对象（后续节点可以使用）

---

#### 🔹 工位2：Doc Extractor（拆包台）

**做什么**：把 Excel 文件"拆开"，将里面的内容提取为纯文本。

**类比**：就像把快递拆开，把里面的东西倒出来铺在桌上。Excel 是一个二进制文件，电脑没法直接"读懂"，Doc Extractor 负责将它转成人和后续代码都能看懂的纯文本。

**输入**：Excel 文件
**输出**：一段文本，大致长这样：
```
调价申请表
门店    测试二店
村      白石龙
楼栋    民治白石龙老村44栋
楼栋月成本(元)    250000

房间号    调整后价格(元/月)
201       1400
202       1350
203       1500
```

---

#### 🔹 工位3：Code Node「解析文件 + 生成SQL」（翻译台）

**做什么**：两个任务——

**任务A：翻译**。把上一步的纯文本"翻译"成程序能理解的结构化数据（JSON），好比把一封中文信翻译成标准表格：
```json
{
  "store": "测试二店",
  "village": "白石龙",
  "building": "民治白石龙老村44栋",
  "building_unit_cost": 50.5,
  "rooms": [
    {"room_no": "201", "adjusted_price": 1400},
    {"room_no": "202", "adjusted_price": 1350},
    {"room_no": "203", "adjusted_price": 1500}
  ]
}
```

**任务B：写查询单**。根据翻译出来的楼栋名和村名，生成两条 SQL 查询语句，告诉下一步"去数据库查什么数据"。好比写了两张查询单交给资料室：
- 查询单1：请给我「民治白石龙老村44栋」所有房间的信息
- 查询单2：请给我「白石龙」村所有房间的信息

**输入**：上一步提取出的文本
**输出**：结构化数据 + 两条SQL查询语句

---

#### 🔹 工位4 & 5：Tool Node（查资料台 × 2）

**做什么**：拿着上一步写好的"查询单"（SQL），去 MCP 数据库里查数据。

**类比**：就像派两个人同时去资料室查资料：
- **工位4**：拿着查询单1，去查「这栋楼的所有房间」的详细信息（价格、是否已租、空置天数、户型等）
- **工位5**：拿着查询单2，去查「这个村的所有房间」的详细信息（用于计算同村同房型的入住率和均价）

这两个人可以**同时去查**（并行执行），不用排队等。

**输入**：SQL 查询语句
**输出**：数据库返回的房源数据（类似一张大表格）

**返回的数据长什么样**（举例）：
```json
{
  "columns": ["房间号", "户型", "定价", "是否已租", "空置天数", ...],
  "rows": [
    ["201", "开间", 1529, 0, 45, ...],
    ["202", "开间", 1529, 1, 0, ...],
    ["203", "一居室", 2100, 1, 0, ...]
  ]
}
```

---

#### 🔹 工位6：Code Node「执行审核逻辑」⭐ 核心工位

**做什么**：这是整个流水线**最核心**的工位。它拿到三样东西：
1. 店长的调价申请（哪些房间要改成什么价格）
2. 楼栋所有房间的实时数据（来自工位4）
3. 全村所有房间的实时数据（来自工位5）

然后逐个房间进行审核，做以下判断：

```
对于每一个要调价的房间：

  1️⃣ 调增还是调减？
     → 调增：直接通过 ✅
     → 调减：继续往下判断 ↓

  2️⃣ 楼栋成本溢价率 ≥ 100%？
     → 不满足：不通过 ❌ 原因=成本线不达标
     → 满足：继续往下 ↓

  3️⃣ 计算各项指标（代码自动算）：
     • 空置了多少天？
     • 同楼栋同户型的房间，租出去了多少比例？
     • 同村同房型的房间，租出去了多少比例？
     • 最近7天同村同房型成交了几套？
     • 调完价后，均价会变成多少？

  4️⃣ 进入决策树判断（就是PRD里的那棵树）：
     根据上面算出的指标，一层一层判断
     → 最终给出：通过 ✅ 或者 不通过 ❌ + 原因

  5️⃣ 杠杆规则校验（决策树通过后的额外检查）：
     对通过决策树的房间，检查同楼栋同户型各楼层价格是否呈现规律性
     • 某层价格比楼上楼下都低 → 拒绝（异常偏低）
     • 某层价格比楼上楼下都高 → 限价在户型最大最小值之间
```

**类比**：这个工位就像一个**严格的审核员**，手里拿着一本审核手册（决策树规则），对照着每一条规则逐项检查，最终给出结论。区别在于这个"审核员"是代码写的，所以100%按规则执行，不会有主观偏差。

**输入**：调价数据 + 楼栋房源数据 + 全村房源数据
**输出**：
- `all_passed`：全部通过还是有不通过的（true/false）
- `summary`：审核结果的文字摘要
- `detail_json`：每个房间的详细审核结果

---

#### 🔹 工位7：End（出货台）

**做什么**：把审核结果打包输出。

如果这个 Workflow 被 Chatflow 调用，这些结果就会返回给 Chatflow 的 Agent，Agent 再用友好的语言展示给店长。

### 3.4 Chatflow 和 Workflow 是怎么配合的？

```
店长的视角（他只能看到这一层）：
┌──────────────────────────────────────┐
│  💬 调价审核助手（Chatflow）          │
│                                      │
│  店长：上传了一个Excel文件             │
│                                      │
│  助手：收到文件，正在审核...           │
│        （此时后台调用了Workflow）       │
│                                      │
│  助手：审核完成！以下3间房通过...       │
│        （Workflow返回了结果）          │
│                                      │
│  助手：已发送邮件给审核负责人 📧       │
│        （Agent调用了邮件工具）          │
└──────────────────────────────────────┘

后台实际发生的事情：
┌──────────────────────────────────────────────┐
│  Agent（大脑）                                │
│    │                                          │
│    ├── 看到店长上传了文件                       │
│    ├── 决定调用「调价审核引擎」工具（Workflow）   │
│    ├── 等待Workflow返回结果                     │
│    ├── 读取结果，用友好语言展示给店长             │
│    ├── 如果全部通过 → 调用「邮件发送」工具       │
│    └── 回复店长                                │
└──────────────────────────────────────────────┘
```

**简单总结**：
- **Chatflow** 是前台，负责跟店长聊天、接收文件、展示结果、发邮件
- **Workflow** 是后台计算引擎，负责做数学计算和规则判断，不直接面对用户
- Chatflow 里的 Agent（AI大脑）负责协调：什么时候调 Workflow、什么时候发邮件、怎么回复店长

## 5. Chatflow「调价审核助手」设计

### 4.1 创建步骤

1. Dify 后台 → 创建应用 → 选择 **Chatflow**
2. 配置 Agent 节点

### 4.2 Agent 节点配置

**模型**：GPT-4o 或同等能力模型

**工具列表**：

| 工具 | 来源 | 用途 |
|------|------|------|
| 调价审核引擎 | Workflow-as-Tool（上面创建的Workflow） | 核心审核 |
| query_local_data | MCP 工具 | 按需查询补充数据 |
| list_local_tables | MCP 工具 | 查看可用数据表 |
| 发送邮件 | 邮件插件/自定义工具 | 审核通过后发邮件 |

**文件上传**：开启，允许上传 `.xlsx` 文件

### 4.3 System Prompt

```markdown
# 角色
你是安居乐寓的「门店调价审核助手」。你的职责是帮助店长审核房源调价申请。

# 工作流程
当店长上传调价Excel文件时，按以下步骤操作：

## Step 1：调用审核工具
将店长上传的文件传递给「调价审核引擎」工具进行审核。

## Step 2：展示审核结果
根据工具返回的结果，向店长展示：
- 如果全部通过：展示通过摘要
- 如果存在不通过：展示不通过明细表格（房间号、当前价格、调整后价格、不通过原因）

## Step 3：发送邮件（仅审核通过时）
如果审核全部通过，调用邮件发送工具，发送审核通过通知邮件：
```

## 6. 邮件发送方案

### 方案A（推荐）：Dify 插件市场

1. 进入 Dify → 插件市场
2. 搜索 `email` 或 `SMTP` 或 `邮件`
3. 安装邮件发送插件
4. 配置 SMTP 信息：
   - SMTP 服务器：公司邮件服务器地址
   - 端口：465(SSL) 或 587(TLS)
   - 发件人邮箱 + 密码/授权码
5. 在 Chatflow 中将此插件添加为 Agent 工具

### 方案B：HTTP Request + 邮件微服务

如果插件市场没有合适的，可以搭建一个简单的邮件 API：

1. 用 Python Flask 写一个邮件发送接口（约30行代码）
2. 部署在内网服务器上
3. 在 Dify 中用「自定义工具」接入（通过 OpenAPI Schema）

需要的话我可以帮你写这个邮件微服务的代码。

### 方案C：Workflow 内 HTTP Request 节点

在 Workflow 中直接用 HTTP Request 节点调用公司的邮件 API（如果有的话）。

## 7. MCP 数据表准备

### 7.1 三表结构

本项目使用 **3 张 MCP 数据表**，通过「楼栋」字段关联：

#### 表1：`de_定价agent数据集`（房源主表）

| 字段 | 类型 | 说明 |
|------|------|------|
| 村 | TEXT | 所属村名 |
| 楼栋 | TEXT | 所属楼栋 |
| 房间号 | TEXT | 房间编号 |
| 户型 | TEXT | 格局（开间/一居室等） |
| 资产户型 | TEXT | 户型代码（A/B/C） |
| 定价 | REAL | 当前挂牌价 |
| 是否已租 | TEXT | 是/否 |
| 最后出租时间 | TEXT | YYYY-MM-DD HH:MM:SS |
| 成交时间 | TEXT | 成交日期 |
| 面积 | REAL | 套内面积㎡ |
| 是否电梯房 | TEXT | 是/否 |
| 采光系数 | INTEGER | 1-5 |
| 阳台 | TEXT | 有/无 |
| 楼层 | INTEGER | 所在楼层 |

#### 表2：`精装简装楼栋分类`（装修分类表）

| 字段 | 类型 | 说明 |
|------|------|------|
| 楼栋 | TEXT | 楼栋名（关联键） |
| 装修类型 | TEXT | 精装 / 简装 |

#### 表3：`应收明细表`（成本表）

| 字段 | 类型 | 说明 |
|------|------|------|
| 楼栋 | TEXT | 楼栋名（关联键） |
| 应付金额 | REAL | 该楼栋应付金额（元/月） |

> **关联方式**：三张表通过「楼栋」字段关联。

### 7.2 确认方法

在 Dify 中调用 `list_local_tables`，确认三张表都存在，且字段名与上表一致。
