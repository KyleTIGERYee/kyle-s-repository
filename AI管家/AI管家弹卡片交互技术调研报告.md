# AI Agent 弹卡片交互技术调研报告

**调研目标**：研究千问App点外卖弹卡片等生成式UI交互的技术实现方案，为AI管家项目提供技术参考。

**调研日期**：2026年4月10日

---

## 一、技术背景

### 1.1 交互范式演进

人机交互历史本质上是不断降低用户意图与机器执行之间"翻译损耗"的历史：

| 阶段 | 交互模式 | 特点 |
|------|---------|------|
| CLI时代 | 命令行界面 | 用户必须"像机器一样思考"，任何输入错误导致失败 |
| GUI时代 | 图形用户界面 | 降低认知门槛，但界面静态固化，用户需适应工具 |
| **GenUI时代** | 生成式用户界面 | 界面按需生成，工具适应用户意图 |

**核心转变**：从"用户适应工具"→"工具适应用户"，从"界面布局设计"→"意图结果导向设计"。

### 1.2 千问App案例分析

千问App"一句话点外卖"功能的核心流程：

1. **意图识别**：AI识别用户"点奶茶"的自然语言指令
2. **服务编排**：调用淘宝闪购/支付宝等原子化能力
3. **动态渲染**：在对话中直接生成可交互的商品卡片，无需跳转App
4. **闭环操作**：用户在卡片内完成选择、支付全流程

**关键特性**：
- 界面"用后即弃"，消除体验断点
- 服务以原子化能力形态直接呈现
- 支付能力原生融入AI对话

## 二、技术实现方案对比

### 2.1 四种主流方案

| 方案 | 实现原理 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **硬编码规则** | 服务端返回特定格式数据结构，前端映射到固定UI | 可控性强、稳定性高 | 灵活性差、扩展成本高 | 固定业务流程 |
| **AI生成HTML** | AI直接生成HTML字符串，前端渲染 | 灵活性高 | 效果不稳定、需持续调prompt、安全风险 | 低频非关键场景 |
| **模块化卡片模板** | 预定义卡片模板，AI按schema填充数据 | 效果稳定、开发成本适中 | 模板需人力开发 | **推荐方案** |
| **A2UI组合式Schema** | AI生成UI Schema，引擎解析动态组件组合 | 高度灵活、可组合 | 技术门槛高、引擎开发成本 | 前沿探索场景 |

### 2.2 方案详细对比

根据实验报告，各方案在关键指标上的表现：

| 指标 | 硬编码规则 | AI生成HTML | 模块化卡片模板 | A2UI组合式Schema |
|------|-----------|-----------|--------------|-----------------|
| 首次可用率 | 100% | 60-70% | 95% | 85% |
| 回退率 | 0% | 30-40% | 5% | 15% |
| 可控性 | 高 | 低 | 高 | 中 |
| 开发成本 | 低 | 低 | 中 | 高 |
| 迭代成本 | 改代码 | 改prompt | 改模板/代码 | 改引擎 |

**结论**：模块化卡片模板方案在稳定性、可控性、开发成本之间取得最佳平衡，适合AI管家项目。

## 三、核心技术原理

### 3.1 模块化卡片模板实现流程

```
┌─────────────────────────────────────────────────────────────────┐
│                     模块化卡片模板实现流程                           │
│                                                                 │
│  用户输入自然语言                                                │
│       │                                                         │
│       ▼                                                         │
│  Agent意图识别 + 参数提取                                         │
│       │                                                         │
│       ▼                                                         │
│  选择对应卡片模板 + 填充数据                                       │
│       │                                                         │
│       ▼                                                         │
│  输出结构化JSON（卡片Schema）                                      │
│       │                                                         │
│       ▼                                                         │
│  前端卡片引擎解析并渲染                                            │
│       │                                                         │
│       ▼                                                         │
│  用户交互 → 回传数据 → 继续对话                                    │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 流式输出机制（SSE）

生成式UI需配合流式输出技术解决大模型推理延迟问题：

**SSE (Server-Sent Events) 特点**：
- 基于HTTP协议，无需建立WebSocket复杂连接
- 单向通信（服务器→客户端），适合流式响应
- 自动重连机制，断线后自动恢复
- 与React Server Components结合，实现组件级流式传输

**Vercel AI SDK 流式组件原理**：
```
LLM流式输出 → React组件流式传输 → 前端逐步渲染
     │              │                    │
   token级      组件级流式           用户感知"即时响应"
   响应         RSC封装
```

## 四、Dify平台卡片渲染机制

### 4.1 Dify工作流与卡片交互

Dify支持通过工作流节点实现结构化输出：

| 节点类型 | 功能 | 卡片渲染支持 |
|---------|------|-------------|
| LLM节点 | 核心处理，输出结构化数据 | 可配置输出格式 |
| Answer节点 | 定义回复内容格式 | 支持Markdown渲染 |
| Parameter Extractor | 提取结构化参数 | 配合卡片使用 |
| Tool节点 | 调用外部工具 | 返回数据可渲染 |

### 4.2 Dify + 自定义前端集成方案

Dify官方推荐的卡片渲染架构：

```
┌─────────────────────────────────────────────────────────────────┐
│                     Dify + 前端卡片集成架构                         │
│                                                                 │
│  Dify Agent/Workflow                                           │
│       │                                                         │
│       │ 输出结构化JSON                                           │
│       ▼                                                         │
│  Dify API                                                       │
│       │                                                         │
│       │ SSE流式传输                                              │
│       ▼                                                         │
│  小程序前端                                                      │
│       │                                                         │
│       ├── 卡片渲染引擎（解析JSON Schema）                         │
│       │                                                         │
│       ├── 预置卡片模板库（Form/Select/Info/Action）               │
│       │                                                         │
│       └── 用户交互 → 回传数据 → Dify继续处理                       │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 NebulaAI + PagePlug方案（Dify生态）

Dify生态中NebulaAI + PagePlug组合方案：

- **NebulaAI**：在对话中动态渲染问答类、选择类卡片
- **PagePlug**：自定义卡片样式，支持带参数的复杂表单

**优势**：
- 无需额外开发卡片组件
- 支持动态配置卡片样式
- 与Dify工作流无缝集成

## 五、AI管家项目技术选型建议

### 5.1 推荐方案：模块化卡片模板 + SSE流式输出

**理由**：
1. **稳定性优先**：业务场景（登记租客、报修工单等）对准确性要求高，不能接受AI生成HTML的不确定性
2. **开发成本可控**：4类卡片模板（Form/Select/Info/Action）开发工作量可预估
3. **迭代灵活**：新业务场景可通过新增模板快速扩展
4. **Dify原生支持**：Dify工作流可直接输出结构化JSON，前端解析渲染

### 5.3 与千问App方案对比

| 对比维度 | 千问App | AI管家项目 | 差异原因 |
|---------|---------|-----------|---------|
| **生态整合** | 淘宝、支付宝等阿里生态深度整合 | 租务系统API | 生态差异 |
| **卡片复杂度** | 商品展示、支付流程 | 表单填写、工单创建 | 业务差异 |
| **支付能力** | 原生融入对话 | 跳转小程序支付页 | 合规要求 |
| **用户场景** | 高频生活服务 | 租客全生命周期服务 | 场景差异 |
| **技术方案** | 模块化卡片模板（推测） | 模块化卡片模板 | 一致 |

### 5.4 关键实现要点

1. **Dify侧配置**：
   - 在LLM节点中配置System Prompt，明确输出格式要求
   - 使用Template节点生成卡片Schema
   - 配置结构化输出（Structured Output）确保格式一致

2. **前端卡片引擎**：
   - 开发4类卡片组件（Form/Select/Info/Action）
   - 实现卡片类型路由器
   - 支持字段验证和错误提示
   - 实现用户操作数据回传

3. **流式渲染优化**：
   - SSE连接保持与自动重连
   - 卡片骨架屏（加载态）
   - 字段级流式填充（提升感知速度）

## 七、总结

千问App弹卡片交互的本质是**生成式UI技术**的落地应用，核心实现方案为**模块化卡片模板 + SSE流式输出**。该方案在稳定性、可控性、开发成本之间取得最佳平衡，适合AI管家项目的业务场景。

**关键结论**：
1. 推荐采用模块化卡片模板方案，而非AI生成HTML
2. Dify平台原生支持结构化输出，可与前端卡片引擎无缝对接
3. 前端需开发4类卡片组件，支持Schema解析和动态渲染
4. SSE流式输出提升用户体验，解决大模型推理延迟问题
