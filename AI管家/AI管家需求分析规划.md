## 一、需求全景：你到底要做什么？

### 1.1 一句话定义

> 基于 MCP 架构，将 AI 管家从「找房工具」升级为覆盖 **租前→租中→租后** 全生命周期的「7×24 数字管家」，核心交互模式是 **Agent 调用 MCP 工具 + 前端推送交互卡片**。

### 1.2 与之前规划文档的关键差异

你之前两份文档（`AI管家需求分析规划.md`、`AI项目规划_优化版.md`）把签约、换房、转租列为 **P3 暂缓**，理由是"13年老旧代码，动则生变"。但你这次的描述明显更激进——**要做全闭环**。

| 维度 | 之前文档结论 | 本次规划 | 变化原因 |
|------|------------|---------|---------|
| 签约 | P3暂缓，风险极高 | **租前闭环核心环节** | MCP架构解耦，AI管家只调API不碰核心代码 |
| 自助看房 | P2可以做 | **租前闭环关键一环** | 接口已具备，Agent只需编排调用顺序 |
| 退租/换房/转租 | P3暂缓 | **参考登记租客方式给卡片** | 同上：外挂式调用API，不碰核心系统 |
| 退款查询 | P3暂缓，金融敏感 | **纳入规划，查流水账** | 只做「查询」不做「操作」，风险可控 |

**这个转变成立的前提是**：后端已经/即将把各业务能力封装为 MCP 工具，AI管家**只是API的消费者**，不触碰核心系统代码。

## 二、架构统一认知：MCP + 登录前置

### 2.1 统一前置：MCP登录

根据你的价格计算接口 YML 文件，所有需要调用租务系统 API 的场景，**都必须先调用 `get_access_token` MCP 工具**：

```
用户发起请求
    │
    ├── Agent判断：需要调用租务系统API
    │
    ├── 第1步：调用 get_access_token（MCP工具）
    │   ├── 入参：account（用户手机号/系统账号）、password
    │   ├── 内部：RSA公钥加密 → 调用AJLY登录接口
    │   └── 返回：access-token
    │
    └── 第2步：携带 access-token 调用目标业务API
```

**⚠️ 关键问题待确认**：
- 登录账号密码从哪来？是用户的小程序登录态自动传递，还是用环境变量中的固定服务账号？
- 小程序场景下，用户已登录，token 可以直接透传给 Agent 还是需要重新走 MCP 登录？
- 如果用固定服务账号，是否存在权限越界风险（一个账号能操作所有租客的数据）？

## 三、租前闭环 —— 逐环节拆解

这是你描述最详细的部分，我逐个环节深挖。

### 3.1 环节①：AI找房（MCP升级版）

**现状**：已有2.0版本，本次升级为MCP形式，增加大量入参出参。

**基于 API_DOC.md 的能力分析**：

| 能力 | 入参 | 说明 |
|------|------|------|
| 价格筛选 | min_price, max_price | ✅ 已有 |
| 户型关键词 | key_word（模糊匹配area_name/pattern_name/lighting_name/balcony_name） | ✅ 已有 |
| 门店指定 | store_names | ✅ 已有 |
| 通勤搜索 | select_lat, select_lng, duration, transportation | ✅ 已有 |
| 地址搜索 | address, distance | ✅ 已有 |
| 电梯/阳台 | is_lift, is_balcony | 🆕 新增 |
| 自助看房筛选 | is_self_help | 🆕 新增（为后续自助看房铺路） |
| 楼层/村 | floor_num, village | 🆕 新增 |
| 配套信息出参 | 冰箱/洗衣机/热水器等12项 | 🆕 新增（Agent可推荐配套更好的房） |

**产品要点**：
- `is_self_help=1` 是「自助看房」的筛选入口，Agent 可以在用户表达看房意愿时自动加上
- 出参中新增的配套信息让 Agent 有能力做「比选推荐」——这是 3.0 的核心价值

### 3.2 环节②：自助看房

**知识库验证后的完整流程**：

```
用户看到合适房源，表达看房意愿
    │
    ├── Agent推送【身份信息卡片】
    │   ├── 真实姓名（必填）
    │   └── 身份证号（必填）
    │   └── ⚠ 二要素校验：提交时校验身份证合法性
    │
    ├── Agent推送【预约时间卡片】
    │   ├── 日期：当天起6天内
    │   └── 时间段：2小时一段（8:00-10:00, 10:00-12:00...）
    │
    ├── Agent调用【自助看房下单接口】（MCP工具）
    │   ├── 前置：get_access_token
    │   ├── 入参：用户姓名、身份证、手机号、房间ID、预约时间
    │   └── 出参：看房单号、看房单状态
    │
    └── Agent回复确认信息
        └── "已为您预约 [门店名] [日期] [时段] 自助看房"
```

**⚠️ 基于知识库的风险提示**：

1. **看房单数量限制**：同时最多 10 条待看房单。Agent 需要先查询用户当前待看房数量，超限时告知用户。
2. **重复预约限制**：同一楼栋在偏好设置天数内（默认7天）有开门记录则不可重复预约。Agent 下单前需校验。
3. **人脸识别**：看房执行时需要人脸验证，这一步**必须在小程序端完成**，Agent 无法代替。Agent 应提醒用户"到达现场后请在小程序中打开看房单进行人脸识别开门"。
4. **看房状态自动流转**：开门后1小时自动变更为「已看房」，晚9点定时任务将未看房的置为「已过期」——Agent 无需关心状态变更，但可以在用户询问时查询。

- [ ] 看房单查询接口是否已有？（Agent 需要查用户当前待看房数量）
- [ ] 预约限制的校验是前端做还是后端接口自带？

### 3.3 环节③：价格咨询（动态定价）

**已验证的流程**（基于价格计算接口 YML）：

```
用户看完房，问"这间房租一年多少钱？"
    │
    ├── Agent推送【租期选择卡片】
    │   └── 可选租期：3个月/6个月/1年/2年 等
    │
    ├── 用户选择租期后，Agent 执行：
    │   ├── 1. get_access_token（MCP登录）
    │   ├── 2. 调用「获取租期优惠」接口
    │   │   └── GET https://xxx/web/api/lease/v1/fixed/price/set/{storeId}/1
    │   ├── 3. 调用「定价和特价系数」接口
    │   │   └── GET https://xxx/web/api/v3/tenant/price/compute/{roomId}/0/1
    │   └── 4. 计算引擎：定价 × 特价系数 × (折扣/100)
    │
    └── Agent 输出 Markdown 价格详情表
```

**⚠️ 关键注意**：
- 根据知识库中「退转换续」文档，**真实的价格公式是**：`成交价 = 定价 × 房间优惠 × 租期优惠 × 企业优惠 × 续租系数`。当前 MCP 只实现了 `定价 × 特价系数 × 租期折扣` 三因子，**缺少「企业优惠」和「续租系数」两个因子**。
- 对于新签场景（租前），续租系数=1，影响不大。但企业优惠（深梦启航/深梦扬帆身份）如果漏算，价格会偏高。
- **建议**：Agent 推送价格时应标注"参考价格"而非"最终成交价"，与之前文档的决策一致。

### 3.4 环节④：登记租客

**基于知识库 CM-005 的深度分析**：

登记租客表单极其复杂（20+ 字段），直接在聊天卡片中复现全表单是**不现实**的。

**建议的卡片精简方案**：

```

│  📝 登记租客信息                                  │
│                                                  │
│  【自动带出·不可编辑】                             │

│                                                  │
│  【用户填写·必填】                                 │
│  租客姓名：[          ]                           │
│  身份证号：[                  ]                   │
│  手机号码：[           ]                          │
│  合同开始：[2026-04-10]（默认当天）                │
│  合同期限：[12个月 ▼]（下拉3/6/12/24月）           │

│                                                  │

└─────────────────────────────────────────────────┘
```

**精简逻辑**：
- **保留6项核心必填**：姓名、身份证、手机、合同开始、期限、缴费方式
- **自动填充**：房间信息从上下文获取、出房价格由定价接口获取、押金默认「押一」
- **省略项**：同住人、其他电话、租客住址、渠道来源、业务员（默认当前门店管家）、递增约定（默认无递增）、水电表底数（入住交接时录入）、各类其他押金和服务费

**⚠️ 重大风险**：
1. **业务员字段**：系统必填，但用户不知道该选谁。需要后端接口支持"默认分配门店管家"逻辑。
2. **合同类型**：默认「电子合同」还是「纸质合同」？AI签约场景应当强制为电子合同。
3. **字段校验**：身份证18位校验、手机号11位校验、合同时间合法性校验——这些必须在卡片前端做，不能等提交后报错。

**信息缺口**：
- [ ] 登记租客的接口是否已封装为 MCP 工具？接口入参出参？
- [ ] 哪些字段可以为空（后端是否支持）？还是必须全量提交？
- [ ] 业务员自动分配逻辑后端是否支持？

### 3.5 环节⑤：电子合同签约

```
登记租客提交成功
    │
    ├── Agent 调用【电子合同模板选择接口】（MCP工具）
    │   ├── 根据房间类型（毛坯/简装/改装/精装）+ 业务类型匹配模板
    │   └── 返回可用模板列表
    │
    ├── Agent 自动选择匹配的合同模板（或推卡片让用户确认）
    │
    ├── Agent 调用【生成电子合同接口】（MCP工具）
    │   ├── 入参：登记信息 + 模板ID
    │   └── 返回：电子合同链接/预览地址
    │
    ├── Agent 推送【合同预览卡片】到聊天窗口
    │   └── 用户预览合同内容
    │
    └── 用户确认签约
        └── ⚠ 需要人脸识别签约 → 唤起小程序签约页面
```

**⚠️ 基于知识库的风险提示**：
1. **合同模板配置**：知识库 `SS-002-系统参数配置.md` 显示，合同模板需按「业务类型 × 托管类型 × 合同种类 × 房屋类型 × 适用楼栋」匹配。Agent 自动选择模板的逻辑需要非常精确，**选错模板风险极大**。
2. **人脸签约**：电子合同签署需要人脸识别，这一步**无法在聊天窗口内完成**，必须跳转小程序的签约页面。Agent 应推送小程序路径卡片，引导用户跳转完成签约。
3. **首期款支付**：根据知识库「退转换续」文档，新签合同若**当天未缴纳押金和首期租金，0点自动关闭合同**并做退房处理。Agent 在签约成功后**必须立即提醒用户支付**，否则合同作废。
4. **合同模板自动匹配**：建议后端提供一个"根据房间ID自动返回适用合同模板"的 MCP 工具，而非让 Agent 自行判断业务类型、托管类型等复杂维度。

### 4.2 投诉工单

与维修工单交互模式一致，区别在于：
- 问题类型改为投诉类型（噪音/卫生/设施/服务等）
- 投诉工单走不同的处理流程（受理→转派→跟进→回复→关单）
- 增加「期望解决方案」字段

**接入难度评估**：🟢 低。流程成熟，与维修工单共用工单架构。

## 五、租后阶段 —— 逐场景拆解

### 5.1 统一交互模式：参照登记租客

你的描述是"在续约、退租、换房、转租等基础业务上，都参考登记租客的方式，给接口让 Agent 调用，前端给一个卡片，让客户自己填写基础信息"。

这个思路正确，但每个业务的**表单字段和风险等级差异极大**：

| 业务 | 用户需填信息 | 系统自动处理 | 风险等级 | 说明 |
|------|-------------|-------------|---------|------|
| **续约** | 续约期限 | 自动计算续约价（定价×房间优惠×租期优惠×企业优惠×续租系数） | 🟡 中 | 续租系数是新概念，逻辑相对清晰 |
| **退租** | 退租日期、退租原因 | 自动计算结算金额（超期房租+违约金+能源费+押金退还） | 🔴 高 | 涉及押金退还和违约金计算，金额敏感 |
| **换房** | 选择新房间、换房原因 | 计算差价+手续费+退补金额、生成新合同 | 🔴 高 | 跨两份合同，逻辑极复杂 |
| **转租** | 新租客信息（同登记租客） | 原租客结算（同退房）+ 新租客登记 + 300元手续费 | 🔴 高 | 等于退租+新签两步合一 |

## 九、架构师挑刺反思

> ⚠️ **【架构师挑刺反思 Reflection】**
> 
> 我仔细审视了以上方案，有如下高风险盲区需要你决策：
> 
> 1. **登录态方案是最大的架构风险**。如果用固定服务账号登录调用所有API，那这个账号拥有操作所有租客数据的权限——等于把钥匙交给了AI。一旦Agent幻觉或被恶意Prompt注入，可能替错误的用户登记租客、创建工单、发起退租。**强烈建议**：使用用户自身的登录态token透传给Agent，而非固定服务账号。
> 
> 2. **登记租客的精简字段方案与后端兼容性**。我把20多个字段精简到了6个，但后端接口是否允许其他字段为空？如果后端校验"业务员不能为空"，前端卡片又没有让用户选，会直接报错。**必须**跟后端确认每个字段的可空性，或者让后端提供一个"AI渠道登记租客"的专用接口，字段精简+自动填充。
> 
> 3. **电子合同签约存在断链风险**。聊天窗口→跳转小程序签约→人脸识别→签约成功→回到聊天窗口。这个链路中，用户跳出聊天窗口后，Agent 如何知道用户是否签约成功？需要一个回调机制或轮询接口。如果没有，Agent 只能被动等用户回来说"我签完了"。
> 
> 4. **首期款支付的时间炸弹**。新签合同当天不付款0点自动关闭——这意味着如果用户晚上11点通过AI签约，只有1小时付款窗口。Agent 必须在签约成功后**强提醒**用户立即支付，并在接近0点时发Push通知。
> 
> 5. **退租/换房/转租的审批链路不在Agent控制范围**。Agent能发起申请，但审批在企业微信中进行（管家→店长→财务）。Agent对审批进度没有可见性，除非后端提供"查询审批状态"的MCP工具。否则用户问"我的退租审批到哪了"，Agent答不上来。
> 
> 6. **价格计算的精度问题**。当前MCP只算了3个因子（定价×特价×租期折扣），真实公式是5个因子。如果用户是深梦扬帆身份，Agent算出来的价格比实际高，用户会质疑AI不靠谱。**要么补全5因子计算，要么在价格旁边明确标注"最终成交价以合同为准"**。
