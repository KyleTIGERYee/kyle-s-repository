## 一、需求背景与变更点

### 1.1 需求场景与痛点

当前AI管家仅具备「找房+知识库问答+找管家」三项能力，覆盖租客生命周期中极小的一部分。租客在找到房源后，如需看房、了解价格、签约入住、日常报修、查询账单、续约退租等，均需退出AI对话、自行摸索小程序功能或联系人工管家。

**本次需求目标**：基于MCP架构，将AI管家升级为覆盖 **租前→租中→租后** 全生命周期的智能服务入口，核心交互模式为 **Agent调用HTTP API + 小程序前端推送交互卡片**。

### 1.2 改动范围

| 端别 | 改动内容 |
|------|---------|
| **Dify Agent平台** | 新增/升级MCP工具（HTTP API封装），扩展Agent工具箱 |
| **小程序（租客C端）** | 新增聊天窗口内的交互卡片组件（表单卡片、选择卡片、跳转卡片），支持Agent推送后用户在卡片内完成操作 |
| **租务系统后台** | 不做代码改动，仅作为API提供方被AI管家调用 |

**鉴权方案**：
- **AI管家专用账号**（管理员权限）：在Dify环境变量中配置，Agent通过`get_access_token`MCP工具获取token，用于调用所有租务系统后台API
- **用户token**：小程序前端获取，用于识别当前对话用户的身份（手机号、所属门店、房间等）

### 2.1 全生命周期业务总览

```
┌─────────────────────────────────────────────────────────────────┐
│                    租客全生命周期服务地图                           │
│                                                                 │
│   【租前】                 【租中】                  【租后】      │
│   ┌────────────┐       ┌────────────┐          ┌────────────┐  │
│   │ ① AI找房    │       │ ⑥ 维修工单  │          │ ⑩ 续约办理  │  │
│   │ ② 自助看房  │       │ ⑦ 投诉工单  │          └────────────┘  │
│   │ ③ 价格咨询  │       │ ⑧ 账单查询  │                         │
│   │ ④ 登记租客  │       │   + 拉起支付 │                         │
│   │ ⑤ 电子合同  │       │ ⑨ 远程开门禁│                         │

│                         └────────────┘                          │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 统一前置规则

**规则1：MCP登录前置**
- 所有需要调用租务系统API的操作，Agent必须先调用`get_access_token`获取AI管家专用账号的access-token
- Token获取失败时，Agent应回复"系统暂时繁忙，请稍后重试或联系管家"

**规则2：用户身份识别**
- 用户进入AI管家聊天时，小程序前端将用户token传递给Agent
- Agent通过用户token获取用户基础信息（手机号、姓名、当前在租房间等）
- 已入住租客：Agent可关联其门店、房间、合同信息，支持租前/租中/租后服务
- 未入住用户：Agent仅支持租前服务（找房、看房、咨询）

**规则3：卡片交互统一规范**
- Agent根据业务场景推送不同类型的前端卡片
- 卡片在小程序聊天窗口中渲染，用户在卡片内完成操作后，数据回传Agent
- 卡片类型见第三章详述

## 三、前端交互设计与页面动线

### 3.0 卡片类型总览

| 卡片类型 | 用途 | 示例场景 |
|---------|------|---------|
| **表单卡片** | 用户填写信息并提交 | 登记租客、报修工单、续约办理 |
| **选择卡片** | 用户从选项中选择 | 选择租期、选择预约时间 |
| **信息展示卡片** | 展示查询结果 | 账单详情、电子合同 |
| **跳转按钮卡片** | Markdown按钮跳转小程序页面 | 自助看房、支付、深梦扬帆 |

### 3.1 功能①：AI找房（MCP升级版）

**触发条件**：用户表达找房意图（如"帮我找房"、"西乡有没有1500的房"）

**Agent行为**：
1. 从用户自然语言中提取搜索参数
2. 调用`search_houses` MCP工具
3. 将返回的房源列表以**房源卡片**形式推送

**接口**：已有MCP工具`search_houses`，入参出参见`API_DOC.md`

**前端交互**：沿用现有房源卡片，无需改动。

**Agent处理流程图**：

```
用户表达签约意向
    │
    ├── Step 1: 查询【用户大学生认证状态】
    │   ├── ✅ 已通过认证 ──▶ 标记 enterprise_discount = "深梦启航"
    │   └── ❌ 未认证 ──────▶ 标记 enterprise_discount = 无
    │
    ├── Step 2: 调用【价格计算接口】
    │   └── 定价 × 房间优惠 × 租期优惠 × 企业优惠(深梦启航 or 1)
    │
    ├── Step 3: 调用【获取服务费接口】
    │
    ├── Step 4: 查询【房间关联家具家电租赁业务】
    │   ├── 关联基础家具家电 ──▶ 获取基础家具家电租金+押金
    │   └── 关联增值家具家电 ──▶ 获取增值家具家电租金
    │
    ├── Step 5: 推送【登记租客表单卡片】
    │   └── 费用信息区显示企业优惠状态+家具家电费项
    │
    ├── Step 6: 用户填写信息 → 点击提交
    │
    ├── Step 7: 调用【获取水电表底数接口】（提交后校验）
    │   ├── ❌ 获取失败 ──▶ 终止流程，提示联系管家，推送管家联系卡片
    │   └── ✅ 获取成功
    │
    ├── Step 8: 调用【登记租客提交接口】
    │   └── 携带全量数据（含水电底数、企业优惠标识、家具家电费项）
    │
    └── Step 9: 登记成功 → 自动进入功能⑤电子合同
```

**业务规则**：

| 规则 | 说明 |
|------|------|
| **水电表底数提交后校验** | 用户点击「确认提交登记」后，Agent调用水电表底数接口。**获取失败则整个登记流程终止**，Agent提示用户联系管家进行线下登记，并推送管家联系方式 |
| **企业优惠·默认不选** | 默认不应用任何企业优惠，按标准价格（定价×房间优惠×租期优惠）计算 |
| **企业优惠·深梦启航自动带出** | Agent查询用户的大学生认证状态，若已通过认证，则**自动带出「深梦启航」优惠**，纳入出房价格计算。用户无需手动选择 |
| **企业优惠·仅限深梦启航** | AI管家渠道仅支持「深梦启航」一种企业优惠。其他企业优惠（企业框架合同等）不在AI管家中处理，需联系管家 |
| 出房价格计算（无优惠） | 定价 × 房间优惠 × 租期优惠 |
| 出房价格计算（深梦启航） | 定价 × 房间优惠 × 租期优惠 × 深梦启航折扣 |
| 押金 | 固定等于一个月出房价格（押一） |
| 缴费方式 | 固定值：月付，不提供选择 |
| 合同类型 | 固定值：电子合同，不提供选择 |
| 业务员 | 默认填入AI管家专用账号对应的用户，不需要用户选择 |
| 服务费 | 调用服务费接口获取固定金额，只读展示 |
| **家具家电租赁·自动关联** | Agent根据房源楼栋关联的供应商信息，自动带出家具家电租赁费项，用户不可编辑、不可删除 |
| **基础家具家电** | 楼栋关联了基础家具家电业务的供应商且供应商状态为“合作中”时，系统自动增加「基础家具家电租金」和「基础家具家电押金」（押金=租金），不可修改不可删除 |
| **增值家具家电** | 楼栋关联了增值家具家电业务的供应商且供应商状态为“合作中”时，系统自动增加「增值家具家电租金」，不可修改不可删除 |
| **无关联供应商** | 若楼栋未关联任何家具家电供应商，或供应商已禁用/已到期，则不展示家具家电费项区域 |
| **家具家电定价来源** | 家具家电租金金额来自供应商管理中审批通过的定价模板 |
| 水电表底数 | 用户提交后调用接口获取实时数据，自动填入提交参数，卡片中不展示 |
| 身份证校验 | 18位格式校验 |
| 手机号校验 | 11位格式校验 |
| 合同开始时间 | 默认当天，不能早于当前日期 |

│  │  基础家具家电租金：¥80.00/月                        │    │

**业务规则**：

| 规则 | 说明 |
|------|------|
| 续约计算公式 | 定价 × 房间优惠 × 租期优惠 × 企业优惠 × 续租系数 |
| 续租系数 | 根据新合同租期决定，由后端接口返回 |
| **续约·基础家具家电** | 楼栋关联了基础家具家电供应商且状态为“合作中”时，续约仍会生成「基础家具家电租金」，但**不产生基础家具家电押金** |
| **续约·增值家具家电** | 楼栋关联了增值家具家电供应商且状态为“合作中”时，续约仍会生成「增值家具家电租金」 |
| **续约·供应商禁用或到期** | 若供应商已禁用或合同已到期，续约时不再生成相关的家具家电费项 |
| 企业优惠校验 | 默认使用用户当前的企业优惠（如深梦启航），但提交时需校验：①用户是否有通过大学生认证；②毕业时间是否在深梦启航对应的企业框架优惠规定的时间范围内；③企业框架协议是否仍在有效期内。若任一条件不满足，则续约失败，提示用户联系管家进行续约 |
| 续约失败处理 | 企业优惠校验失败时，Agent回复"您的企业优惠已失效，请联系管家协助续约"，并推送管家联系方式卡片 |
| 电子合同生成失败 | 若电子合同生成失败，Agent回复"合同生成遇到问题，请联系管家协助处理"，并推送管家联系方式卡片 |

**需调用的接口（从Apifox获取）**：
1. 查询当前合同信息
2. 查询用户大学生认证状态（含毕业时间、企业框架协议有效期）
3. 续约价格计算接口（含续租系数）
4. 查询房间关联家具家电租赁
5. 续约提交接口

### 6.2 前端交互卡片

小程序前端需开发4类卡片组件，支持Agent推送后在聊天窗口中渲染。

#### 6.2.0 实现方案说明

本需求采用**模块化卡片模板方案**实现Agent弹卡片交互。

**方案原理**：

```
┌─────────────────────────────────────────────────────────────────┐
│                   模块化卡片模板实现流程                           │
│                                                                 │
│  用户输入自然语言                                                │
│       │                                                         │
│       ▼                                                         │
│  Agent意图识别 + 参数提取                                         │
│       │                                                         │
│       ▼                                                         │
│  选择对应卡片模板 + 填充业务数据                                   │
│       │                                                         │
│       ▼                                                         │
│  输出结构化数据（卡片Schema）                                      │
│       │                                                         │
│       ▼                                                         │
│  前端卡片引擎解析并渲染                                            │
│       │                                                         │
│       ▼                                                         │
│  用户交互 → 回传数据 → 继续对话流程                                 │
└─────────────────────────────────────────────────────────────────┘
```

**实现要点**：

| 环节 | 要点 |
|------|------|
| Dify侧 | 配置结构化输出（Structured Output），确保卡片Schema格式一致 |
| 前端侧 | 开发卡片渲染引擎，解析Schema动态渲染对应模板 |
| 数据传输 | 采用SSE流式输出，提升用户感知响应速度 |
| 用户交互 | 卡片操作数据回传Agent，触发后续对话流程 |

### 6.4 工具集成要求

#### 6.4.1 Dify平台集成规范

所有工具集成到Dify平台时需遵循以下规范：

| 要求项 | 说明 |
|--------|------|
| **工具命名** | 使用清晰的功能描述命名，便于Agent识别和调用 |
| **参数格式** | 入参出参使用JSON格式，字段命名使用snake_case |
| **错误处理** | 工具需返回明确的成功/失败状态和错误信息，Agent据此判断是否继续流程 |
| **超时配置** | 根据接口实际响应时间配置合理超时时间，避免用户等待过长 |
| **鉴权集成** | AI管家专用账号的token获取逻辑集成在各工具内部调用，不作为独立工具暴露给Agent |
| **日志记录** | 记录每次调用的入参、出参、耗时，用于问题排查 |

#### 6.4.2 Agent调用流程

```
Agent收到用户消息
    │
    ├── 1. 意图识别（NLU）
    │       └── 判断用户意图：找房/看房/咨询/签约/报修/账单/开门/续约
    │
    ├── 2. 参数提取
    │       └── 从自然语言中提取业务参数
    │
    ├── 3. 工具选择
    │       └── 根据意图选择要调用的工具
    │
    ├── 4. 调用业务工具（工具内部自动获取token）
    │       ├── 成功 → 解析返回数据，决定下一步
    │       └── 失败 → 返回错误提示，引导用户联系管家
    │
    ├── 5. 生成回复
    │       ├── 需要用户操作 → 推送卡片
    │       └── 仅需告知 → 直接回复文本
    │
    └── 6. 等待用户反馈
            └── 用户操作卡片后，接收回调数据，继续流程
```

#### 6.4.4 验收标准
