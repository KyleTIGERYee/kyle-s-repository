## 调研目标

竞品调研不是简单的"找几个竞品看看他们做了什么"，而是：

1. **找出该领域的标杆产品和头部玩家** - 了解市场格局
2. **深入分析，提取他们的核心策略、设计决策和创新点** - 理解他们为什么这样做
3. **形成可行动的洞察，为产品决策提供依据** - 知道我们该怎么做

### 典型触发场景

当用户提出以下需求时，执行竞品调研：
- "帮我看看这个领域有哪些竞品"
- "我想了解xxx功能的竞品实现"
- "我们和xxx相比有什么优势"
- "这个功能应该怎么设计"
- "我们要如何差异化竞争"

## 竞品类型识别

在开始调研前，先明确竞品类型，避免遗漏或误判。

### 直接竞品

**定义**：产品形态相同或类似，目标用户重叠度高，解决相同核心问题。

**特征**：
- 用户会在两者之间做选择
- 价格、功能直接可比
- 用户流失的主要来源

**示例**：
- 对于"安居乐寓"，自如、万科泊寓、泊寓等长租公寓平台是直接竞品

### 间接竞品

**定义**：解决相同问题，但产品形态不同，目标用户有部分重叠。

**特征**：
- 不是直接替代，但会争夺用户时间/注意力
- 功能上有交集，但核心价值不同
- 可能成为直接竞品（如果业务扩展）

**示例**：
- 对于"安居乐寓"，Airbnb（民宿）、贝壳（二手房）、链家（租房中介）是间接竞品

### 替代方案

**定义**：用户可能用的其他方式，不是产品竞争，而是方式竞争。

**特征**：
- 不是严格意义上的竞品
- 但用户确实会选择这种方式
- 需要了解为什么用户选择它

**示例**：
- 对于"安居乐寓"，直接找房东租、住酒店、合租等是替代方案

### 潜在竞品

**定义**：尚未进入市场或业务覆盖面还不够，但具备优势的玩家。

**特征**：
- 可能有技术、资源或渠道优势
- 可能会改变游戏规则
- 需要持续关注

**示例**：
- 对于"安居乐寓"，大型互联网公司（如字节跳动、美团）如果进入长租领域就是潜在竞品

## 搜索策略

### 通用搜索

先找出该领域"公认最好"的资源：

```
找竞品：
"{产品类型}" app
"{产品类型}" platform
"{产品类型}" saas
"{产品类型}" 产品

找头部玩家：
"{产品类型}" market leaders
"{产品类型}" top companies
"{产品类型}" 竞争格局
"{产品类型}" 市场份额
```

### 功能对比搜索

针对特定功能进行搜索：

```
"{功能名称}" feature comparison
"{功能名称}" best practices
"{功能名称}" implementation examples
"{功能名称}" 对比
"{功能名称}" 竞品
```

### 中文搜索

找到中文市场的竞品：

```
"{产品类型}" 竞品分析
"{产品类型}" 市场报告
"{产品类型}" 行业报告
"{产品类型}" 头部企业
"{产品类型}" 排名
```

### 验证品质

找到竞品后，验证其品质：

```
"{竞品名称}" review
"{竞品名称}" 口碑
"{竞品名称}" 投诉
"{竞品名称}" 用户评价
```

### 优先来源

1. **官方网站**：获取最权威的产品信息
2. **应用商店**：看用户评价、评分、下载量
3. **媒体报道**：了解行业地位和融资情况
4. **第三方分析平台**：如 SimilarWeb、App Annie（了解流量数据）
5. **社交媒体**：了解用户真实反馈

## 深度分析框架

对每个关键竞品，按照以下框架进行分析：

### 1. 产品定位与核心价值主张

**问题**：
- 他们解决什么问题？
- 目标用户是谁？
- 核心差异化是什么？

**分析要点**：
- 一句话描述这个产品是什么
- 他们的Slogan或核心卖点是什么
- 他们和同类产品的区别是什么

**示例**：
- 自如："高品质租住生活与生活服务"
- 万科泊寓："长租公寓第一品牌"
- 贝壳："住有所居，安居乐业"

### 2. 核心功能与架构

**问题**：
- 核心功能有哪些？
- 功能架构如何设计？
- 哪些功能是必选项？

**分析要点**：
- 列出所有主要功能模块
- 画出功能架构图
- 识别MVP（最小可行产品）和差异化功能

**对比维度**：
| 功能模块 | 自如 | 泊寓 | 我们 |
|---------|------|------|------|
| 在线看房 | ✅ | ✅ | ? |
| 在线签约 | ✅ | ✅ | ? |
| 智能门锁 | ✅ | ✅ | ? |
| 房租分期 | ✅ | ❌ | ? |

### 3. 用户体验与设计

**问题**：
- 信息架构如何设计？
- 关键流程是什么样的？
- 设计风格和交互特点？

**分析要点**：
- 首页布局和信息层级
- 关键用户流程（如看房、签约、退租）
- 设计风格（颜色、图标、字体）
- 交互细节（动画、反馈、引导）

**常见设计模式**：
- 首页：搜索框+推荐房源+分类导航
- 房源详情：图片+价格+位置+设施+评价
- 签约流程：分步引导+进度指示+确认页面

### 4. 商业模式与定价

**问题**：
- 如何盈利？
- 定价策略是什么？
- 有哪些增值服务？

**分析要点**：
- 收入来源（租金差价、服务费、广告等）
- 定价策略（按月、按季度、按年）
- 增值服务（保洁、维修、搬家等）

**对比维度**：
| 维度 | 自如 | 泊寓 | 我们 |
|-----|------|------|------|
| 收入模式 | 租金差价+服务费 | 租金差价 | ? |
| 付款周期 | 按月 | 按季度 | ? |
| 服务费 | 7% | 5% | ? |

### 5. 技术实现与创新点

**问题**：
- 有什么技术创新？
- 有哪些独特的设计决策？
- 技术架构有什么特点？

**分析要点**：
- 是否有独特的功能或流程设计
- 是否采用了新技术（AI、大数据、区块链等）
- 是否有开放API或第三方集成

**示例**：
- 自如的"VR看房"技术
- 泊寓的"智能家居"集成
- 贝壳的"VR+AI"房源描述

### 6. 弱点与机会

**问题**：
- 用户抱怨什么？
- 有哪些未被满足的需求？
- 我们的机会在哪里？

**分析要点**：
- 应用商店差评分析
- 社交媒体用户反馈
- 竞品论坛或社区讨论
- 自己试用发现的体验问题

**机会识别**：
- 用户抱怨"签约复杂" → 我们的"简化签约流程"
- 用户抱怨"服务费高" → 我们的"透明定价"
- 用户抱怨"找房难" → 我们的"智能推荐"

## 质量检核

在将竞品纳入分析前，确认通过以下检核。

### 必要条件（全部要通过）

### 相关性
- [ ] 直接与调研主题相关（不是"相关领域"）
- [ ] 目标用户有重叠
- [ ] 解决相同或相似的问题

### 可信度
- [ ] 来源可辨识（有官网或权威报道）
- [ ] 产品信息可验证（不是虚假或过时信息）
- [ ] 不是纯营销内容或业配文

### 可访问性
- [ ] 官网可正常访问
- [ ] 核心功能可体验（至少了解功能描述）
- [ ] 不需要特殊权限或VIP

### 竞品质检

| 检核项目 | 说明 |
|---------|------|
| [ ] 活跃度 | 最近有更新、有新功能发布 |
| [ ] 用户量 | 有一定的用户基数或市场份额 |
| [ ] 评分 | 应用商店评分4.0+或口碑良好 |
| [ ] 差异化 | 有自己的特色，不是完全抄袭 |

## 洞察提炼

### 洞察维度

#### 1. 市场格局

**问题**：
- 市场成熟度如何？
- 主要玩家有哪些？
- 我们的机会在哪里？

**输出格式**：
```markdown
### 市场格局
- **市场成熟度**：新兴 / 成长 / 成熟
- **主要玩家**：列出3-5个关键竞品
- **我们的定位机会**：我们的差异化机会在哪里
```

#### 2. 功能借鉴

**问题**：
- 哪些功能是行业标配？
- 哪些功能是差异化关键？
- 我们应该优先实现哪些？

**输出格式**：
```markdown
### 功能借鉴优先级
| 功能 | 竞品覆盖率 | 重要性 | 建议 |
|-----|----------|--------|------|
| 在线看房 | 100% | 高 | 必做 |
| 在线签约 | 80% | 高 | 必做 |
| 智能门锁 | 60% | 中 | 可选 |
| 租金分期 | 40% | 低 | 暂缓 |
```

#### 3. 设计启示

**问题**：
- 行业通用的设计模式有哪些？
- 有哪些创新的设计方式？
- 我们应该如何设计？

**输出格式**：
```markdown
### 设计启示
- **通用模式**：行业普遍采用的设计（如首页布局、流程引导）
- **创新做法**：值得借鉴的创新设计（如VR看房、智能推荐）
- **我们的设计方向**：基于竞品分析的设计建议
```

#### 4. 可行动建议

**问题**：
- 我们的差异化机会在哪里？
- 应该避免哪些坑？
- 可以超越竞品的地方？

**输出格式**：
```markdown
### 可行动建议
1. **短期**：近期应该做的（如必须实现的核心功能）
2. **中期**：下一步考虑的（如差异化功能）
3. **长期**：未来规划的（如技术创新、生态建设）

### 风险规避
- ⚠️ 竞品的常见问题：需要避免的坑
- ⚠️ 用户反馈的痛点：竞品没做好的地方
```

## 常见错误

| 错误 | 问题 | 修正 |
|-----|------|------|
| 只找直接竞品 | 可能忽略潜在威胁 | 找出所有类型的竞品 |
| 只看功能列表 | 不了解"为什么这样做" | 深入分析设计决策和策略 |
| 浅尝辄止 | 信息不够深入 | 每个竞品至少试用核心流程 |
| 只描述不分析 | 没有可行动的洞察 | 形成具体建议和优先级 |
| 只列优点 | 不了解竞品弱点 | 分析竞品的弱点和用户抱怨 |
| 数量过多 | 无法深入分析 | 聚焦3-5个关键竞品 |
| 信息过时 | 决策依据错误 | 验证信息的时效性 |
| 忽视用户反馈 | 不知道真实问题 | 关注差评和用户抱怨 |

## 执行建议

### 优先级

1. **头部玩家 > 小玩家**：先分析市场份额高的竞品
2. **直接竞品 > 间接竞品**：优先分析用户流失的主要来源
3. **活跃度 > 历史地位**：关注正在增长而不是曾经辉煌的竞品

### 深度 vs 广度

- **宁可深入分析 2-3 个关键竞品**，不要浅浅列出 10 个
- **每个洞察都要有具体依据**，不要泛泛而谈
- **形成可行动的建议**，而不是简单的信息罗列

### 差异化视角

- **不只描述"他们做了什么"**，要分析"为什么这样做"
- **不只罗列功能**，要理解背后的业务逻辑和设计决策
- **不只看优点**，也要看弱点和机会

### 持续更新

- 竞品会不断变化，需要定期重新调研
- 关注竞品的新功能、新策略、新动向
- 建立竞品信息更新机制
