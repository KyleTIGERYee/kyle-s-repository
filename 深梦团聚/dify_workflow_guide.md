# 深梦团聚 - Dify Workflow 详细配置指南

为了让你的 Dify 工作流能和我们刚才写好的“前端界面”及“Python 服务器”完美对接，你需要在 Dify 的管理后台中按照以下步骤严格配置**输入变量**与**输出格式**。

## 0. 创建或进入你的 Dify工作流
确保你在 Dify 中创建的是一个 **“工作流 (Workflow)”** 类型的应用，而不是“聊天助手 (Chatbot)”。
并且在**“大语言模型 (LLM)”**节点的设置中，必须**启用“视觉 (Vision)”能力**！

---

## 1. 配置“开始节点 (Start Node)”的输入变量
在工作流的最开头，你需要定义接收的所有变量。**变量的系统名称必须与下方表格一字不差**，因为我们在代码中就是按照这些名称传递参数的。

### 1-1. 文本字段（类型全选：`String` / 文本）

| 变量名称 (Variable Name) | 表单对应含义 | 建议最大长度限制 | 必填项 (Required) |
| :--- | :--- | :--- | :--- |
| `name` | 申请人姓名 | 50字 | 是 |
| `idCard` | 身份证号 | 18字 | 是 |
| `phone` | 手机号码 | 20字 | 是 |
| `workUnit` | 当前就职单位全称 | 200字 | 是 |
| `jobType` | 职业类型（如医护人员等） | 50字 | 是 |
| `store` | 意向入住的门店 | 50字 | 是 |
| `familySize` | 团聚家庭人数 | 10字 | 是 |
| `relationships` | 成员关系（多选会被拼接，如“父母, 配偶”） | 100字 | 是 |
| `checkInDate` | 计划入住日期 | 20字 | 是 |
| `stayDays` | 预计停留天数 | 10字 | 是 |
| `story` | 是否愿意参与记录（"yes" 或 "no"） | 10字 | 是 |

### 1-2. 图片/文件字段（类型选：`File Array` / 文件数组）
由于我们已经打通了原生图片的多图上传处理逻辑，你需要增加接收图片的专属信道：

| 变量名称 (Variable Name) | 对应材料 | 类型选择 (Type) |
| :--- | :--- | :--- |
| `idImages` | 身份证明材料（最多3张） | **File Array (文件数组)** |
| `relationImages` | 关系证明材料（最多10张） | **File Array (文件数组)** |

> [!TIP]
> 配置完后，记得把 Start 节点的所有变量线，牵引连接到下一节点（通常是你的大语言模型节点）。对于那两个 `Images` 开头的图片变量，请将它们连向模型的图片/视觉输入区。

---

## 2. 配置“大语言模型节点 (LLM Node)”的提示词 (Prompt)
你可以在此自由发挥你的审核逻辑。在写 Prompt 时，可以通过输入 `/` 来插入我们前面定义的变量。

**示例 Prompt 模板：**
```text
你是一个智能的“深梦团聚春节免费住”审核官。现在有一位申请人提交了资料。
【申请人信息】
姓名：{{name}}
身份证：{{idCard}}
电话：{{phone}}
单位：{{workUnit}}
职业类别：{{jobType}}

【申请意向】
意向门店：{{store}}
家庭人数：{{familySize}}
成员关系：{{relationships}}
入住日期：{{checkInDate}}
停留天数：{{stayDays}}
接受采编意愿：{{story}}

【用户提供的身份图片与关系证明图片已附带在上下文中】
你的任务是：
1. 识别并校验图片中的身份信息是否为外卖员、快递员、网约车司机及一线基层建设者。
2. 校验人员与工作单位是否合理。
3. 请以第三人称视角，最终决策是通过（PASSED）还是驳回（FAILED），并给出一段50字以内、语气友善的中文短语表明原因。
```

---

## 3. 核心：配置“结束节点 (End Node)”的输出结构
这是决定前端能否正确跳转“申请成功页”和“申请失败页”的**命脉**。如果你不设置这些，前端接收不到状态，会永远卡死展示报错。

在 End 节点的“输出变量 (Output Variables)”中，请配置**两个必填的固定结构变量**，它们的数据源都来自于上方大模型生成的对应内容：

### 必须配置的 2 个输出变量 (类型：String)：

1. **`audit_result`**
   * **值限制**：这一项的结果只能输出纯文本的 `"PASSED"` 或者 `"FAILED"`。
   * **意义**：前端 Vue 代码会精确读取这个值，如果是 `PASSED` 就闪出礼花绿色的通过页；如果是 `FAILED` 则拉起重新提交的驳回页。
2. **`audit_reason`**
   * **值限制**：一段具体的中文评语，尽量控制在一两句话。
   * **意义**：这段话将直接被渲染到结果页面的正中央向用户展示。
   * *比如："尊敬的王先生，您好！您的身份已被识别为基层建设者，关系证明图片清晰，您的深梦团聚资格已审核通过，期待您的入住。"*

> [!IMPORTANT]
> 如果你的大语言模型节点直接生成了一段含糊的综合长文本（把 PASSED 和理由揉在了一起），强烈建议你在 LLM 节点与 End 节点之间，增加一个 **“代码执行节点 (Code Node)”**，写一段简单的 Python 或 JS 脚本，将模型的自然语言文本自动拆分成规范的 `audit_result` 和 `audit_reason` 两个变量传给 End。

## 4. 最终发布
保存所有节点配置，并在工作流页面的右上角点击 **“发布 (Publish)”**。只要不发布，我们在本地的 Python 代码死也调不到新的配置！
